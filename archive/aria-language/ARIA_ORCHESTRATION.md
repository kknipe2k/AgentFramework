# ARIA: Agentic Rail-based Intent Architecture

> *"The LLM writes the recipe. The rails ensure it's followed."*

## What is ARIA?

ARIA is **not a programming language** - it's a structured orchestration format that:

1. **LLM generates** → Plans written in constrained, verifiable syntax
2. **Rails execute** → Deterministic execution with zero ambiguity
3. **Gates verify** → Every step validated before proceeding
4. **Intent preserved** → Original intent checked at every checkpoint

```
┌─────────────────────────────────────────────────────────────────┐
│                      USER INTENT                                │
│            "Add user authentication to my app"                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    LLM PLAN GENERATION                          │
│              Writes structured ARIA recipe                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     ARIA EXECUTOR                               │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐          │
│   │  RAIL   │→ │  GATE   │→ │  RAIL   │→ │  GATE   │→ ...    │
│   │ (action)│  │(verify) │  │ (action)│  │(verify) │          │
│   └─────────┘  └─────────┘  └─────────┘  └─────────┘          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   VERIFIED OUTPUT                               │
│         Tests pass, docs generated, intent matched              │
└─────────────────────────────────────────────────────────────────┘
```

## Core Concepts

### 1. INTENT (The Sacred Contract)

The original user intent is captured and checked at every step:

```yaml
intent:
  summary: "Add JWT authentication to Express API"
  must_have:
    - User can register with email/password
    - User can login and receive JWT token
    - Protected routes reject invalid tokens
  must_not:
    - Store plain text passwords
    - Expose tokens in URLs
    - Break existing API endpoints
  success_criteria:
    - All auth tests pass
    - Existing tests still pass
    - Security scan clean
```

### 2. RAILS (Constrained Actions)

Rails are atomic, verifiable actions with explicit inputs/outputs:

```yaml
rail create_file:
  input:
    path: string, required
    content: string, required
  output:
    created: boolean
    path: string
  verify:
    - file_exists(output.path)
    - content_matches(output.path, input.content)

rail run_command:
  input:
    command: string, required
    expect_exit: int, default=0
  output:
    exit_code: int
    stdout: string
    stderr: string
  verify:
    - output.exit_code == input.expect_exit

rail edit_file:
  input:
    path: string, required
    find: string, required
    replace: string, required
  output:
    modified: boolean
    matches_found: int
  verify:
    - output.matches_found >= 1
    - not contains(read_file(input.path), input.find)
```

### 3. GATES (Verification Checkpoints)

Gates block progression until conditions are met:

```yaml
gate tests_pass:
  run: npm test
  expect: exit_code == 0
  on_fail:
    action: rollback_to_checkpoint
    message: "Tests failed, reverting changes"

gate intent_check:
  type: llm_verify
  prompt: |
    Original intent: {intent.summary}
    Current state: {checkpoint.summary}
    Question: Does the current state align with the intent?
  expect: response.aligned == true
  on_fail:
    action: pause_and_report
    message: "Intent drift detected"

gate security_scan:
  run: npm audit --audit-level=high
  expect: exit_code == 0
  on_fail:
    action: block
    message: "Security vulnerabilities found"
```

### 4. CHECKPOINTS (Rollback Points)

Automatic state capture for recovery:

```yaml
checkpoint before_auth_implementation:
  capture:
    - git_state
    - file_hashes: [src/**, tests/**]
    - test_results: npm test --json

checkpoint after_user_model:
  capture:
    - git_state
    - new_files: [src/models/User.js]
    - test_results: npm test --json
```

## ARIA Recipe Syntax

A complete recipe that an LLM generates:

```yaml
# ============================================
# ARIA RECIPE: Add JWT Authentication
# Generated by: Claude
# Intent hash: a3f2b1c9
# ============================================

aria: 1.0
name: add-jwt-auth
generated: 2024-01-15T10:30:00Z

# THE SACRED CONTRACT
intent:
  summary: "Add JWT authentication to Express API"
  user_said: "Add user authentication to my app"
  must_have:
    - User registration with email/password
    - Login endpoint returning JWT
    - Middleware for protected routes
    - Password hashing with bcrypt
  must_not:
    - Plain text passwords
    - Tokens in URLs or logs
    - Breaking changes to existing API
  done_when:
    - All new tests pass
    - All existing tests pass
    - Security scan clean
    - Can register, login, access protected route

# ENVIRONMENT REQUIREMENTS
requires:
  node: ">=18.0.0"
  packages: [express, jsonwebtoken, bcrypt]
  env_vars: [JWT_SECRET]

# EXECUTION PLAN
plan:

  # ─────────────────────────────────────────
  # PHASE 1: Setup Dependencies
  # ─────────────────────────────────────────
  - phase: setup
    description: "Install required packages"

    - rail: run_command
      id: install_deps
      input:
        command: npm install jsonwebtoken bcrypt express-validator
        expect_exit: 0

    - gate: deps_installed
      verify:
        - package_has_dep("jsonwebtoken")
        - package_has_dep("bcrypt")

    - checkpoint: deps_ready

  # ─────────────────────────────────────────
  # PHASE 2: Create User Model
  # ─────────────────────────────────────────
  - phase: user_model
    description: "Create User model with password hashing"

    - rail: create_file
      id: create_user_model
      input:
        path: src/models/User.js
        content: |
          const bcrypt = require('bcrypt');

          class User {
            constructor(email, passwordHash) {
              this.email = email;
              this.passwordHash = passwordHash;
            }

            static async create(email, password) {
              const hash = await bcrypt.hash(password, 10);
              return new User(email, hash);
            }

            async verifyPassword(password) {
              return bcrypt.compare(password, this.passwordHash);
            }
          }

          module.exports = User;

    - gate: no_plain_passwords
      type: content_scan
      files: [src/models/User.js]
      must_not_contain:
        - /password\s*=\s*['"][^'"]+['"]/  # literal passwords
        - /\.password\s*=/                  # storing raw password

    - checkpoint: user_model_done

  # ─────────────────────────────────────────
  # PHASE 3: Create Auth Routes
  # ─────────────────────────────────────────
  - phase: auth_routes
    description: "Create registration and login endpoints"

    - rail: create_file
      id: create_auth_routes
      input:
        path: src/routes/auth.js
        content: |
          const express = require('express');
          const jwt = require('jsonwebtoken');
          const User = require('../models/User');

          const router = express.Router();

          // POST /auth/register
          router.post('/register', async (req, res) => {
            try {
              const { email, password } = req.body;
              const user = await User.create(email, password);
              // Store user (simplified - use DB in production)
              res.status(201).json({ message: 'User created' });
            } catch (error) {
              res.status(400).json({ error: error.message });
            }
          });

          // POST /auth/login
          router.post('/login', async (req, res) => {
            try {
              const { email, password } = req.body;
              // Verify user (simplified)
              const token = jwt.sign(
                { email },
                process.env.JWT_SECRET,
                { expiresIn: '1h' }
              );
              res.json({ token });
            } catch (error) {
              res.status(401).json({ error: 'Invalid credentials' });
            }
          });

          module.exports = router;

    - gate: no_token_exposure
      type: content_scan
      files: [src/routes/auth.js]
      must_not_contain:
        - /console\.log.*token/i
        - /res\.redirect.*token=/

    - checkpoint: routes_done

  # ─────────────────────────────────────────
  # PHASE 4: Create Auth Middleware
  # ─────────────────────────────────────────
  - phase: middleware
    description: "Create JWT verification middleware"

    - rail: create_file
      id: create_middleware
      input:
        path: src/middleware/auth.js
        content: |
          const jwt = require('jsonwebtoken');

          function authenticateToken(req, res, next) {
            const authHeader = req.headers['authorization'];
            const token = authHeader && authHeader.split(' ')[1];

            if (!token) {
              return res.status(401).json({ error: 'Token required' });
            }

            jwt.verify(token, process.env.JWT_SECRET, (err, user) => {
              if (err) {
                return res.status(403).json({ error: 'Invalid token' });
              }
              req.user = user;
              next();
            });
          }

          module.exports = { authenticateToken };

    - checkpoint: middleware_done

  # ─────────────────────────────────────────
  # PHASE 5: Write Tests
  # ─────────────────────────────────────────
  - phase: tests
    description: "Create comprehensive test suite"

    - rail: create_file
      id: create_tests
      input:
        path: tests/auth.test.js
        content: |
          const request = require('supertest');
          const app = require('../src/app');

          describe('Authentication', () => {
            describe('POST /auth/register', () => {
              it('should create a new user', async () => {
                const res = await request(app)
                  .post('/auth/register')
                  .send({ email: 'test@example.com', password: 'password123' });
                expect(res.status).toBe(201);
              });

              it('should reject weak passwords', async () => {
                const res = await request(app)
                  .post('/auth/register')
                  .send({ email: 'test@example.com', password: '123' });
                expect(res.status).toBe(400);
              });
            });

            describe('POST /auth/login', () => {
              it('should return JWT on valid login', async () => {
                const res = await request(app)
                  .post('/auth/login')
                  .send({ email: 'test@example.com', password: 'password123' });
                expect(res.status).toBe(200);
                expect(res.body).toHaveProperty('token');
              });
            });

            describe('Protected Routes', () => {
              it('should reject requests without token', async () => {
                const res = await request(app).get('/api/protected');
                expect(res.status).toBe(401);
              });

              it('should accept requests with valid token', async () => {
                const loginRes = await request(app)
                  .post('/auth/login')
                  .send({ email: 'test@example.com', password: 'password123' });
                const token = loginRes.body.token;

                const res = await request(app)
                  .get('/api/protected')
                  .set('Authorization', `Bearer ${token}`);
                expect(res.status).toBe(200);
              });
            });
          });

    - checkpoint: tests_written

  # ─────────────────────────────────────────
  # PHASE 6: Final Verification
  # ─────────────────────────────────────────
  - phase: verify
    description: "Run all verification gates"

    - gate: all_tests_pass
      run: npm test
      expect: exit_code == 0
      on_fail:
        action: rollback_to_checkpoint
        checkpoint: tests_written

    - gate: existing_tests_pass
      run: npm test -- --testPathIgnorePatterns=auth
      expect: exit_code == 0
      on_fail:
        action: pause_and_report
        message: "Existing tests broken - review changes"

    - gate: security_audit
      run: npm audit --audit-level=high
      expect: exit_code == 0

    - gate: intent_satisfied
      type: llm_verify
      prompt: |
        Original intent: {intent.summary}
        Must have: {intent.must_have}
        Must not: {intent.must_not}

        Files created:
        - src/models/User.js
        - src/routes/auth.js
        - src/middleware/auth.js
        - tests/auth.test.js

        Test results: {test_output}

        QUESTION: Is the original intent fully satisfied?
        ANSWER (yes/no with explanation):
      expect: response.startswith("yes")

# POST-EXECUTION
on_complete:
  - generate_docs:
      template: api_endpoint
      output: docs/auth.md
  - update_changelog:
      entry: "Added JWT authentication (register, login, protected routes)"
  - create_commit:
      message: "feat(auth): Add JWT authentication system"

on_fail:
  - rollback_to: initial_state
  - report:
      include: [failed_gate, error_log, intent_diff]
```

## Execution Model

### How ARIA Runs

```
┌─────────────────────────────────────────────────────────────────┐
│                     ARIA EXECUTOR                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. PARSE recipe (validate syntax, check requirements)         │
│                          │                                      │
│                          ▼                                      │
│  2. CAPTURE initial checkpoint                                  │
│                          │                                      │
│                          ▼                                      │
│  3. FOR EACH phase:                                            │
│     │                                                          │
│     ├─► FOR EACH rail:                                         │
│     │   ├─► Execute action                                     │
│     │   ├─► Verify output                                      │
│     │   └─► IF fail: trigger on_fail handler                   │
│     │                                                          │
│     ├─► FOR EACH gate:                                         │
│     │   ├─► Run verification                                   │
│     │   ├─► IF pass: continue                                  │
│     │   └─► IF fail: rollback/pause/block                      │
│     │                                                          │
│     └─► Capture checkpoint                                     │
│                          │                                      │
│                          ▼                                      │
│  4. RUN on_complete hooks                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Gate Types

| Type | What It Does | Example |
|------|--------------|---------|
| `run` | Execute command, check exit code | `npm test` |
| `content_scan` | Regex scan files for patterns | No plain passwords |
| `llm_verify` | Ask LLM to verify semantic correctness | Intent alignment |
| `diff_check` | Verify only expected files changed | No surprise edits |
| `schema_validate` | Validate output against schema | API response format |

### Failure Handlers

| Action | Behavior |
|--------|----------|
| `rollback_to_checkpoint` | Revert to last known-good state |
| `pause_and_report` | Stop execution, show error, wait for human |
| `block` | Hard stop, no recovery |
| `retry(n)` | Attempt action n times |
| `skip_with_warning` | Continue but flag for review |

## LLM Integration Points

### 1. Plan Generation

```python
def generate_aria_plan(user_intent: str, codebase_context: str) -> str:
    """LLM generates the ARIA recipe"""

    prompt = f"""
    You are an ARIA plan generator. Create a structured execution plan.

    USER INTENT: {user_intent}

    CODEBASE CONTEXT:
    {codebase_context}

    Generate a complete ARIA recipe following the schema.
    Include:
    - Clear intent capture with must_have/must_not
    - Atomic rails for each action
    - Verification gates after each phase
    - Checkpoints for rollback
    - Tests that verify the intent

    OUTPUT ONLY VALID ARIA YAML:
    """

    return llm.generate(prompt, schema=ARIA_SCHEMA)
```

### 2. Intent Verification (llm_verify gates)

```python
def verify_intent(intent: Intent, current_state: State) -> bool:
    """LLM checks if current state matches intent"""

    prompt = f"""
    ORIGINAL INTENT: {intent.summary}
    MUST HAVE: {intent.must_have}
    MUST NOT: {intent.must_not}

    CURRENT STATE:
    - Files modified: {current_state.modified_files}
    - Tests status: {current_state.test_results}
    - Diff summary: {current_state.diff_summary}

    QUESTION: Does the current implementation satisfy the original intent?

    Answer YES or NO, then explain in one sentence.
    """

    response = llm.generate(prompt)
    return response.lower().startswith("yes")
```

### 3. Error Recovery

```python
def handle_failure(failed_gate: Gate, context: ExecutionContext) -> Action:
    """LLM determines recovery action"""

    prompt = f"""
    A verification gate failed during ARIA execution.

    FAILED GATE: {failed_gate.name}
    ERROR: {failed_gate.error}
    CURRENT PHASE: {context.current_phase}
    AVAILABLE CHECKPOINTS: {context.checkpoints}

    What recovery action should be taken?
    Options:
    1. rollback_to_checkpoint(checkpoint_name)
    2. retry_with_fix(suggested_fix)
    3. pause_for_human
    4. abort

    Recommend ONE action with reasoning:
    """

    return llm.generate(prompt, schema=RECOVERY_ACTION_SCHEMA)
```

## Why ARIA Works for LLMs

### The Problem with Current Approaches

| Issue | Current State | ARIA Solution |
|-------|---------------|---------------|
| **Intent drift** | LLM forgets original goal | Intent checked at every gate |
| **Ambiguous actions** | "Update the file" - which file? how? | Rails have explicit inputs/outputs |
| **No verification** | Hope it worked | Gates verify before proceeding |
| **No rollback** | Stuck with broken state | Checkpoints enable recovery |
| **Missed requirements** | "Oh, I forgot about tests" | Plan includes all requirements upfront |

### The ARIA Advantage

```
TRADITIONAL LLM:
  User: "Add auth"
  LLM: *writes code*
  LLM: *writes more code*
  LLM: *forgets to add tests*
  LLM: *introduces security bug*
  User: "This doesn't work"

ARIA:
  User: "Add auth"
  LLM: *generates ARIA plan with tests, security gates*
  Executor: *runs rail: create user model*
  Executor: *gate: no_plain_passwords* ✓
  Executor: *runs rail: create auth routes*
  Executor: *gate: no_token_exposure* ✓
  Executor: *runs rail: create tests*
  Executor: *gate: all_tests_pass* ✓
  Executor: *gate: intent_satisfied* ✓
  User: "It works!"
```
