# Common Gotchas (lessons learned)

The spec is large and the project covers a lot of ground. These are traps that have already bitten the work or are predictable based on the design. CLAUDE.md §15 references this file as the source.

1. **Tool ≠ Skill ≠ Agent.** Three distinct concepts (§0b). Tools are called. Skills are loaded into context. Agents are spawned. Don't conflate them; the schemas, file formats, and runtime mechanics are different.
2. **Capability narrowing on Agent→Agent edges.** A child agent's `allowed_tools` and `allowed_skills` cannot exceed the parent's. The Builder Canvas (Phase 9) enforces this automatically; manual JSON editing must respect it.
3. **v0.1 ships STANDARD mode hardcoded.** No mode router (§3b). The framework JSON's `modes` field still exists in schema but is not evaluated at runtime in v0.1.
4. **v0.1 ships `fresh_context_per_task` only.** The continuous loop policy (Ralph-style) is in the schema but not implemented. `examples/ralph/framework.json` exists but won't run on v0.1.
5. **v0.1 ships Novice + Promoted tiers only.** No Operator tier. Promoted is blocked from auto-accepting `shell: true` and `network: ["*"]` artifacts even though the tier's general behavior is auto-accept-when-validated.
6. **v0.1 is single-session.** §1c Multi-session is v1.0. Do not write multi-session code paths in v0.1; they create surface area without benefit.
7. **v0.1 is Windows-only.** Not because Tauri is Windows-only — Tauri is cross-platform — but because we test only on Windows in v0.1, and macOS/Linux ports come at v1.0. CI still runs on all three OSes to catch drift early.
8. **No telemetry, ever.** No analytics, no crash reporter, no "anonymous metrics," no phone-home. Per §13 of spec. Adding any requires an ADR with public dashboard plan + opt-in mechanism.
9. **Anthropic API is hit directly.** No `@anthropic-ai/sdk` dep, no `anthropic-rs`. `reqwest` + `eventsource-stream` only. The API surface is small and stable; direct HTTP keeps the dependency surface minimal.
10. **Tauri allowlist is the security boundary.** The renderer has no Node API. Anything the renderer needs from Rust goes through a typed `#[tauri::command]`. Don't widen the allowlist without considering capability implications.
11. **Drone ≠ Main ≠ Sandbox.** Three Rust processes. Drone owns SQLite + snapshots + recovery (per session). Main owns the agent loop, MCP, providers, framework loader, capability enforcer. Sandbox is per-artifact, OS-isolated, used for L3 validation. Don't blur these.
12. **IPC is two layers.** Renderer↔Main is Tauri typed IPC. Main↔Drone is framed JSON over Unix socket / named pipe. Different mechanisms with different semantics. Don't try to use Tauri IPC for drone communication.
13. **SQLite WAL pragmas matter.** `PRAGMA journal_mode = WAL`, `PRAGMA synchronous = NORMAL`, `PRAGMA busy_timeout = 5000`, `PRAGMA foreign_keys = ON`. Set them in this order at every connection open. Missing busy_timeout causes flaky tests under contention.
14. **Snapshots are append-only.** Drone never updates a snapshot row; new snapshot = new row. State_hash deduplication happens at read time, not write time.
15. **Resume rebuilds history, doesn't re-execute.** Tool calls in flight at crash time are flagged `tool_call_uncertain` and surfaced to the user. Don't replay tool calls automatically.
16. **`request_capability` is a meta-tool.** It's auto-injected into every agent's tool list. When the model calls it, the runtime translates to `tool_missing` or `skill_missing` and routes through gap flow. Agents can decline `skill_load_requested` events but typically comply.
17. **Mode-variant skills filter sections.** A `skill.md` with `mode_variants: { LITE: { include_sections: ["quick"] }, ... }` has its body filtered by section header at load time. The full markdown is on disk; the model sees only the slice for the active mode.
18. **JSONLogic for triggers.** Programmatic skill triggers use a JSONLogic-style expression language. Operators allowed in v0.1: `var`, `==`, `!=`, `<`, `<=`, `>`, `>=`, `and`, `or`, `not`, `in`, `+`, `-`, `*`, `/`. Adding operators requires extending the evaluator; do not silently extend.
19. **Capability declarations are mandatory for generated artifacts.** Hand-authored artifacts can omit the `capabilities` block, but they default to Operator-tier-only loading. Generated artifacts (Phase 8) must declare capabilities; the validator rejects missing blocks.
20. **DCO sign-off is mandatory.** `git commit -s`. Without it, the commit is rejected by the CI hook (once configured at M1+).
21. **Common clippy pedantic+nursery traps in this project.** Compound entry covering recurring patterns from M01.B onward — appeared in every M02 stage retrospective:
    - `redundant_pub_crate` — use plain `pub` in private modules, not `pub(crate)`.
    - `derive_partial_eq_without_eq` — `serde_json::Value` containment requires `#[allow]` with one-line rationale; manual `Eq` is wrong because Value contains floats.
    - `unused_async` — cross-platform `cfg` variants where one branch awaits and another doesn't trip this; suppress per-fn when both variants must share signature.
    - `default_trait_access` — use `HashMap::default()` over `Default::default()`; explicit type avoids the lint.
    - `match_wildcard_for_single_variants` — bind explicitly (`Foo::Bar(_)`) rather than `_` when only one variant remains.
    - `cast_precision_loss` / `suboptimal_flops` — pin numeric ops to exact types (`f64`, `u64`) and document the precision claim inline.
    - `struct_excessive_bools` — collapse 3+ `bool` fields into a typed flag enum or struct.
    - `missing_const_for_fn` — `const fn` opt-in for pure constructors; clippy nags otherwise.
    - `unnecessary_literal_bound` / `doc_markdown` — backtick code identifiers in doc comments; align literal slice bounds.
    All five M02 stages tripped at least one of these; consolidating here so M03+ stages don't rediscover. (Source: M02-summary.md "Friction patterns that recurred".)
22. **Subprocess-spawning integration tests must derive paths from `std::env::current_exe()`.** Never hard-code `target/debug/<binary>` literals — `cargo llvm-cov --workspace` uses a distinct target dir, and the test binary's location depends on whether you ran `cargo test` (debug) or `cargo llvm-cov` (instrumented). Archetype: `crates/runtime-main/tests/drone_ipc_loopback.rs::drone_binary`. The pre-M02 `crates/runtime-drone/tests/integration*.rs` fixtures still use literals and need retrofitting before M03. (Source: M02.D Decisions.)
23. **Tauri 2.x E2E uses `tauri-driver` + WebdriverIO, not Playwright `_electron`.** Playwright's `_electron` API is Electron-specific and won't drive a Tauri 2.x WebView2 / WebKitGTK window. Per official docs (https://v2.tauri.app/develop/tests/webdriver/), only Linux + Windows are supported (macOS unsupported). M02 Stage E ships renderer-level Playwright against the Vite dev server with `@tauri-apps/api` mocked at the module level; full desktop-shell E2E is M03 carry-forward. (Source: M02.E friction r1.)
24. **ESLint 9 default is flat-config (`eslint.config.js`).** The legacy `.eslintrc.cjs` requires `ESLINT_USE_FLAT_CONFIG=false` to load, which most CI environments don't set. New projects should use `eslint.config.js` directly. (Source: M02.E friction r3.)
25. **Vite project root holds `index.html`.** The convention is `<root>/index.html` (not `<root>/src/index.html`); the script tag references `/src/main.tsx` (absolute path under root). Move the file or set `root: 'src'` in `vite.config.ts` if your layout differs. (Source: M02.E friction r9.)
26. **`serde(tag = "type")` requires struct-shape variants, not newtype-wrapped primitives.** `Provider(String)` errors at runtime because the tag-form serializer can't add `{ "type": "Provider" }` to a String; `Provider { message: String }` works because struct variants accept inline-tagging. Pattern: any error/event enum surfaced over Tauri IPC or HTTP wire format must use struct-shape variants. (Source: M02.E friction r7.)
27. **Vitest + React Testing Library: re-query DOM refs after each await.** Capturing a button's DOM ref before an awaited re-render leaves it stale even when `waitFor(toBeEnabled())` passes — `expect(button).toBeEnabled()` succeeds, then `userEvent.click(button)` fires on the stale ref. Sync on a visible-in-DOM indicator (`findByLabelText(...)` for the result, then re-query the button) before re-querying dependent UI. Archetype: `tests/unit/App.test.tsx::save_key_then_run_smoke_renders_event_list`. (Source: M02.E friction r8.)
28. **Bound test-fixture streams.** `futures::stream::repeat(...)` against a stateful pipeline (in-memory accumulator, channel, cancellation token) will OOM the harness — M02.D measured ~34 GB allocated before the test runner aborted. Use `take(N)` or `iter([...]).chain(pending())` to bound the stream while still triggering whatever timeout/cancellation behavior you're testing. Archetype: `tests/sdk_cancellation.rs::drops_mid_text_burst_no_panic`. (Source: M02.D friction r1.)
29. **`keyring` 3.x ships NO platform backend by default.** Bare `keyring = "3.6"` compiles fine but uses a stub that silently succeeds on writes and returns `NoEntry` on reads. The renderer sees "Save key ✓" then `setup_required` on read and the gap is invisible without log output. Always opt into platform features explicitly. Workspace pattern (M02 carry-forward fix): `keyring = { version = "3.6", default-features = false, features = ["apple-native", "windows-native", "sync-secret-service"] }`. Verify the dep tree includes `windows-sys` (Windows) / `core-foundation-sys` (macOS) / `secret-service` (Linux) — if those are absent from `Cargo.lock`, the stub backend is in use. Triggered the M02.E live-test "[object Object]" smoke-test failure. (Source: M02 Stage E live debugging, post-merge.)
30. **Tauri renderer's `catch(e)` gets a plain object when Rust throws a `serde(tag = "type")` enum.** `e instanceof Error` returns `false`; `String(e)` yields `"[object Object]"`; the actual error is hidden in `e.message` (or `e.type` for unit variants). Always unwrap structured errors via a typed helper — see `src/lib/ipc.ts::unwrapCmdError` for the canonical pattern (Error instance → graceful object → `e.type`/`e.message` → fallback). And ALWAYS `console.error(label, e)` before user-facing dispatch so DevTools shows the full structured object even when the renderer collapses it to a string. Pairs with #29: without the console log, the keyring-stub failure surfaced as "[object Object]" in the UI and zero diagnostic signal anywhere. (Source: M02 Stage E live debugging, post-merge.)
31. **Tauri main process binary needs `tracing_subscriber::fmt::init()` in `main()`.** `tracing::info!` / `tracing::error!` calls compile fine without it but emit to a null sink. Symptoms: `cargo run` / `npm run tauri dev` shows zero output from inside Tauri commands even when they error. Fix: add `init_tracing()` at the top of `main()` calling `tracing_subscriber::fmt::EnvFilter`-configured init. Default level should be `info` globally + `debug` for project crates; honor `RUST_LOG` env override. Per spec §13.5 Dev Logging. (Source: M02 Stage E live debugging, post-merge.)

32. **Hand-authored cross-stack integration examples in build prompts.** When a build prompt contains a code example for a cross-stack integration (Tauri WebDriver capabilities, OS-keychain backend feature flags, MCP JSON-RPC framing, OAuth flows, ESLint flat-config shape, etc.) authored from docs/memory rather than copied verbatim from a known-working upstream reference, the build agent will ship it verbatim and CI will fail with a setup-shaped error (capability mismatch, "no binary at...", silent stub backend, "Failed to match capabilities"). Bit M02 (PR #45 — keyring 3.x stub backend silently passing writes; gotcha #29) and M03 (PR #47 — tauri-driver capabilities, three iterations: 'edge'/'webkit2gtk' → 'wry' → omit). Mitigation: any cross-stack integration example in a build prompt MUST (a) be quoted verbatim from a working upstream example with a commit SHA referenced in a comment, OR (b) carry an explicit "verify against upstream reference X before shipping" guard naming the file/repo/issue to check. The discriminator is whether the example crosses a stack boundary (Rust ↔ TS, Rust ↔ OS, runtime ↔ third-party protocol) — pure-Rust and pure-React/TS examples don't trigger this rule. (Source: M02 PR #45 + M03 PR #47 hotfix iterations; STAGE-PROMPT-PROTOCOL.md §10 + §13.)

33. **snake_case schema discipline.** Schemas in `schemas/*.v1.json` use snake_case for all field names (`tokens_in`, `agent_id`, `duration_ms`); generated Rust types preserve snake_case via serde rename derives, and TS types are likewise generated with snake_case. Authoring stage prompts in camelCase before schema regeneration locks the convention is the recurring failure mode — sample bodies need to match the lock. Recurred across M03.A, M03.B, M03.D. Mitigation: regenerate types BEFORE writing any sample bodies, then reference the generated types as the contract. (Source: M03 Stage A–D retros; M03-summary.md "Friction patterns that recurred".)

34. **Prettier and `cargo fmt` are the first `verify_gates` step, not the last.** Each multi-file authoring stage's first round-1 friction was 1–9 files tripping prettier formatting. Run `prettier --write` (TS) / `cargo fmt --all` (Rust) BEFORE any downstream lint to catch mechanical formatting churn early — clippy/eslint findings on formatting-mangled code are noise that disappears once fmt runs. Recurred M03.B, M03.C, M03.D, M03.E. Mitigation: bake `fmt --write` into stage prompt `<execution_steps>` immediately after `implement` and before any lint or test. (Source: M03-summary.md "Friction patterns that recurred".)

35. **React Flow + happy-dom emits unavoidable `act()` warnings.** React Flow uses `useLayoutEffect` for size measurements; happy-dom doesn't fully simulate the browser layout pipeline, so React schedules updates outside RTL's `act()` boundary. Tests still pass and gates stay green; the warnings come from React Flow internals (`MarkerDefinitions`, `EdgeRendererComponent`, `Pane`, `ZoomPane`). Don't try to wrap in `act()` — it doesn't work; the source is below user code. Mitigation: ignore for components using React Flow under happy-dom; if signal-to-noise becomes a problem, filter via console-error mocks scoped to those components. (Source: M03.B + M03.C retros.)

36. **For renderer components without yet-defined event variants, pass populated state directly into `<NodeComponent>` rather than dispatching events through the store.** M03 shipped GapNode, HITLNode, PlanNode, TaskNode, VerifyNode, HookNode renderers before their corresponding event variants exist (M04+/M05 wires them). Testing them via `applyEvent('plan_created', {...})` would force inventing event shapes pre-spec; testing via `<PlanNode data={syntheticPlanState} />` locks the renderer's contract without spec churn. Pattern locked M03.C; durable through M03.D. (Source: M03.C–D retros; M03-summary.md.)

37. **Don't add explicit type assertions after TypeScript narrows discriminated unions.** After `node.type === 'plan'`, TS already narrows `node` to the `PlanNode` branch — adding `as PlanNode` is dead weight and trips `@typescript-eslint/no-unnecessary-type-assertion`. Surfaced as eslint errors in M03.B and M03.D. Mitigation: write code that lets the narrow do its work; if TS doesn't narrow as expected, the discriminant tag or interface shape is wrong, not the missing assertion. (Source: M03-summary.md.)

38. **WebdriverIO v9 changed `await $('selector')` semantics — chainables don't extend `PromiseLike`.** The v8 pattern `const el = await $('button'); await el.click();` silently breaks under v9 because `ChainablePromiseElement` no longer satisfies `PromiseLike`. Call methods directly on `$()` and `$$()` chainables without intermediate await: `await $('button').click()`. Add `import type {} from 'webdriverio'` as a side-effect import to bring in `WebdriverIO.Browser` augmentations. Bit M03.F with three round-trips during typing setup. (Source: M03.F retro; M03-summary.md.)

39. **`npm audit` failures from transitive deps fix via `package.json` `overrides`, not by swapping the dependent.** When a new devDep brings in a patched-but-old transitive that triggers `npm audit --audit-level=high`, force the patched version via `overrides.<package>: "^<patched-version>"` rather than dropping the dependent or pinning the dev dep to an older release. M03.F example: `overrides.serialize-javascript: "^7.0.5"` resolved 3 high-severity findings from `@wdio/mocha-framework`'s transitive tree without touching the framework itself. (Source: M03.F retro.)

40. **The standard eslint override glob `**/*.config.{ts,js}` matches `vite.config.ts` but NOT `wdio.conf.ts`.** Tauri 2.x WebDriver convention names the config file `wdio.conf.ts` (per upstream docs), and the `.conf.ts` extension falls outside the `.config.ts` glob. Configure an explicit named override (`'wdio.conf.ts'`) plus `parserOptions.projectService: false` for that file; otherwise `projectService` cannot find it in tsconfig includes and eslint errors with "file not in tsconfig". (Source: M03.F retro.)

41. **Phase doc claims about codebase reality must be grep-verified at authoring time, not derived from spec/retros alone.** When a milestone Phase doc names file paths, function names, struct names, IPC command variants, integration points, or any other claim about the codebase's current state, every claim must be verified via `grep` / `Test-Path` / file inspection at *authoring time* — before the doc ships. Trusting `agent-runtime-spec.md` text or `docs/gap-analysis.md` carry-forward chains as proxies for codebase state is a named failure mode: spec describes intent; gap-analysis is append-only and items "carry forward" but don't auto-close when work happens incidentally; both can drift from current code reality. Bit M04 (Phase doc claimed `event_translation.rs`, `prompt_template.rs`, `WriteSignal` IPC variant, `vdr.rs` in `runtime-main`, `RevertToSnapshot` as new — none matched reality; 8 of 11 plan/task events already existed; structurally misaligned across Stages B/D/F). Different from #32: that gotcha is about *third-party* examples (Tauri docs, etc.); this gotcha is about claims about *your own* codebase. Mitigation: at authoring time, for every X.2 file path and every X.3 symbol/function/struct/path claim, run the verification command and confirm reality before shipping the Phase doc. The "Pre-existing legacy file inventory" section in `docs/build-prompts/TEMPLATE.md` is *verified*, not *descriptive* — see TEMPLATE.md addition + STAGE-PROMPT-PROTOCOL.md §10 v1.3 hardening rule. (Source: M04.A2 friction event Sev-4 + M04 codebase audit post-A2.)

42. **Origin is a partial view of project state when CLAUDE.md §8 forbids per-stage pushes.** Any orchestration session reading `origin` alone may infer "stage X unexecuted" when in fact the build machine has the work locally. Different discriminator from #41: #41 is *grep-verify-claims* (verify against the codebase); this rule is *verify against THE RIGHT codebase* — origin is one view, the build machine's local milestone branch is another, and they diverge whenever §8's "no push between stages" rule is in effect. Mitigation: phase-doc edits > 50 lines require the user to paste `git log --oneline main..HEAD` from the build machine for the active milestone branch BEFORE the edit; if the output contains commits with retrospectives in `docs/build-prompts/retrospectives/M[NN].<X>-retrospective.md`, those stages are DONE regardless of what origin shows. Stronger backstop: every stage end surface includes git log + retrospective file listing as a standard item (CLAUDE.md §19 rule 7) — when the user pastes the surface to any downstream orchestration session, that session sees real project state, not origin's partial view. Bit M04 (PR #53 rewrote the M04 phase doc against "A1+A2 unexecuted per origin" while the build machine had `c5fe035` + `2b0e8d2` with full retros; 4+ hours of work merged on a false premise; reverted via PR #54). (Source: M04 PR #53 → revert PR #54; CLAUDE.md §8 phase-doc-edit pre-flight + §19 rule 7 + STAGE-PROMPT-PROTOCOL.md §10 v1.4 hardening.)
