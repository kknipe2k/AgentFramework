## [LIVE] Schema-reality discovery (gotcha #41 follow-through)

The phase doc B.3.2 prescribed "for each `tools[]` / `skills[]` path-referenced entry, parse frontmatter into the inline type → replace the entry with the inline variant." Reading `crates/runtime-core/src/generated/framework.rs` showed this is **schema-imprecise for tools and skills**:

| Type | File:line | Shape | Inline variant? |
|---|---|---|---|
| `FrameworkAgentsItem` | `framework.rs:1375` | `oneOf` enum: `Object {id,path}` \| `Agent(Agent)` | **YES** — variant flip target |
| `FrameworkSkillsItem` | `framework.rs:2786` | flat struct `{ name, path?, registry_id?, source }` | NO |
| `FrameworkToolsItem` | `framework.rs:3029` | flat struct `{ name, path?, registry_id?, source }` | NO |

Only agents have an inline `oneOf` variant; tools/skills are always references in the framework struct. Resolution model resolved into a clean asymmetry:

- **Agents:** variant transformation `Object → Agent(_)` in `framework.agents[]`.
- **Tools/Skills:** read the referenced `.md` → surface body as `Companion { file_name: "tools/git_checkpoint.md", body }`. The framework's `tools[]`/`skills[]` entries are unchanged in shape.

This is captured in the impl + the green commit body. The phase doc text was right in spirit (resolve every path reference) but the schema reality meant the implementation distinguishes the two cases. Codified for future stages in the green commit message + this retrospective.

## [LIVE] Splitter precedent (gotcha #41 follow-through)

Pre-existing frontmatter splitter pattern found at `crates/runtime-core/tests/round_trip.rs:140` — `text.splitn(3, "---\n")` with CRLF normalization at line 139 (`text.replace("\r\n", "\n")`). No reusable function existed; the test re-implemented the pattern in-place. Stage B writes the reusable `pub fn split_frontmatter(text: &str) -> Option<(&str, &str)>` in `crates/runtime-main/src/builder/persist.rs` using the identical pattern. The CRLF-normalize step lives in the caller (the resolver), so the splitter takes LF-only `&str`. Pattern documented inline.
