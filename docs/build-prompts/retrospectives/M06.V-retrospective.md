## Findings

#### 🟡 #1 — Wire: §5a re-resolution-on-connect/disconnect (trace #6) has no production driver

**Spec claim:** §5a step 5 "re-resolution on connect/disconnect" — V.3 trace #6: "`McpClient::add_server` + `remove_server` call `NamespaceResolver::re_evaluate_short_names`; new ambiguities emit `tool_alias_ambiguous` event."
**Observed:** `McpClient` (C.3.1 shape: transports/connections/registry/secret_store/audit) holds **no** `NamespaceResolver` field. `grep` for `connect_server|disconnect_server|re_evaluate_short_names` in `crates/runtime-mcp/src/client/{mod,lifecycle}.rs` returns nothing. The `NamespaceResolver` lives in `McpDispatcher` (`crates/runtime-mcp/src/dispatch.rs:93`, `Arc<RwLock<NamespaceResolver>>`), which has **no production constructor** (no `impl ConnectionResolver for McpClient`; concrete `McpDispatcher` not constructed in `src-tauri` — ADR-0011 Context #1–#3). The re-resolution primitive (`connect_server`/`disconnect_server`/`newly_ambiguous_since`) is fully implemented + unit-tested (`namespace_resolution.rs`: `re_evaluate_short_names_emits_new_ambiguity_when_server_connects_with_overlapping_tool`, `disconnect_server_re_resolves_and_resolves_ambiguity_without_new_ambiguity` — green). `tool_alias_ambiguous` IS emitted on the *dispatch-time* `Ambiguous` outcome (`mcp_dispatch.rs:173` → `graphStore.ts:1562`), but **not** on lifecycle connect/disconnect.
**Expected (per spec):** A production path where adding/removing an MCP server re-evaluates short-name ambiguity and emits `tool_alias_ambiguous` for newly-ambiguous names.
**Trace / harness:** Wire 5-step; broke at step 4 (consumer) — zero production consumers of `re_evaluate_short_names` / `connect_server` / `disconnect_server`.
**Disposition — 🟡 not 🔴:** Same root cause as ADR-0011 item (b) ("construct `CapabilityEnforcer` + `NamespaceResolver` (populated from connected servers) in the shell" — M07 carry-forward). The resolver belongs to the M07-constructed `McpDispatcher` per the ADR-0010/0011 architecture V is directed to read and apply; V.3 trace #6's "McpClient drives re-resolution" expectation is architecturally inconsistent with that design and folds into ADR-0011's M07 carry-forward. The §5a primitive ships delivered + tested; only the production driver is deferred, sharing an already-maintainer-adjudicated M07 root cause. Blocking merge would be inconsistent with the ADR-0009/0011 carry-forward mechanism. **However** it is NOT explicitly enumerated in any `<scope_change>` or ADR carry-forward list — so it must be surfaced.
**Action:** Carry forward to M07 Stage A. M07's ADR-0011-derived carry-forward (b) must be **explicitly extended** to enumerate: "`McpClient` (or its M07 successor) drives `NamespaceResolver` connect/disconnect re-resolution on server add/remove and emits `tool_alias_ambiguous` for newly-ambiguous short names." M07.V Wire trace #6 becomes the expected endpoint (ADR-0009/0011 mechanism, one milestone later).
