## Findings

#### 🔴 #1 — Wire: `tier_gate` defined but never invoked; Novice "Reject" doesn't reject

**Spec claim:** §8.security L4 — "Novice — Manual review of capabilities + diff + validation report; explicit 'Install' click — every install is reviewed." MVP §M7 acceptance: "tier-gate review screen (capability disclosure + L3 report + Install/Reject)" — Install/Reject implies the click gates persistence. Phase doc C.3.1: `tier_gate(&art, tier)?;  // reuse L4 (M05) → TierReviewRequired for Novice`.

**Observed:** Three-step contract drift:

1. `tier_gate` (`crates/runtime-main/src/import/mod.rs:491`) returns `ImportError::TierReviewRequired(TierReview { capabilities, l3_report, requires_secrets })` for `Tier::Novice`. **The function is never called.** Grep `tier_gate\(` in `crates/runtime-main/src/import/` returns ONE definition-site hit and zero call-sites.
2. `import_artifact_with` (`mod.rs:592-660`) skips `tier_gate` entirely (lines 629-635 inline comment: *"L4 tier-gate is applied by the Tauri shell command… The seam itself always installs + hash-locks: the M07.B integrity model is lock-on-first-install, and the review is a renderer confirm-gate, not a backend refusal"*) and always calls `install_with` → `skills_lock::write_entry`. For `ArtifactKind::McpServer` it also unconditionally calls `reg.upsert(...)` (mod.rs:636-639) BEFORE the install assertion, so a Novice rejection still leaves a registry row.
3. The Tauri command (`src-tauri/src/commands.rs:1199-1219`) ALSO does not call `tier_gate`; it sets `review_required: matches!(tier, Tier::Novice)` POST-install. The renderer `dismissImport` (`src/lib/graphStore.ts:1769-1777`) only `delete next[ref]` from the store — no Tauri command, no backend rollback, and there is **no `uninstall_artifact` / `skills_lock::remove` anywhere in the codebase** (`grep uninstall_artifact|remove_artifact|skills_lock::remove` → zero matches).

The test that exercises Reject (`tests/unit/components/ImportPanel.test.tsx:112` `reject_dismisses_the_import_record`) asserts only that the renderer's `useGraphStore.getState().imports[…]` is undefined — it never inspects `skills.lock` or the M06 registry to verify the backend was rolled back. Classic M04-class contract gap: implementation tests green, contract tests missing.

**Expected (per spec):** A Novice "Reject" yields no `skills.lock` entry AND no M06 registry row. A `tier_gate` call before `install_with` would surface `TierReviewRequired`; the renderer's modal would either confirm-then-`install`-then-`lock` or dismiss-and-do-nothing. Either of those would satisfy the spec; the current "always install, post-hoc flag, no rollback" satisfies neither.

**Trace / harness:** 5-step Wire trace.
  (1) Spec claim: "Novice — explicit 'Install' click; every install is reviewed."
  (2) Source event: would be `ImportError::TierReviewRequired` from `tier_gate`.
  (3) Projection/store: no production caller of `tier_gate`.
  (4) Consumer: renderer `dismissImport` ignores the backend, no `uninstall_artifact` command exists.
  (5) Consumer reads what projector writes: **NO** — the consumer (renderer Reject button) writes to local store only; the backend (`skills.lock`, `mcp_servers` registry) is unchanged from the install moment.

**Action:** Open D.fix iter 1; cite finding #1 in D.fix's `<deliverable>`. Two paths are spec-faithful and small:

- **Option A (Recommended — spec-faithful, install-after-confirm).** Move the `tier_gate(&art, tier, &report)?` call into `import_artifact_with` BETWEEN L3 and `install_with` (and BEFORE `reg.upsert` for `McpServer`). When it returns `TierReviewRequired(review)`, the Tauri command surfaces a new `pending_review_id` over the IPC enrichment shape (ADR-0015), the renderer's `confirmImport` calls a new `complete_import_artifact(pending_review_id)` command which finishes the pipeline (`reg.upsert` + `install_with`), `dismissImport` calls a new `cancel_pending_import(pending_review_id)` which drops the in-flight state. ADR-0014's "lock-on-first-install" is preserved (the lock is still written exactly once, at the point of true Install).
- **Option B (Lock-on-import + uninstall).** Keep the current "always lock" but add a `uninstall_artifact(lock_key)` Tauri command that calls a new `skills_lock::remove_entry(path, key)` AND `Registry::remove(name)` for `mcp_server` kind. `dismissImport` calls it. The contract that "Reject reverts the install" is preserved at a small cost (a window of time where the artifact is locked-but-unconfirmed; harmless given finding #2's no-production-load-path).

Either option closes the spec gap. Option A matches the spec phrasing most literally; option B preserves the implementation's chosen "integrity model" rationale.

**Severity rationale (not Dec-6):** This is NOT the Dec-6 pattern (delivered + driver-absent + accepted-ADR carry-forward). The primitive (`tier_gate`) IS shipped but the wire that should drive it has been _replaced_ with a different, non-spec-faithful contract. No ADR documents this divergence (ADR-0015 covers IPC enrichment, not the install-before-confirm change). The build agent may dispute via an ADR-class waiver if the maintainer accepts the "lock-on-first-install + renderer-only acknowledgment" model as a spec amendment.
