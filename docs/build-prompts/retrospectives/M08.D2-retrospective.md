## [END] Stage D2 special log (per `<retrospective_requirements>`)

### The pinned `validate_framework` wire

`validate_framework` ships (Stage B, `src-tauri/src/commands.rs:1394`) as `async fn validate_framework(doc: Value) -> FrameworkValidationReport` — params `{ doc }`, returning the report **directly** (not `Result`-wrapped; the command never errors — even a corrupt document produces a report with `schema_errors`). The `validateFramework` `ipc.ts` wrapper is `invoke<FrameworkValidationReport>('validate_framework', { doc })` — `{ doc }` matched the phase-doc assumption with zero drift.

`FrameworkValidationReport`, `NodeError`, `FrameworkCapabilitySummary`, and `SpawnEdgeNarrowing` are **hand-written Rust structs** (`#[derive(serde::Serialize)]`, snake_case field names — NOT schema-generated; they live in `crates/runtime-main/src/builder/{validate,summary}.rs`). They are **hand-mirrored** into `ipc.ts` (the `McpTool` / `ImportOutcome` precedent). Confirmed: `capability_summary` is a field on the report (`Option<FrameworkCapabilitySummary>` → TS `FrameworkCapabilitySummary | null`, `null` when schema validation fails), and it carries `spawn_edges: SpawnEdgeNarrowing[]`. `narrowed_caps` is a Rust `Result<Vec<CapabilityDeclaration>, String>` → serde externally-tagged → TS `{ Ok: CapabilityDeclaration[] } | { Err: string }`. **Key finding:** a `NodeError`'s `node_path` is always `"(root)"` (a whole-document schema failure) or a **bare agent id** (`gap.agent_id` for a reference gap, `edge.child_id` for a folded narrowing failure) — never a prefixed canvas node id — so the red badge keys on the bare agent id.

### Agent→Agent narrowing — Stage B's Rust report, NO TS intersection

Confirmed. `NarrowingNotice` reads the `SpawnEdgeNarrowing` triple off `report.capability_summary.spawn_edges[]` and renders whichever `narrowed_caps` arm the backend returned — the `Ok` surviving set or the `Err` rejection message — verbatim. It computes no intersection; the L2a `narrow()` lives only in `crates/runtime-main/src/capability/narrowing.rs` (spec §9). The contract test `surfaces_the_backend_narrowed_caps_verbatim_no_TS_intersection` (`NarrowingNotice.test.tsx`) is the guard: it scripts an `Ok` set carrying one declaration an empty `parent_caps` could never yield under a naive TS `child ∩ parent`, and asserts the rendered surviving set IS that backend value — a recomputing renderer would render `(none)`.

### Continuous validation — debounced, one validator two triggers

Confirmed. `builderStore.scheduleValidation` (a closure over the store's `set`/`get`, module-scoped `validateTimer`) debounces every `framework` mutation (`addNode`, `updateNode`, `connectEdge`) into one `validateFramework` call after `VALIDATION_DEBOUNCE_MS` (250ms); `a_burst_of_framework_mutations_fires_validateFramework_once` pins the coalescing. The same `validateFramework` (= the one Rust `validate_framework` command) is what Stage E's explicit **Validate** button will call with no debounce — one validator, two triggers (spec §9, no TS duplication).

### The `connectEdge` reject path + the `parseNodeId` helper

`connectEdge` reads each endpoint's kind off its node-id prefix via a new exported helper `parseNodeId(id) -> { kind, ref }` (`builderStore.ts`). It maps `(srcKind, tgtKind)` to one of the four edge types and returns the state unchanged for every other pair (no `framework` mutation → the pure edge projection paints no wire). `parseNodeId` is exported precisely so Stage E / M09 reuse one node-id parser rather than re-splitting `${kind}:${ref}` ad hoc — **a reusable helper the reject path needed**.

### Hook→Task — implemented, not canvas-drawable (closeout gap)

The Builder has no Task node in v0.1 (the Palette has 5 tabs — Tools/Skills/Agents/HITL/Hooks — and `BuilderNodeKind` has no `'task'`). `connectEdge` implements the `hook->task` case per the phase doc (pushes a `{ $ref }` entry to `task_defaults.post_hooks`) and `connect_hook_to_task_pushes_hook_id_to_task_defaults_post_hooks` unit-tests it directly with a synthetic `task:` target — but it is **not reachable from the canvas**. Logged for the M08 closeout gap-analysis (per the user's red-gate disposition).

Criterion 2 (`Agent→Skill edge adds the skill and paints a builder-edge wire`) and criterion 3 (`Agent→Agent edge whose child over-declares surfaces a narrowing rejection` — `.narrowing-notice__rejected` shows `net.fetch`) are demonstrated end-to-end, plus the badge half of criterion 4 (`an invalid framework paints a red badge`) and the reject path (`an invalid node-pair connection is rejected — no edge appears`).
