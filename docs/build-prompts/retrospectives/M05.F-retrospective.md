## [LIVE] Friction events

| When (commit / step) | What happened | What the prompt should have said | Severity (1–5) |
|---|---|---|---|
| Vitest round 1 (Zustand selector infinite loop) | The naive selector `useGraphStore((s) => s.nodes.filter(...))` returned a fresh array on every call. Zustand v5 + React 18 detected the unstable identity and triggered "Maximum update depth exceeded" / "The result of getSnapshot should be cached to avoid an infinite loop" — all 8 GapPanel tests failed before the React tree could even mount. Fixed in one round by wrapping the selector with `useShallow` from `zustand/react/shallow`. Pre-existing project pattern (the M03 phase doc names `useShallow` as the v4→v5 migration primitive) but the phase doc F.3.1 code sample didn't use it. | Phase doc F.3.1's sample uses `useGraphStore((s) => s.gaps)` which assumes a Map-typed slot. The actual store keeps gaps inside `state.nodes`, so any filter-based selector pays the Zustand v5 stable-reference tax. The phase doc should call out `useShallow` for any selector returning a derived array. Could surface as a graduated gotcha if M06+ adds derived-array selectors. | 2 |
| Vitest round 1 (CapabilityBadge title test stale-tier leak) | The `renders_P_letter_after_tier_transition_to_promoted` test mutates `currentTier` to 'promoted'; the subsequent `exposes_title_attribute_for_keyboard_hover_disclosure` test asserted the title contained 'novice'. Failed: `expected 'tier: promoted (1 grants)' to contain 'novice'`. The graphStore's `clear()` deliberately preserves `currentTier` (per-installation preference per spec §8.security L4), so vitest's test isolation guarantee doesn't hold for tier-mutating cases. Fixed by adding `useGraphStore.setState({ currentTier: 'novice' })` to both beforeEach and afterEach. | Stage D's contract that `clear()` preserves `currentTier` is correct per spec §8.security L4 but is a hidden test-isolation trap for downstream renderer tests. Could surface as a graduated gotcha: "graphStore.clear() preserves user-preference slots (currentTier); tests mutating those slots must reset them explicitly." | 1 |
| Playwright round 1 (Vite cold-start dep-optimizer timeout) | First test of the new spec file timed out at page.goto: `Test timeout of 120000ms exceeded`. Tests #2 and #3 of the same file passed at <6s each. Pattern matches gotcha #53 (Vite re-optimizes on first request to new code paths). Resolved by warming the dev server with a 16s curl probe BEFORE invoking Playwright. The 120s describe-block timeout was already comfortably above the typical 30–60s warmup, so the failure was specifically a cold-cache + first-spec-file scenario. | Gotcha #53 confirmed at M05.F (now 2 stages: M04.C plan_approval + M05.F gap_panel).
