## [LIVE] Seam choice + reframe pattern

Two implementation seams considered for "auto-layout fires on LOAD only":

1. **`replaceFramework` computes layout when it replaces the document.** Rejected ‚Äî `replaceFramework` is ALSO called by the JSON-tab edit (per its doc-comment + ADR-0020), where the user may be mid-typing; auto-layouting on JSON-tab swap would discard the user's manual node positions, violating the ADR-0020 "nodePositions is editor-local view state" invariant.
2. **A new `applyLoadedFramework(fw)` store action ‚Äî the load-only seam.** Adopted. Inspector.onLoad switches from `replaceFramework` to `applyLoadedFramework`; the JSON-tab edit path stays on `replaceFramework`. The third unit test (`replaceFramework_does_not_auto_layout_so_JSON_tab_edits_preserve_manual_positions`) pins this invariant ‚Äî it would BREAK if a future drift moved layout into `replaceFramework`.

The reframe at `tests/unit/components/builder/Inspector.test.tsx`
(`clicking_load_picks_a_directory_loads_and_replaces_the_framework`
‚Üí `..._and_applies_the_framework`) is the M08.6.C precedent: an
ADR/seam convention change reframes the assertion, NOT a regression.
The strict-TDD invariant binds only the new test files added in the
red commit (gotcha #66's TDD scope discipline); existing tests whose
assertions track the prior seam name are deliberately updated in the
impl commit with documented rationale.

## [LIVE] Layout-engine generic widening (no scope creep)

`src/lib/layout.ts::layoutGraph` was typed `(GraphNode[], GraphEdge[])
=> GraphNode[]` ‚Äî narrow to the live-graph store's discriminated union.
The function body reads only `node.id` + writes only `node.position`;
the narrow types were accidental. Widened to
`<N extends Node, E extends Edge>(N[], E[]): N[]` so the Builder's
narrower agent / tool / skill / hitl / hook `Node[]` flows through
unchanged. Both existing callers (`GraphCanvas`, `TesterGraphPane`)
continue to pass `GraphNode[]` and receive `GraphNode[]` back ‚Äî no
ripple. The phase doc D.3 explicitly anticipated this ("touched only
if the Builder node shape needs a bounds tweak"); the generic widening
is the cleanest enabler (one layout engine, two callers).

## [LIVE] tauri-driver test seam vs OS file dialog

The Inspector's Load button opens `@tauri-apps/plugin-dialog::open({
directory: true })` ‚Äî an OS-native dialog tauri-driver cannot drive
(WebDriver controls the WebView, not the OS file picker). The
real-app test invokes the production Tauri command +
the new `applyLoadedFramework` store action directly via
`browser.execute()`, using `window.__TAURI_INTERNALS__.invoke` +
`window.__builderStore` ‚Äî the EXACT code path `Inspector.onLoad` runs,
minus the dialog click. This mirrors the precedent at `src/App.tsx:50-52`
where `window.__graphStore` was exposed for
`tests/e2e/plan_approval.spec.ts` (per the comment on that line: "the
store carries no secrets, the same data is already inspectable via
React DevTools, and CLAUDE.md ¬ß9 anti-patterns calls out feature-flag
shims that don't earn their cost"). The Builder-store expose is the
same pattern with no production-only conditional.
