## Friction patterns that recurred (across A + B)

1. **PowerShell-via-Bash-tool command escaping.** Stage A hit it once (Severity 2) when running B.4 Select-String checks via a Bash heredoc that mangled `$_` and `$checks` as bash variables. Stage B avoided it by routing the same intent through the Grep tool directly. Pattern bit gotcha-class (build agent runs PowerShell shell + Bash-tool fallback for cross-stack work). Validates Stage B's deliverable: the v1.3 `<runtime_environment>` tag explicitly addresses this pattern, and its description includes the realistic Windows-PowerShell-via-Bash-tool case (per Stage A's carry-forward decision).

2. **Phase doc verbatim-prose hard-coded values.** Stage A almost shipped a `schemas/error.v1.json` with an `$id` URL (`kknipe2k.github.io/...`) that doesn't match the existing `schemas/*.v1.json` convention (`schemas.aria-runtime.dev/...`). Caught by the prompt's own `<gotchas>` trap (verify `$id` against existing schemas before authoring). Validates v1.3's `<fan_out_grep>` tag: a `<grep pattern='"$id"' purpose="verify schema $id base URL pattern"/>` would have caught this proactively at pre-flight. Stage B's `<fan_out_grep>` description was extended (per Stage A's carry-forward decision) to cover value-consistency verification alongside rename-fan-out.

3. **Time-box under-estimation.** Both stages estimated ~3.5h, both took ~0.5h (ratio 0.14×). Doc/protocol-only mini-milestone stages should estimate at the PR #48 ~1h-per-stage baseline, not at parent-milestone heuristics. Same finding from both stages; consistent.

## Patterns that worked durably (across A + B)

1. **Strict reference-first + verbatim-prose discipline.** Both stages used `<deliverable ref="..." section="..."/>` form pointing at the phase doc's X.3 / X.4 / Key constraints / X.6 sections; the Edit tool's exact-match anchors on the verbatim phase-doc text yielded zero `not unique` / `not found` errors across all 22 Stage A edits and 6 Stage B edits. The discipline is durable when the phase doc is authored with surrounding-context citations.

2. **Per-stage retrospective Decisions section as forward-information channel.** Stage A's `[END] Decisions for the next stage` section produced two specific carry-forward inputs (`<fan_out_grep>` value-consistency expansion + `<runtime_environment>` PowerShell-via-bash note) that folded cleanly into Stage B's v1.3 tag descriptions without redesigning the tags. The retrospective protocol demonstrated forward-leverage, not just backward-assessment.

3. **`<gotchas>` traps as safety valves.** Stage A's trap #5 (verify `$id` against existing schemas) caught the verbatim-prose hard-coded-value blind spot that the strict reference-first design otherwise wouldn't have surfaced. Validates the per-stage `<gotchas>` slot as a complementary discipline to reference-first, not a redundancy.
