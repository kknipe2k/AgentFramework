## Findings

### 🔴 #1 — Wire: L1 capability enforcer never invoked from production SDK

**Spec claim:** §8.security L1 — "every tool dispatch passes through a capability enforcer that reads the calling agent's capability declaration, looks up what the requested action declares it needs, and rejects mismatches with a `capability_violation` event". Phase doc M05.B.1 deliverable 3: "Wiring of the enforcer into the SDK's tool dispatch path + sub-agent spawn path"; B.2 (`sdk/mod.rs` row): "wrap tool dispatch + sub-agent spawn through enforcer"; B.3.4 shows the exact `dispatch_tool` wrap example.

**Observed:** `grep -rn '.check(' crates/runtime-main/src/ --include='*.rs'` returns 14 matches — all inside `crates/runtime-main/src/capability/enforcer.rs` `#[cfg(test)] mod tests` (lines 306–519). Zero production call sites. `crates/runtime-main/src/sdk/mod.rs` is edited at this milestone (adds the `request_capability` module per M05.A) but contains no enforcer construction, no `CapabilityEnforcer` reference, no `.check(` wrap. The streaming pipeline at `crates/runtime-main/src/sdk/event_pipeline.rs:57–66` translates `ProviderEvent::ToolUse` → `AgentEvent::ToolInvoked` *without* an enforcer check. The header comment of `crates/runtime-main/tests/capability_enforcer_smoke.rs` records the rationale: "v0.1 has no `dispatch_tool` path in the production SDK (M02-shipped single-turn streaming only)" — but the phase doc B.1/B.3.4 contract is unfulfilled at the SDK call site nonetheless.

**Expected (per spec):** A production code path in `crates/runtime-main/src/sdk/` constructs a `CapabilityEnforcer`, calls `.check(agent_id, &needed)` before dispatching the tool (or before processing the `ToolInvoked` event), and emits `capability_grant` on Ok or `capability_violation` on Err. The `tool_call_with_grant_succeeds_and_emits_capability_grant` integration test passes because it stands in as the "what SDK will call" surface; the real SDK never calls it.

**Trace / harness:** 5-step trace V.3 #2. Step 1 (claim) ✓. Step 2 (source event `ProviderEvent::ToolUse` / `AgentEvent::ToolInvoked`) ✓. Step 3 (no projection; enforcer runs in-process) ✓ by design. Step 4 (consumer: `enforcer.check` call before `provider.invoke`) **breaks — zero matching consumers in non-test code**. The trace endpoint per V.3 is the phase doc B.2 row for `sdk/mod.rs`; no such call exists in that file or its sibling files.

**Action:** Two routes — maintainer adjudicates:
1. **D.fix iter 1**: add the dispatch wrap in `sdk/agent_sdk.rs` `run_agent_with_provider_stream` so that processing a `ProviderEvent::ToolUse` first calls `enforcer.check(agent_id, &translate_tool_to_capability(name))`; emit `capability_grant` / `capability_violation` per outcome. Estimated 1–2h. Includes wiring `CapabilityEnforcer` into `AgentSdk` construction + grant population from `framework_loader`.
2. **Waiver-as-ADR**: build agent files `docs/adr/NNNN-waiver-M05-finding-1.md` documenting that v0.1's streaming-only SDK has no synchronous dispatch path to wrap (Anthropic's API dispatches tools server-side); the wire is structurally deferred until M06 multi-turn tool loops land; the smoke test stands in as the canonical "what-SDK-will-call" surface. Carry forward as M06 Stage A pre-flight.

Cite finding #1 in whichever path is chosen.

### 🔴 #2 — Wire: L2a `narrow()` never invoked from sub-agent spawn path

**Spec claim:** §8.security L2a — "sub-agent spawn grants ⊆ parent grants"; phase doc B.1 deliverable 3 + B.2 (`sdk/mod.rs` row): "wrap tool dispatch + sub-agent spawn through enforcer"; B.3.4 second paragraph: "Sub-agent spawn does the parallel narrowing check. Both call sites become the enforcement boundary."

**Observed:** `grep -rn 'narrow' crates/runtime-main/src/ --include='*.rs'` returns matches only inside `crates/runtime-main/src/capability/narrowing.rs` (unit tests + proptest at lines 94–257). `AgentEvent::AgentSpawned` is emitted unconditionally at `crates/runtime-main/src/sdk/agent_sdk.rs:124` with no narrowing call. No production code path constructs a `Vec<CapabilityDeclaration>` proposed-for-child and calls `narrow(parent, proposed)`.

**Expected (per spec):** A production spawn path in the SDK calls `narrow(&parent_grants, &proposed_child_grants)` before emitting `AgentSpawned`; on Ok proceeds and emits `capability_grant` (with `narrowed_from` populated); on Err emits `capability_violation` and routes through HITL.

**Trace / harness:** 5-step trace V.3 #3. Step 4 (consumer: SDK spawn path calls `narrow(parent_grants, proposed)`) **breaks — zero matching consumers**. Trace endpoint per V.3 is the phase doc B.2 row for `sdk/mod.rs`; no narrow call exists.

**Action:** Same two routes as Finding #1 — D.fix iter 1 adds the narrowing call to whichever sub-agent spawn path is in scope for v0.1 (M02 streaming may not actually spawn sub-agents; verify before patching) OR waiver-as-ADR documents the deferral to M06+. Findings #1 and #2 are the same architectural class — likely shipped together.

  - The four-pass shape (Inventory + Wire + Behavior + Multi-call) felt adequate at M05's scope.
