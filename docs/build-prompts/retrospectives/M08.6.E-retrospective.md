## [LIVE] Design decisions made

**Three decisions made unilaterally per CLAUDE.md §12 "Own technical
decisions; escalate only on user-domain forks":**

1. **Source enum field on `PaletteItem`.** Added a
   `source: 'builtin' | 'installed' | 'framework'` discriminant + a
   `data-source` HTML attribute on the `<li>`. The phase doc E.3 said
   "a source label / data attribute" (OR); the data attribute is the
   testable invariant (asserted by 3 of the 7 new tests). HITL trigger
   types + hook firing points are also `'builtin'` — they are runtime
   primitives, not artifacts the user installs.

2. **Dedupe precedence: built-ins → installed → framework.** The phase
   doc didn't specify which source wins on a (kind, ref) collision.
   Reasoning: installed lock entries carry version pins + install
   timestamps + L3 reports; a framework declaration is a document
   reference. Installed wins → the surviving item's `data-source` is
   `'installed'`. The `palette_de_duplicates_a_kind_ref_present_in_both_installed_and_framework`
   test pins this precedence so a future drift surfaces. (The
   built-ins-first ordering is structural — built-ins exist in all
   sessions, so they should never be displaced.)

3. **Visible badge for framework-sourced items.** Added an inline
   `<span> · framework</span>` after the label for `source === 'framework'`
   items, separately from the `data-source` attribute. Rationale:
   "distinguishable" in the phase doc means the user sees the
   distinction — a data attribute alone is invisible to a normal user.
   The label flow is unchanged for the existing filter test (the
   filter matches `it.label.toLowerCase().includes(needle)`; the
   inline badge is a separate sibling element).

None of the three are user-domain forks (scope / priority /
irreversible architecture). All three are documented in code and pinned
by tests; a future change is forced through a test reframe.

## [LIVE] useShallow consideration (gotcha #75)

The new `useBuilderStore((s) => s.framework)` selector returns a single
object reference — `s.framework` is the document itself, not a derived
array/filter/map. Per gotcha #75, `useShallow` is the fix for derived
selectors whose elements re-create per render; a bare-object selector
returns the same reference across renders until `set()` swaps it, so
reference equality is the right cache key. The `useMemo` over
`paletteItemsForTab(tab, installed, framework)` then takes the stable
references and memoizes the projected items list. No infinite-loop
risk; no `useShallow` import needed.
