### rmcp 1.7.0 API surface vs phase-doc illustrative shapes — **THREE drifts logged as one entry per user direction**

**Drift 1: stdio child-process feature flag.** Phase doc B.3.1 prescribed `features = ["client", "transport-io", "transport-streamable-http-client-reqwest"]`. Actual rmcp 1.7.0: `transport-io` is the SERVER-SIDE stdin/stdout helper (tokio's `io-std` feature); for CLIENT-SIDE child-process spawning via `TokioChildProcess`, the correct feature is `transport-child-process`.

**Drift 2: HTTP transport TLS pin.** Phase doc didn't anticipate that `transport-streamable-http-client-reqwest` alone enables only `__reqwest` (private dep feature) without pinning a TLS backend. For rustls-based TLS, the consumer must additionally enable the `reqwest` feature (alias that turns on `reqwest?/rustls`). Final feature list: `["client", "transport-child-process", "transport-streamable-http-client-reqwest", "reqwest"]`.

**Drift 3: HTTP transport type name.** Phase doc B.3.6 named `rmcp::transport::StreamableHttpClient`. Actual rmcp 1.7.0 type: `rmcp::transport::StreamableHttpClientTransport` (the `Transport` suffix is part of the canonical name).

### Transport-trait shape suitability for Stage C consumption

The `Transport` (factory) + `Connection` (live handle) trait pair is clean for Stage C lifecycle wiring:

- `Transport::connect() -> Result<Box<dyn Connection>, McpError>` is the install/connect surface Stage C wires from the `mcp_servers` registry. The `Box<dyn Connection>` return type lets Stage C hold a heterogeneous map of `BTreeMap<McpServerName, Box<dyn Connection>>` for the connected-server registry.
- `Connection::list_tools() -> Result<Vec<McpTool>, McpError>` is what Stage C calls after connect to populate the §5a tool namespace (Stage D's input).
- `Connection::call_tool(name, args) -> Result<Value, McpError>` is what Stage D's dispatch path will call.
- `Connection::ping() -> Result<(), McpError>` is what Stage C's health-ping loop calls. Currently implemented as a `list_tools` round-trip (no dedicated rmcp client-side ping); Stage C can budget the call frequency knowing this is a tool-list-sized request.
- `Connection::shutdown() -> Result<(), McpError>` is a no-op at the trait level — `rmcp::service::RunningService::cancel` takes `self` by value, but `Connection` takes `&self`. **Forward-applicable note for Stage C:** when Stage C owns the `StdioConnection` / `HttpConnection` by value (in the registry's `BTreeMap`), it can downcast to call `cancel()` directly OR rely on Drop to tear down the subprocess. **Decision for Stage C:** evaluate whether the registry should own `Box<dyn Connection>` (current trait shape) or own concrete `enum ConnectionVariant { Stdio(StdioConnection), Http(HttpConnection) }` (would enable direct `.cancel()` calls). My recommendation: **keep the Box<dyn Connection> pattern + add an explicit `Connection::shutdown_owned(self: Box<Self>) -> Result<(), McpError>`** method to the trait surface in Stage C, which preserves the heterogeneous-map pattern while exposing the clean teardown path. This is a Stage C design decision; not Stage B scope.

### Coverage outcome on the two transport files (stdio + http)

**Pattern adopted: M02.C `anthropic.rs` exclusion, NOT M05.C2 `seccomp/landlock` exclusion.**

The difference:
- **M02.C pattern (chosen here):** The wrapped resource is a network/process service whose protocol handshake cannot be faithfully mocked without reimplementing the protocol. The wrapper is excluded from the gate; wire-format logic lives in a separate file (M02.C's `providers/anthropic_sse.rs`) which IS in-gate via wiremock-driven tests. **For runtime-mcp, the equivalent "wire-format logic" lives in the rmcp upstream crate — not in our codebase — so the wiremock-style separation isn't applicable.** Our test surface is: (a) wrapper-level tests (`build_command`, error-path connect failures via nonexistent command / wiremock-404); (b) the in-process scripted mock for downstream consumers' tests; (c) the feature-gated integration test against the real reference MCP server.
- **M05.C2 pattern (NOT chosen):** OS-fence-installing functions (`seccomp::install`, `landlock::restrict_self`) whose bodies install per-process restrictions on the calling test process — running them in-process would poison the test runner. Forward path: seam decomposition (`install_with(load_fn)`) lifts the testable layer back into the gate.

For `runtime-mcp`'s stdio + http transports, **seam decomposition won't help** — the OS calls aren't "install a fence" but "spawn a process speaking JSON-RPC" or "open a TLS connection negotiating MCP-over-HTTP." The protocol handshake is what makes happy-path testing structurally hard, not an OS-side-effect on the test runner.

**Stage C+ enhancement opportunity:** Build a tiny stub MCP server binary in the test fixtures (a few hundred lines, just enough to answer `initialize` + `tools/list` per the MCP spec). With that binary, `stdio.rs::connect()` happy path becomes unit-testable (spawn the stub, connect, assert success); the gate could re-include `stdio.rs`. Same for `http.rs` with a wiremock that satisfies the streamable-HTTP MCP handshake (less likely to be feasible — wiremock isn't designed for stateful protocol mocks). **Out of scope for Stage B** per the budget; Stage C/D/E may revisit if the feature-gated integration test proves insufficient for catching regressions.
