## [END] Special log (per the stage's `<special_log>`)

**DACL research outcome + the option picked.** tokio 1.52.1 `ServerOptions::create` passes a null `lpSecurityAttributes` to `CreateNamedPipe`; per Microsoft's "Named Pipe Security and Access Rights", the default descriptor grants full control to LocalSystem/administrators/creator-owner and **read access to Everyone and the anonymous account** — the in-code "owner only" claim was false. Maintainer picked **(b1)**: corrected citation comment (drone `ipc.rs` module doc; sandbox points to it), `reject_remote_clients(true)` pinned at all four `ServerOptions` sites (tokio's documented default, pinned against CVE-2023-22466-class clobbering), owner-only descriptor routed to the M12 security ADR. (b2) waiver-ADR not authorized; (b3) safe-crate rejected (only dormant candidates: `windows-acl` 2018, `windows-permissions`). **Rider 1 standing for G:** the read-eavesdropping residual gets its own TD entry; TD-053's resolution language stays qualified ("resolved — …; read-eavesdropping residual → TD-0XX (M12)").

**Realistic max-frame evidence behind the cap.** Largest legitimate frames: `SessionRecovered` (full snapshot `state_json` + plan/task projections), `SignalLog` (entire session signal log), `QueryResult` (rows incl. `snapshots.state_json`); sandbox ceiling is `ValidateArtifact.artifact_code`. Observed frames today are KB-scale (the production `SnapshotNow` carries `{"agent_id": …}`); the bounding future case is full-context conversation state ≈1–2 MB escaped JSON. 4 MiB (the review's number) gives ≥2× margin; evidence lives in the constant's doc comment. **Rider 2 resolved:** the limit is **delimiter-exclusive** (pinned from the tokio-util 0.7.18 source, `lines_codec.rs:151` — error at `buf.len() > max_length` with no `\n` in the first `max_length + 1` bytes); exactly-4 MiB content + `\n` round-trips at all five layers, 4 MiB+1 unterminated errors. **Rider 4 resolved:** decode-side-only enforcement stated once, in the constant's doc.

**The error arms went from dead to live** (the review's exact phrase). Red proved the dead state at every layer (the real drone accepted 4,194,305 unterminated bytes and kept serving heartbeats past the deadline); post-impl the same tests observe: drone `Alert` carrying "max line length" + connection dropped + **the next connection served** (`SnapshotWritten` on a fresh client); sandbox `handle_connection` returns `false`; both main-side connections surface typed `Codec("max line length exceeded")` through the pre-existing `From` impls. Non-panic at every consumer, verified by reading each arm at grounding and by the green runs.

**Subprocess-test friction on Windows-local.** Two distinct classes, both closed in-stage: (1) the orphan-inherits-stdout-handle pipeline hang at red (closed by `kill_on_drop(true)`; cross-ref the MCP node-orphan observation — same failure-path process-cleanup theme); (2) the only-if-absent fixture staleness that would have green-lit the uncapped binary post-impl (closed structurally by `27a40a4`: unconditional `cargo build`, ~1s no-op when fresh, rebuild-on-source-change verified once by mtime + green loopback test).

**Mutation gate, two rounds (blocking — critical modules).** Round 1 (15 mutants, `--in-diff --in-place --cap-lints true`): 5 caught, 2 unviable, **8 missed** — 4× const-operator (`4 * 1024 * 1024` → `+`/`/`) because cargo-mutants tests only the mutated package and runtime-core's own suite never exercised the cap; 2× `reconnect → Ok(())` (pre-existing: no test exercised a *successful* reconnect through the `open()` seam); 2× cfg(unix) `accept_loop → Ok(())` (don't compile on this host). Additive killers (`c0753e3`): an in-package value-pin (`MAX_IPC_FRAME_BYTES == 4_194_304`) and cross-platform reconnect-success tests against real temp socket/pipe endpoints. Round 2: **11 caught, 2 unviable, 2 missed — both misses the platform-bound cfg(unix) `accept_loop` mutants**, justified inline: on Linux, `accept_loop → Ok(())` makes `serve` return immediately, so the existing in-crate server tests panic at `open_client` and the new `unix_socket_mode_is_0600_after_bind` tests fail; the nightly ubuntu mutants workflow (TD-056) is the standing check. No surviving mutant on the cap or the error arms.
