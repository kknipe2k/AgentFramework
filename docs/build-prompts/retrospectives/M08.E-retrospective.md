## [END] Stage E special log (per `<retrospective_requirements>`)

### The pinned `save_framework` / `load_framework` wire + the `LoadedFramework` shape

`save_framework` ships (`src-tauri/src/commands.rs:1411`) as `async fn save_framework(dir: String, framework: Framework, companions: Vec<Companion>) -> Result<(), CmdError>` — **three** params `{ dir, framework, companions }`, NOT the phase-doc-assumed `{ dir, fw }`. `load_framework` ships (`commands.rs:1440`) as `async fn load_framework(dir: String) -> Result<LoadedFramework, CmdError>` — `{ dir }` → `LoadedFramework`. `Companion` (`crates/runtime-main/src/builder/persist.rs:27`) is `{ file_name: String, body: String }`; `LoadedFramework` (`persist.rs:42`) is `{ framework: Framework, companions: Vec<Companion> }`. Both are **hand-written Rust structs** (`#[derive(serde::Serialize/Deserialize)]`, snake_case) — NOT schema-generated — so they are **hand-mirrored** into `ipc.ts` (the `McpServerSummary` / `ImportOutcome` precedent). The `saveFramework` wrapper exposes `companions` with a `[]` default — the v0.1 canvas authors no inline markdown bodies (M09's Generators will); the Inspector's Save calls `saveFramework(dir, framework)`. The `{dir,fw}`→`{dir,framework,companions}` B↔E phase-doc drift is logged for the closeout gap-analysis (per the user's red-gate disposition).

### The Canvas | JSON two-way binding falls out of ADR-0020 — no bespoke sync machinery

Confirmed. `builderStore.framework` is the single source of truth (ADR-0020); the canvas (D1/D2) and the JSON tab (E) are two editors over it. Canvas → JSON: a canvas edit mutates `framework`; `JsonView`'s `useEffect` re-seeds the draft. JSON → Canvas: a valid edit calls `replaceFramework`; the canvas projection (`canvasNodes`/`canvasEdges`, pure functions of `framework`) re-derives — verified by the `replaceFramework_causes_the_canvasNodes_projection_to_re_derive` test. The only thing E adds beyond "render `framework` / call `replaceFramework`" is the invalid-JSON rejection path + a `useRef` self-edit guard (so the JSON tab's own valid edit does not re-pretty-print the user's in-progress draft). No two-way-sync state machine.

### The Validate button shares the D2 `validate_framework` validator — one validator, two triggers

Confirmed. The Inspector's `onValidate` calls `validateFramework` — the SAME `ipc.ts` export `builderStore.scheduleValidation` (D2's debounced continuous pass) calls. There is no second TS validator. The `the_validate_button_uses_the_same_validate_framework_the_continuous_pass_uses` contract test (gotcha #66) scripts a backend report carrying a verdict no TS-side validator would invent and asserts the Inspector surfaces it verbatim — spec §9, no duplication.

### The capability summary is read from the report FIELD, not a separate command

Confirmed. The Inspector reads `(report ?? validation)?.capability_summary` — the `capability_summary` field on `FrameworkValidationReport` (Stage B B.3.4). There is no `framework_capability_summary` command call in the renderer.

### The invalid-JSON path does NOT call `replaceFramework` — no desync

Confirmed. `JsonView.onChange` parses the draft; on a `JSON.parse` throw it sets an inline parse error and `return`s — `replaceFramework` is never reached. The `an_invalid_json_edit_does_NOT_call_replaceFramework` test (a spied `replaceFramework`) and the Playwright `invalid JSON surfaces a parse error and leaves the canvas unchanged` test both pin it.

### The Test button is inert-but-wired

Confirmed. The Inspector's Test button calls `builderStore.openTester` (E ships `testerOpen` + `openTester`/`closeTester`); Stage F2 delivers the Tester modal that renders on `testerOpen`. E's tests assert the `openTester` call, not a modal — the M07.E incremental-construction precedent.

### MVP §M8 criteria 4 / 6 / 7 / 8 — Playwright result

`tests/e2e/builder_inspector.spec.ts` — **5/5 passing**. Criterion 4 (`Validate runs validate_framework and surfaces the capability summary`), criterion 6 (`a JSON-tab edit round-trips to the canvas` + `invalid JSON surfaces a parse error and leaves the canvas unchanged` — the no-desync guard), criterion 7 (`Save calls save_framework with the picked directory`), criterion 8 (`Load reads a directory and the canvas re-derives`) all demonstrated end-to-end against the Vite dev server with `@tauri-apps/api` + the `@tauri-apps/plugin-dialog` picker mocked through the `__TAURI_INTERNALS__.invoke` boundary.
