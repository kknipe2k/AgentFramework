### The Builder Canvas is a NEW interactive component

Confirmed — `git diff --name-only main..HEAD | grep -E 'GraphCanvas|components/nodes/'` returns nothing: **`src/components/GraphCanvas.tsx` and `src/components/nodes/*.tsx` are untouched.** The Builder Canvas is `src/components/builder/BuilderCanvas.tsx` + five new `src/components/builder/nodes/Builder*Node.tsx` components; they reuse the §3 node CSS class families (`agent-node` / `tool-node` / …) but are distinct interactive components. `builderNodeTypes` is module-level (the GraphCanvas re-mount trap — pinned via `<existing_pattern_audit>`).

### The M05 / M07.E capability-disclosure surface was REUSED — via a behavior-preserving extraction

The M07.E `import-capability-disclosure` surface was coupled to `ImportPanel.tsx`'s import-review modal. D1 lifted it into a shared `src/components/CapabilityDisclosure.tsx` — a behavior-preserving extraction, not a rebuild. The component takes plain-English `capabilities: string[]` lines + an `emptyMessage`; this generic interface serves **both** `ImportPanel` (its flat declared-capability list) and `BuilderAgentNode` (the agent's `allowed_*` rendered as plain-English lines). **`ImportPanel.tsx` now consumes the extracted component** — the `<ul className="import-capability-disclosure">` + `<li>` rendering and the `data-testid` are preserved verbatim; the empty-state `<p>` moved from the modal-specific `import-review-modal__none` class to the surface's own `import-capability-disclosure--empty` (same muted-text appearance, same text, same testid — no ImportPanel test asserted the class). All pre-existing ImportPanel tests pass unchanged.

### The framework → canvas projection (the ADR-0020 model in code)

`builderStore.canvasNodes()` derives the React-Flow `Node[]` from `framework` + `nodePositions` — node *existence* from `framework` (so Stage E's load reconstructs the canvas), positions from the editor-local `nodePositions`. **Shape friction noted:**
- **Memoization beyond `useShallow`.** `canvasNodes()` runs inside a render selector; `useSyncExternalStore` calls it repeatedly per render for its consistency check, and a fresh element array each call infinite-loops — `useShallow` alone does not break it because the projected element objects are fresh. The projection is memoized on the `framework` + `nodePositions` identities so the selector returns a referentially stable array. A test pins this (`canvasNodes_returns_a_stable_reference_when_inputs_are_unchanged`). This is an implementation refinement of ADR-0020's stated `useShallow` mitigation, not a contradiction of the model.
- **The pre-valid builder framework.** `builderAgent()` writes a minimal agent (`{id,role,model,allowed_tools,allowed_skills,spawns}`) into `framework.agents`; the generated `Agent` type models the complete valid shape, so one documented `as unknown as Agent` boundary cast is needed — the same pre-valid stance `emptyFramework()` already documents. D2's continuous validation surfaces the gap.
- **All five kinds project in D1** (resolving the D1.3.3/D1.4.1 contradiction): Agent → `framework.agents`, Tool → `framework.tools`, Skill → `framework.skills`, HITL → `framework.hitl_policy`, Hook → `framework.hooks`.

### MVP §M8 criterion 1 — Playwright result

`tests/e2e/builder_canvas_nodes.spec.ts` — **6/6 passing**. The criterion-1 path is demonstrated end-to-end: switch to Builder → drag an Agent Palette item onto the empty canvas (a `__TAURI_INTERNALS__` mock supplies one installed agent; the drag threads one `DataTransfer` through `dragstart`→`dragover`→`drop`) → the agent node appears → click it → the inline config panel opens → set role + add a tool → the plain-English capability disclosure renders below the node.

### `addNode` idempotency + projection re-derivation under repeated drops

Behaved cleanly. `addNode` is keyed on `${kind}:${ref}` in `nodePositions` and returns early on a re-drop — no duplicate `framework` entry, no position reset. Verified by `addNode_is_idempotent_on_re_drop_of_the_same_item` (vitest) and `re_dropping_the_same_palette_item_does_not_duplicate_the_node` (Playwright — `toHaveCount(1)`). The projection re-derives correctly: the memoized `canvasNodes()` invalidates on every `framework`/`nodePositions` change and returns a stable reference otherwise.
