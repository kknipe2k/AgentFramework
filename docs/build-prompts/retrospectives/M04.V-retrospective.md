## Findings

### 🟡 #1 — Behavior: TaskNode empty-title fallback unprotected

**Spec claim:** §3a "TaskNode (children): Status drives color … HITL flag rendered as a badge." (M04 Phase doc background, LG-02 IRL finding: "TaskNodes from `task_started` events showed up 'untitled' in the inspector + canvas.")
**Observed:** `src/components/nodes/TaskNode.tsx:27` ships the fallback `const displayTitle = title || \`task ${taskId.slice(0, 8)}\`;`. `tests/unit/nodes/TaskNode.test.tsx` (9 tests) never renders TaskNode with `title: ''`. Coverage report confirms line 27 uncovered.
**Expected (per spec / IRL fix):** A regression test rendering with empty title and asserting `task <id-prefix-8-chars>` displays in `.task-node__title` and the `aria-label` mirrors the fallback.
**Trace / harness:** Vitest + jsdom render of `<TaskNode data={{ ...baseData, title: '' }} />`; assert `screen.getByText(/task [a-f0-9-]{1,8}/)` is in document.
**Action:** Carry forward to M05 Stage A. Add `it('renders_task_id_prefix_fallback_when_title_is_empty', …)` to `tests/unit/nodes/TaskNode.test.tsx`. Estimated ~10 min.

### 🟡 #2 — Wire: §4a spec text says `hook_*` events; codebase ships `verify_*` events

**Spec claim:** §4a "Hook events (added to Phase 2 union)": `hook_started`, `hook_passed`, `hook_failed`.
**Observed:** `schemas/event.v1.json` lines 332–369 declare `verify_started`, `verify_passed`, `verify_failed`. These cover ALL `HookCategory` values (verify, lint, build, test, custom) with `category` as a discriminator field, not just the `verify` category. `src/components/nodes/VerifyNode.tsx` reads these and renders pass/fail/active. `crates/runtime-core/src/generated/event.rs` reflects the same shape.
**Expected (per spec):** Either §4a spec text updates to `verify_*` (single-family naming covering all hook categories), OR the event taxonomy reshapes to `hook_*` (matching the spec text). Currently spec and code disagree on the event family name.
**Trace / harness:** Wire-pass step 5 check: spec.§4a names `hook_*`; `event.v1.json` schema names `verify_*`; renderer + projector consume `verify_*`. The Wire is internally consistent; spec/code disagree on naming convention.
**Action:** Carry forward to M05 Stage A. Spec §4a should be reconciled with the codebase reality (the M04 phase doc Background already notes this as an audit-driven adoption). Recommended: file a v1.0 schema-evolution note alongside the existing M04 Stage F `budget_warn` vs `budget_warning` reconciliation (the spec §2a v0.1 implementation note documents the same kind of drift). Either rename events to `hook_*` in `event.v1.1.json` OR update §4a text to match `verify_*`. Decision is user-domain (product nomenclature).

### 🟢 #3 — Behavior: viewport-width (LG-04) fix unpinned by any test

**Spec claim:** Phase 3 "Visual Design Principles: No clutter — only show detail on hover or selection" + spec §1 implicitly assumes the graph canvas occupies the renderer's available width. M04 IRL: "View screen too small — needs to fill width of window." Fix landed in `src/styles.css:38` removing `max-width: 720px` from `main`.
**Observed:** `main` element CSS (`src/styles.css:38`) shows `margin: 1rem; padding: 0;` — no `max-width` constraint. Correct per the fix. No unit/E2E test asserts the canvas fills available width or that `main` lacks `max-width`.
**Expected (per spec / IRL fix):** Either a static-CSS-grep test (mirror of `every_status_class_has_a_corresponding_CSS_rule_in_styles_css`) asserting `main` has no narrow `max-width`, OR a Playwright assertion at viewport width ≥1280 that the graph canvas occupies expected pixel range.
**Trace / harness:** Gotcha #70 codifies that viewport-dependent CSS isn't catchable in Vitest+jsdom; Playwright at multiple viewport configs is the proper harness.
**Action:** Log to `docs/tech-debt.md` under "test-coverage gap: viewport-dependent CSS." M05 Playwright suite (if added) could include viewport-sweep tests for the graph canvas. Low-stakes — the IRL fix is in production code and visually self-evident on launch.

### 🟢 #4 — Multi-call: per-method twice-in-sequence tests missing for `read_signals` and `recover_session`

**Spec claim:** §1d IPC channels: "framed JSON over Unix socket / named pipe"; implied invariant that any IPC method survives sequential invocation. M04 IRL SQL-02 surfaced the single-use bug; PR #64's `query_session_db_succeeds_twice_in_sequence` pins it for one method; the `next_event` refactor (`connection.rs::next_event_returns_consecutive_events_without_consuming_reader`) pins it structurally.
**Observed:** `crates/runtime-main/src/drone_ipc/client.rs` has no `read_signals_succeeds_twice_in_sequence` or `recover_session_succeeds_twice_in_sequence`. Both methods invoke `next_event` under the hood; the structural invariant is pinned at the connection level.
**Expected (per gotcha #69):** Per-method belt-and-suspenders sequential-call tests for `read_signals` + `recover_session` mirroring `query_session_db_succeeds_twice_in_sequence`.
**Trace / harness:** Rust integration test pattern from `query_session_db_succeeds_twice_in_sequence` at `client.rs:542` — duplex-pair fixture, two sequential calls, assert distinct expected payloads.
**Action:** Log to `docs/tech-debt.md` under "test-coverage gap: per-method multi-call invariants." Consider rolling in at the next test-rationalization pass. ~20 min for both.

### 🟢 #5 — Multi-call: `respond_uncertainty` lacks soft-Ok-on-stale test

**Spec claim:** §1b Recovery: user decision is recorded as `tool_call_uncertainty_resolved`. Implied that a stale `invocation_id` from the renderer (timeout fires between display + click) should not 500 the renderer — analogous to the `respond_hitl_with_no_pending_awaiter_returns_ok` and `approve_plan_with_no_pending_awaiter_returns_ok_with_warn` patterns documented at `src-tauri/src/commands.rs:1193` and `:999`.
**Observed:** `src-tauri/src/commands.rs::respond_uncertainty_command_with` tests cover success path (`:1109`), unknown-action error (`:1123`), and drone-error propagation (`:1141`). No `respond_uncertainty_with_no_pending_*` analogue.
**Expected:** Test exercising a stale or unknown `invocation_id` and asserting the command soft-Oks (matches HITL + plan-control pattern) OR documents that respond_uncertainty's contract differs.
**Trace / harness:** Mirror of `respond_hitl_with_no_pending_awaiter_returns_ok` at commands.rs:1193.
**Action:** Log to `docs/tech-debt.md` under "test-coverage gap: stale-id behavior parity." First-class question: is respond_uncertainty's intended contract identical to respond_hitl's soft-Ok-on-stale? Decision is user-domain; tech-debt entry should record the question.
