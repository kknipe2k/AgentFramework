### M06.D `<scope_change>` #1 + #2 — status

**CLOSED at the seam + injection-seam level (per ADR-0011), NOT a deferred descope.**

- **#1 (SDK run-loop MCP wiring):** `AgentSdk` now holds `Option<Arc<dyn McpToolDispatch>>` + a `with_mcp_dispatch` builder; `run_agent_with_provider_stream` intercepts `ProviderEvent::ToolUse` and calls `dispatch_if_mcp` FIRST (`None`→Stage A fall-through unchanged; `Some(Ok(Invoked))`→agent_id-correct events; `Some(Ok(Blocked))`→`apply_mcp_dispatch`+existing `on_capability_violation` HITL; `Some(Err)`→`mcp_dispatch_error_event`). Verified by `mcp_dispatch_runloop.rs` (4 tests, mock dispatch) + the `run_smoke_session_with` injection-seam test. **M06.V trace #11a expects DELIVERED.**
- **#2 (agent_id-correct Invoked):** approach taken = **run-loop-emits-Invoked-events-directly** (NOT a signature change). The run loop holds `agent_id` and emits `ToolInvoked`/`ToolResult` itself with it; `apply_mcp_dispatch` + `McpDispatchOutcome::Invoked { server, tool, value }` are left **untouched**, so the D-frozen `mcp_dispatch_wire.rs` integration test stays green. Rationale: F.3-preferred + gotcha #68 trap (don't perturb the frozen `{server,tool,value}` pattern). The load-bearing test asserts `agent_id` is **non-empty AND == the run-loop agent ("worker")** on BOTH `ToolInvoked` and `ToolResult`.

The residual (concrete `McpDispatcher` construction + `impl ConnectionResolver for McpClient` + live agent-loop exercise) is the **explicit ADR-0011 M07 carry-forward** (trace #11b) — documented, not silent.

### src-tauri injection — new seam or reused?

**Reused/extended an existing `*_with` seam — no new seam.** `run_smoke_session_with` (the M01.C/M02.C `*_with` archetype) gained one `Option<Arc<dyn McpToolDispatch>>` parameter applied via `.with_mcp_dispatch`; the 3 call sites (production wrapper + 2 existing tests) pass `None`. The production `run_smoke_session` wrapper passes `None` (ADR-0011: concrete construction = M07). No new Tauri command, no new managed-state, no new HITL/seam primitive (honors ADR-0007 + F `<execution_warnings>`).

### Ambiguous routes via the D-frozen `apply_mcp_dispatch` (per the maintainer's added retro item — non-blocking)

**Confirmed.** `try_mcp_dispatch`'s `Ok(outcome)` arm (everything except `Invoked`) calls the **untouched D-frozen `apply_mcp_dispatch(outcome, args)`**. For `McpDispatchOutcome::Ambiguous { name, candidates }` that emits a single `AgentEvent::ToolAliasAmbiguous { name, candidates }` — a variant with **no `agent_id` field**, so the gotcha #68 empty-agent_id concern does not apply to it (only `Invoked` carried that risk, and `Invoked` is now run-loop-emitted). `outcome_needs_hitl(&Ambiguous)` is `false`, so Ambiguous is **non-blocking** (no HITL await) — it surfaces the ambiguity to the renderer and the run loop continues, exactly as the D-frozen mapping + `outcome_needs_hitl` (Blocked-only) intend. The Stage F wire neither special-cases nor perturbs the Ambiguous path; it rides the frozen mapping verbatim.
