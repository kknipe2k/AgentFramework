## [END] Special-log entries (`<retrospective_requirements><special_log>`)

### Tier-eval display surface — was a Stage D retroactive Tauri command needed?
**No.** Stage D shipped no MCP tier-eval command and none was added. The Add modal computes the outcome renderer-side from the store's `currentTier` (M05.D-driven, `tier_transition` events) + the fixed M06.D §8.security-L4 rule (MCP tool calls run with the `Exec` capability, which the Novice tier forbids; Promoted permits). `currentTier === 'promoted'` → "auto-accept on install"; else → "requires Promoted tier". No new Tauri command, no new tier primitive (honors the execution_warning). This is the cleanest reuse and avoids a redundant round-trip — the tier is already in the store for `CapabilityBadge`.

### CapabilityBadge reuse for the MCP server tier surface — awkwardness as it scales
The M05.F `CapabilityBadge` could **not** be reused as a component: it filters `capabilityGrants` by `agentId`, and the MCP-install context has no agent. What generalizes is the *pattern* — "read `currentTier` from the store; render a `--<tier>` class-modified element; compute the L4 consequence renderer-side." The Add modal re-implements that pattern (`mcp-server-add-modal__tier-eval--<tier>`). As more surfaces need tier display (M07+ registry import, M08 Builder), the pattern is worth extracting into a small `useTierEval(capabilityKind)` hook so the L4 rule lives in one place rather than being re-encoded per surface. Flagging now while it's 2 occurrences (CapabilityBadge + this) — extraction is premature at 2 per §9 (wait for the fourth) but the recurrence is on the radar.

### Settings panel routing — the M04.E source-of-truth
There is **no** M04.E "Settings panel." M04.E shipped `HITLPanel` + `HITLModal` (non-modal panel + modal dialog for HITL prompts) — not a tabbed Settings container. The actual app-shell pattern is independent sibling panels rendered directly into `.graph-layout` (`ApprovalPanel`, `HITLPanel`, `GapPanel`). `MCPServerSettings` follows that — mounted as a sibling panel, no tab routing. The phase-doc E.3.2 "Three-region layout matching the existing Settings panel pattern (M04.E)" and the E.4.4 Playwright `[data-test=open-settings]`/`settings-mcp-tab` pseudocode are drift; reconciled to actual DOM (Playwright uses `window.__graphStore` injection per gotcha #54, exactly like `gap_panel.spec.ts`). If v1.0 wants a unified Settings container with tabs, that is net-new app-shell work, not an M06.E deliverable.
