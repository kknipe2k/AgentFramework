### 🔴 #7 (`mcp_servers` schema divergence) — REFRAMED-AS-INERT (M08.5.5)

Reframing rationale (recorded per the M08.5.5 C.fix red-phase + the M08.5.5-summary § "M08.6 stage prompts — known constraints to encode"):

- **Canonical schema (`session.sqlite::mcp_servers`)**: `crates/runtime-drone/migrations/003_mcp_server_status.sql:25-45` — 4-state CHECK `('connected', 'disconnected', 'health_pending', 'error')`, DEFAULT `'disconnected'`. This is the **live runtime database**, opened on every launch per ADR-0012.
- **Stray schema (`mcp.sqlite`)**: 5-state CHECK `('configured', 'connected', 'errored', 'disabled', 'failed')`, DEFAULT `'configured'`. This was the pre-ADR-0012 legacy shape; the M08.5.5.C.fix cleanup module renamed every existing copy to `.stray-mcp.sqlite.bak`.
- **The divergence is between an ACTIVE canonical DB and an INERT file artifact (`.bak`).** The runtime opens only `session.sqlite` per ADR-0012; the `.bak` is read by nothing. There is no live defect — cleanup alone resolves the user-visible symptom.
- **The M08.5.5 phase doc's prescribed migration 004 was DROPPED at C.fix red-phase** because its labeling was inverted (would have rewritten user `session.sqlite::mcp_servers` from the canonical 4-state to the legacy 5-state, corrupting `'disconnected'`/`'health_pending'`/`'error'` rows). The drop prevented a SEV-1 user-DB-corruption regression.

Disposition: ACKNOWLEDGE-AS-INERT in M08.6 Stage F gap-analysis Carry-forward. M08.6 makes NO `mcp_servers` schema changes (per the Key constraints "No schema change" lock — `schemas/framework.v1.json` is the only schema-class artifact the milestone interacts with, and it stays at v1). The migration-idempotency invariants the M08.5.5 phase doc speculatively assigned to M08.6 are NOT a load-bearing M08.6 deliverable — the structural cleanup is already shipped.

### 🔴 #5 — Builder Canvas state not persisted across app restart → **default route M09 (UX); IRL re-walk recommended**

Two interpretations per phase doc A.3.2:

(a) **User-discipline UX issue**: user edited but did not click Save before quit; Builder always starts from `emptyFramework()` on launch with no autoload.

(b) **Save-then-restore bug**: user saved but the next launch fails to restore.

**Static evidence (gathered without IRL access this session)**:

- `src/lib/builderStore.ts:460` — initial state is `framework: emptyFramework()`; no IPC call or `localStorage` read at store construction.
- `src/components/builder/Inspector.tsx:76-89` — the `onLoad` callback is the ONLY load path; it requires explicit user action (`open({ directory: true })` then `loadFramework`).
- `Grep` for `localStorage|sessionStorage|Open Recent|recent` in `src/lib/builderStore.ts` returns nothing — no auto-load / no recently-opened persistence in the Builder.
- The existing `load_framework` round-trip on a saved inline framework works (`crates/runtime-main/tests/builder.rs:556 load_framework_round_trips_a_saved_framework` per the M08.6 phase doc Stage B `<existing_tests_preserved>`). So if the user did Save and re-Load, the data does restore.

**Static disposition**: **interpretation (a) is strongly supported by the code** — no autoload exists; the user must Load on each launch. The IRL observation of "reverted to old state after restart" is consistent with "user edited in memory but did not Save (or saved to a different directory than the one re-loaded)".

**Route**: **M09 UX work** (default), with the fix path being a "save reminder on quit" + an "Open Recent" Inspector surface. The M08.6 phase doc explicitly lists this disposition as the default if interpretation (a) holds.

**Caveat**: an IRL re-walk on the build machine — load `examples/aria/`, edit, Save to `$SCRATCH/test-fw`, quit, relaunch, open `$SCRATCH/test-fw` — would definitively confirm. If the IRL walk shows interpretation (b) (save-then-restore broken), the fix routes to Stage C-adjacent work in this milestone. Stage A has no real-app harness access; the maintainer's IRL re-walk during V or D's `<stage_0_real_app_discovery>` is the structural verifier.

**Stage A action**: no code; record disposition; flag for verifier confirmation at Stage V or D's real-app discovery walk.
