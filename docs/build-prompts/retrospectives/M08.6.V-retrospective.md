## Findings

### 🟡 #1 — Multi-call: missing per-surface tests for two of the four invariants

**Spec claim:** Phase doc V.5 `<pass name="multi_call_invariants">` — "load_framework on the same directory twice yields the same resolved Framework"; "loading framework A then framework B re-lays-out for B (the load path does not retain A's nodePositions)". Prompt rule: "Missing per-surface test → 🟡".
**Observed:**
- `crates/runtime-main/tests/framework_load_resolution.rs` has 11 tests, none asserting `load_framework(examples/aria/)` twice → identical result. The M08 `framework_loader_smoke.rs::two_consecutive_loads_succeed` (line 64) tests the M08 framework_loader (a different module — `crate::framework_loader`), not the M08.6 `crate::builder::load_framework` which is what this milestone modifies.
- `tests/unit/lib/builderStore.loadLayout.test.ts` has 3 tests, none asserting `applyLoadedFramework(A)` followed by `applyLoadedFramework(B)` replaces A's positions with a fresh layout for B.
**Expected (per spec):** Per the prompt's Multi-call pass each of the four named invariants needs a per-surface test.
**Trace / harness:** `cargo test -p runtime-main --test framework_load_resolution` (11 tests, all single-load); `npx vitest run tests/unit/lib/builderStore.loadLayout.test.ts` (3 tests, single load each).
**Action:** 🟡 — carry forward to M09 Stage A `<read_prior_milestones>` carry-forward list. Adding the two tests is ~15 min total; the underlying behaviors are structurally correct (`load_framework` holds no shared state per the existing pure-function shape; `applyLoadedFramework` is a Zustand `set` that replaces `framework + nodePositions` wholesale), so the absence is a test-coverage gap, not a behavior defect.

### 🟢 #3 — Wire: `walker.rs::agents_item_id` Object branch is structurally unreachable on a loaded framework

**Spec claim:** ADR-0022 — "Once Stage B's loader resolves every `{id, path}` agent to inline, the `FrameworkAgentsItem::Object` variant is gone from `framework.agents[]`".
**Observed:** `crates/runtime-main/src/framework_loader/walker.rs:169` extracts the agent id from a `FrameworkAgentsItem::Object { id, .. }` variant. Post-ADR-0022 the variant only persists in `framework.json` on disk (re-split at save); the in-memory `Framework` the walker traverses post-load carries only `Agent(_)` variants. The branch is defensive M08-era code that survived the milestone.
**Why not a violation of ADR-0022:** The function reads the `id` field — it does NOT read a `.md` file, parse YAML frontmatter, or perform reference resolution. The single-resolution-boundary holds. (The grep for `serde_yaml` + frontmatter parsing returns only `crates/runtime-main/src/builder/persist.rs` and its test files in `**/*.rs`.)
**Action:** 🟢 — log to `docs/tech-debt.md` as a TD-NNN entry. Once the renderer surface is confirmed to never produce `Object` variants in `builderStore.framework.agents[]` (it doesn't today — `builderAgent` always emits an inline shape), the branch can be tightened to `unreachable!("post-ADR-0022 load resolves all agent refs to inline; Object variant should not appear in an in-memory Framework")` with a SAFETY comment. Same applies to `capability_map.rs::inline_agents`'s `Object` filter (it remains correct as a defensive filter; under ADR-0022 it filters zero entries on a loaded framework).
