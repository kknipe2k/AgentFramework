## Cross-stage trends

### Friction patterns that recurred

- **S4 time-box under-run, every stage.** A.fix ~0.75 h vs 4–6 h
  (~0.13–0.19×); B.fix ~0.75 h vs 2–3 h (~0.25–0.38×); C.fix ~1.2 h vs
  3–4 h (~0.3–0.4×); D.fix ~1.0 h vs 4–6 h (~0.17–0.25×). The
  per-stage estimates were calibrated for a higher-friction protocol
  than v1.8 actually produces; the consistent ~3–6× over-estimate is a
  signal that the M08.5 phase doc's effort numbers should be halved or
  better for the next IRL fix cycle. Not a quality concern — under-runs
  vs over-runs.
- **CI-as-verifier divergence, every renderer/shell stage.** A.fix,
  B.fix, and D.fix all cited the same divergence in their G6
  CI-parity table: the `e2e-tauri-driver` job's tests cannot be run
  locally because the build machine has neither `tauri-driver` nor
  `msedgedriver` on PATH. The divergence is structural (a real-app
  gate, by definition, requires the OS-level harness); the pattern is
  durable and the citation language is standardized across the cycle.

### Pattern-level wins

- **Strict v1.8 two-commit TDD held cleanly on all three code stages**
  (B / C / D). Every red commit was committed standalone with the
  falsifiable hypothesis stated, surfaced for red-phase approval
  before any impl, and the impl commit did not modify the test files
  (the `git diff <red>..<impl> -- '<tests>'` EMPTY invariant verified

  byte-identical red→impl in `capability_map.rs` + `agent_sdk.rs` —
  was the M06.5.A.fix precedent applied without confusion.
- **Phase-doc-honest root-cause framing.** Each 🔴's phase-doc section
  cited file:line on `main`@`12f0fb3` for what was verified, and
  D.fix's phase doc explicitly did NOT assert an unverified root cause
  (gotcha #41 — "the exact root cause is not determinable by static
  source reading, and the phase doc says so honestly"). The v1.8
  "root cause is a falsifiable hypothesis the regression test must
  disprove" model was applied honestly at every stage.
- **CI/infra-revival stage class held its own scope** (A.fix). No 🔴
  fix snuck into the harness stage; B / C / D each got a clean
  red→green isolated from cross-stack harness churn.
- **Canonical gate ordering (§6 v1.6 + §6 v1.7 strict-TDD + §6 v1.8

  without thought.

### Surprises across the parent cycle

- **"Version matrix" was a misread.** A.fix's phase doc and ADR-0021
  budgeted 4–6 h and a CI-iteration loop for a "wdio v9 ↔ tauri-driver
  compat" problem. The actual M03 PR #47 failure was two grep-verifiable
  config bugs: (a) `wdio.conf.ts` pointing at `src-tauri/target/` when
  `src-tauri` is a workspace member (binary lives at workspace-root
  `target/`); (b) no Windows `msedgedriver` setup. Both surfaced from
  the M03 disable comment's own iteration log. **Forward implication:**
  when reviving a deferred cross-stack carry-forward, statically
  re-diagnose (workspace layout, the actual error text in the disable
  comment, the current upstream reference) BEFORE iterating a version
  matrix. Candidate `CLAUDE.md` §15 / `docs/gotchas.md` entry (A.fix
  Decision).
- **WebDriver HTML5-DnD via Chromium requires the multi-step pointer
  pattern.** B.fix discovered that bare `dragAndDrop()` does not fire
  `dragstart` (webdriverio#274 / chromedriver#841) and synthetic-JS
  `DataTransfer` dispatch would BYPASS the OS-level handler — passing
  on `main` and missing the bug (gotcha #66 / #82 specialization). The
  W3C Actions API multi-step pattern with an intermediate ~5px move
  past Chromium's `dragstart` threshold is the documented workaround
  that ALSO engages the OS handler. The B.fix test file is now the
  in-project upstream reference for D.fix (and for any future
  renderer-DnD WebDriver test).
- **Live-DOM diagnosis is a feasibility-bounded protocol moment.**
  D.fix's `<execution_steps>` schedule a distinct `diagnose_root_cause`
  step using `elementFromPoint` / computed-styles / viewport-overflow
  inspection on the running app — but the build machine had no
  `tauri-driver`/`msedgedriver` locally and the diagnosis was
  infeasible. Phase doc D.3.2's defensive-fallback language was the
  exact resolution (ship the multi-cause-robust portal triple, name
  the candidate causes in the commit body, CI is the verifier), but
  D.fix's stage prompt wrote the step as if live-DOM is the only
  option. Candidate `STAGE-PROMPT-PROTOCOL.md` addition for the next
  IRL fix-cycle: a `<diagnosis_feasibility>` clause that codifies the
  fallback explicitly (D.fix Decision).
- **Static analysis can rule out every statically-determinable cause
  AND the bug still exists.** D.fix's situation — three correct files
  in isolation, dead buttons in the real WebView2 — is a structurally
  different bug-fix shape from a pinpointed red→green. The defensive
  triple is correct regardless of which candidate cause was the actual
  interceptor; future fix-cycles for "real-runtime behavior the static
  code does not explain" bug classes should expect this shape, not
  treat it as exceptional.

### Hard gate violations across the cycle

- **None.** Every stage's G1 (do-not-commit-until-approved) held. Every
  stage's G2 (no Severity-5 friction) held (max severity 3, A.fix's

  stage's G3 (protocol drift) held (no drift events). G4 was partial

### `CLAUDE.md` updates carrying forward (candidates from per-stage Decisions)

- [ ] **Candidate gotcha (§15)** — re-diagnose-before-version-matrix.
      "When reviving a deferred cross-stack carry-forward (a CI job
      marked `if: false`, a stuck dep upgrade, a deferred config bug),
      statically re-diagnose (workspace layout, the actual error text
      in the disable comment, the current upstream reference) before
      iterating a version matrix. A 'version compat unresolved' disable
      note can be a misread of its own evidence — M08.5.A.fix found the
      M03 `e2e-tauri-driver` disable was a wrong binary path + a
      missing driver, not a wdio/`tauri-driver` incompatibility."
      (Source: M08.5.A.fix Decision.)
- [ ] **Candidate gotcha (§15)** — synthetic-JS DnD bypasses the OS
      handler. "A test that drives HTML5 drag-and-drop via
      `browser.execute()` with a synthetic `DataTransfer` event
      sequence will PASS on `main` (with `dragDropEnabled: true`)
      because the JS-dispatched events never reach the OS-level handler
      — they go directly to the DOM event loop. To regression-guard a
      Tauri-shell drag-handler defect, the test mechanism MUST drive
      real OS pointer events (W3C WebDriver Actions API multi-step
      pattern with an intermediate ~5px move past Chromium's
      `dragstart` threshold). Gotcha #66 specialization for HTML5 DnD."
      (Source: M08.5.B.fix Decision.)
- [ ] **Candidate gotcha (§15)** —
      first-regression-test-in-a-new-mechanism-class becomes the
      project's upstream reference. "When a stage adds the first test
      in a new automation mechanism class (B.fix: WebDriver-driven
      HTML5 DnD via `tauri-driver`), it effectively defines the
      project's upstream reference for it — there is no pre-existing
      in-project example to quote (gotcha #32's case (a)). The author
      MUST cite the external upstream reference inline + name the
      rationale, so the next stage can cite this stage's file as the
      in-project upstream." (Source: M08.5.B.fix Decision; D.fix
      consumed this pattern via citing B.fix's test header as the
      in-project upstream.)
- [ ] **Candidate gotcha (§15)** —
      schema-pattern-bound fixture identifiers. "When authoring a
      `serde_json::from_value::<Framework>(json!({...}))` fixture in
      any test, every `name`/`id`/`model.id` field must match the
      schema regex (kebab-case-only — no dots, no uppercase, no
      underscores with leading letter). Symptom: the first run panics
      at `serde_json::from_value` with `\"doesn't match pattern
      \\\"^[a-z][a-z0-9_-]*$\\\"\"` instead of the right-reason test
      failure. Mitigation: copy a kebab-case identifier from an
      existing fixture in the same test file. Pairs with gotcha #66 —
      a fixture-construction panic is NOT a right-reason red."
      (Source: M08.5.C.fix Decision.)
- [ ] **Candidate gotcha (§15)** —
      walker-archetype-placement: free function in
      `framework_loader::capability_map`, not inline. "When a new SDK
      call site needs to walk `framework.agents[]` or
      `framework.tools[]` (extracting a derived property of an agent /
      a tool / a skill), the natural home is
      `crates/runtime-main/src/framework_loader/capability_map.rs`
      next to `inline_agents` / `parent_grants_for_agent` /
      `capabilities_for_tool` — NOT inline in the consumer
      (`agent_sdk.rs`, `tester.rs`). Pattern: pure free function over
      the typify-generated `Framework`, returning the derived value;
      re-export from `framework_loader::mod.rs`. Justification: keeps
      walker logic unit-testable, mirrors the M06.A archetype, keeps
      consumers thin." (Source: M08.5.C.fix Decision.)
- [ ] **Candidate gotcha (§15)** —
      diagnosis-feasibility-degraded-to-defense-in-depth. "When a
      renderer/shell bug is reproducible only at the WebView2 layer
      AND the build machine cannot run `tauri-driver` locally, the
      diagnosis-first phase doc model degrades gracefully to
      defense-in-depth: the impl ships the multi-cause-robust
      hardening layer (e.g. portal + z-index + max-height/overflow for
      a modal-interception bug), names the candidate causes in the
      commit body, and CI is the verifier. The defensive triple is
      correct regardless of which candidate cause was the actual
      interceptor." (Source: M08.5.D.fix Decision.)
- [ ] **Candidate `docs/style.md` note** — prefer `screen.*` queries
      to `render(...).getBy*` in RTL tests so portal migrations
      preserve test stability (`screen` queries from `document.body`,
      which IS the portal target). (Source: M08.5.D.fix Surprise #1.)
