### Did the `*_with` test-seam pattern hold up for `AgentSdk`?

- [x] **Yes, identical shape to M01.C / M02.C — and now demonstrated four times.** `AgentSdk::run_agent_with_provider_stream` accepts any `Stream<ProviderEvent>`; the production `run_agent` is a 3-line wrapper that constructs the real provider stream. `Connection::from_streams` is the parallel test seam for the drone-IPC client. The pattern now spans named pipes, HTTP+SSE, in-process async streams, and Unix sockets — promote to default TEMPLATE.md guidance for any stage that adds a new safety primitive.
- [x] **Coverage outcome confirms the pattern.** runtime-main hit 99.37% line coverage on first measurement after a targeted addition of the production-wrapper test (`run_agent_drives_provider_stream_to_completion`) + the residual-buffer flush test. The two structural exclusions (`anthropic.rs`, `connection.rs::open`) are documented OS-signal-class holdouts.

### Did the decision-extractor heuristic catch real cases or generate false positives?

- [x] **Behavior matches the spec; false-positive rate untestable until M03 wires real models.** The 9 unit tests + 1 proptest cover happy path + missing-marker rejection + multi-pair last-wins + leading-whitespace tolerance + unrelated-line interleaving. The proptest verifies "no panic on arbitrary input" — a behavior contract.
- [x] **False-positive concern (deferred):** The heuristic matches `Decision:` anywhere in text, including inside code blocks, quoted user content, or model-emitted tutorials. M04 verify+rails replaces the heuristic with a structured emitter injected by the prompt template. Stage D's contract is "ship the simplest version per spec §2"; that's done. Stage E's E2E test should observe whether the renderer's event list shows decision records the user expects vs unexpected matches against ambient prose.

### Did the reconnect policy surface anything worth carrying into the spec?

- [x] **Spec §1d should call out the reconnect surface explicitly.** Currently §1d describes the IPC framing but is silent on reconnect semantics. The M02.D implementation pins: 5 attempts with 200ms → 400ms → 800ms → 1.6s exponential backoff (4 sleeps, no trailing sleep), surfacing `DroneIpcError::Disconnected { retries }` on exhaustion. Worth a one-paragraph add to spec §1d at the post-M02 `docs(spec):` PR.
- [x] **The `events()` stream's "single consumer; reconnect breaks the subscription" semantics also worth adding.** Currently a reconnect drops the read half and respawns; the subscriber hits end-of-stream. M03+ may need a "events() survives reconnect" upgrade if the renderer subscribes long-lived; for M02 the SDK only `send()`s, so this is an acceptable limitation.

### How does the runtime-main drone-IPC coverage compare to drone-server side from M01.C?

- [x] **Comparable.** M01.C drone-side `crates/runtime-drone/src/ipc.rs` was 84.70% (now 86.89% after M02.A's `integration_windows.rs`); M02.D main-side `crates/runtime-main/src/drone_ipc/client.rs` is 100% and `drone_ipc/connection.rs` is 91.62% (with `open()` excluded as the OS-call holdout). The main-side is *higher* coverage than the drone-server side because the main-side `Connection::send_with_reconnect` testable seam exposes the reconnect/backoff state machine to unit tests via `tokio::io::duplex`, while the drone-server side's `accept_loop` uses cfg-platform OS code that's only exercised end-to-end. The pattern that lifted main-side coverage (inject `(rd, wr)` halves at construction) could be retrofitted onto the drone-server side in a future cleanup, but that's not blocking.
