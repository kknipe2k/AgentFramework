### Pinned `list_installed_artifacts` wire (vs the phase-doc assumption)

- **Command signature** — `pub async fn list_installed_artifacts(app: AppHandle) -> Result<Vec<InstalledArtifact>, CmdError>`. `app` is a Tauri-injected argument — the command takes **zero JS args**; the shell resolves `<app_local_data_dir>/skills.lock` internally. **The phase-doc assumption (`<wire_signature_audit>` — "zero JS args") is CONFIRMED, no drift.**
- **`InstalledArtifact` field set** — `{ key: String, kind: ArtifactKind, source: Source, installed_at: String }` (`crates/runtime-main/src/builder/validate.rs:150`). `ArtifactKind` serializes to `'skill' | 'tool' | 'agent' | 'mcp_server'`. **The phase-doc `shape=` claim is CONFIRMED, no drift.** Mirrored verbatim into `ipc.ts` as the `McpServerSummary`-precedent hand-mirrored interface; `source` is left opaque (`unknown`) — the Palette keys items on `kind`.

### builderStore is a separate store; framework.json is the source of truth (ADR-0020)

Confirmed — `builderStore` is a NEW `create()` store, separate from `graphStore`; the C.4.2 `builderStore_is_a_distinct_store_instance_from_graphStore` test passes (a builderStore mutation leaves `graphStore`'s state reference unchanged; `graphStore` has no `framework` slot). ADR-0020 records the model: `framework.json` is the single source of truth, the canvas a projection D1/D2/E build on and M09's Generators write into.

Resolved with one documented `as unknown as Framework` boundary cast — the cold-start document is intentionally pre-valid (D2's continuous validation surfaces the missing `agents` / `session_root_agent` as the user builds).

### `src/types/framework.ts` — the phase-doc drift

The phase doc assumed `src/types/framework.ts` existed; it did not (the `crates/xtask` TS-codegen `ts_targets` list never included `framework`).
