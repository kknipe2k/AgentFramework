## [END] Special log per the prompt's `<special_log>` directive

- **New gotchas about React Flow + non-modal panels (#35 act() warnings):** ApprovalPanel uses no React Flow internals (it's a plain `<aside>` overlay), so the existing `act()` warning patterns from React Flow's `useLayoutEffect` don't apply. The Vitest test suite still emits 1-2 act() warnings from the `App.test.tsx` cases that drive the GraphCanvas; new ApprovalPanel tests are warning-free.
- **Approval round-trip latency:** The renderer→Tauri-command→ApprovalSeam→SDK awaiter→SDK emit→event→renderer round-trip is sub-50ms in normal flow (in-process; no IPC). The Playwright tests verify dismissal happens reliably within the default expect timeout.
- **UI patterns for the multi-action panel (Approve/Revise/Cancel) that future panels (Verify rollback prompt, HITL panels) should mirror:**
  - **Non-modal aside discipline** — `aria-modal="false"` + `role="region"` per WAI APG. The graph stays interactive; ESC dismisses panel-locally without resolving the underlying decision.
  - **Idle/draft mode pattern** — `useState<'idle' | 'revise' | 'abort'>` + an inline form with textarea on draft modes. Approve is single-click; Revise/Abort require typed input with non-empty validation before submit.
  - **State-driven surface** — the panel reads from the graphStore directly via Zustand selector; the click handlers dispatch via Tauri commands; the dismissal is driven by the resulting state transition (`status` → `in_progress` / `aborted`), NOT by optimistic local state. This pattern keeps the panel + the underlying graph in sync without needing in-flight tracking.
  - **Free-text passthrough opacity** — user-typed strings (revisions, reason) pass through opaque per CLAUDE.md §8.security; renderer-side validation limited to length cap (2000) + non-empty trim. Future Verify rollback prompt and HITL response panels should follow the same passthrough discipline.
- **Seam architecture (technical-best-practice fork from phase doc C.3):** The phase doc said "drone-side: extend the IPC command enum to resolve the SDK's approval seam." This is structurally impossible: `ApprovalSeam` is `tokio::sync::oneshot`-backed and lives in `runtime-main` (the same process as Tauri main); cross-process oneshots don't exist. Per CLAUDE.md §12 own-technical-decisions: the 3 Tauri commands resolve the in-process seam DIRECTLY via Tauri-managed `Arc<ApprovalSeam>`. The drone receives the resulting `plan_approved` / `plan_revised` / `plan_aborted` event via the existing Stage B `WriteSignal` path when the SDK emits — no new drone-IPC command needed. Forward-applicable to Stage E HITL where a similar in-process seam will exist.
- **Cumulative per-plan token spend deferral:** Phase doc C.3 PlanNode listed it but the prompt's `<gotchas>` trap forbids new state fields without Stage B amendment. The store doesn't currently track plan→agent linking. Two paths: (a) a Stage B amendment to extend `applyEvent` to compute per-plan tokens; (b) a downstream stage (Stage F Budget?) where plan→agent linking lands naturally with the budget enforcer. Skipped for Stage C.
