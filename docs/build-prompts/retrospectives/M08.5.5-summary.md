## Cross-stage trends

### Friction patterns that recurred

- **Fix-itself-has-a-defect class** (NEW; B2.fix-discovered). B.fix's
  cmd.exe wrapper closed the spawn-layer BatBadBut defect — confirmed
  by 3 Windows-cfg unit tests + 1 assembled regression that all passed.
  When the user manually IRL-re-verified the post-C.fix binary on
  2026-05-24 (the real Tauri app, MCP Servers → Add → Test flow), an
  equivalent-symptom defect surfaced INSIDE the wrapper: cmd.exe's
  parse-layer `/?` rule 2 strips the first + last quote characters of
  the `/C` command line when there are >2 quote chars (rule 1 fails),
  mangling the first token. The B.fix tests asserted on the wrapper's
  PROGRAM (cmd.exe correctly resolved) + the bypass of the OS-layer
  spawn failure — but they did NOT exercise the post-wrapper cmd.exe
  parsing layer, so the parse-layer defect was invisible. B2.fix's
  red-phase assembled test (real-spawn via `Command::output()`, not
  via `transport.connect()`) IS the missing layer of coverage. The
  pattern bit: **assembled-app regression must exercise the full
  PROCESS-COMPOSITION path the running app uses, not just the
  pre-spawn build_command shape.** The CLAUDE.md §6 v1.8
  assembled-app-regression mandate already encodes this intent; B2.fix
  is the M08.5.5 cycle's positive example where IRL re-verify caught
  what the assembled (but partial) regression test missed.
- **Phase-doc-prescribed-API-does-not-exist class.** Both C.fix and
  the M08.5 cycle's prior stages hit the pattern: a phase doc lists a
  function/API/path verbatim, the build machine reads the actual file,
  and the API doesn't match (C.fix: `AuditWriter::write_jsonl` vs the
  actual `AuditWriter::log` async API; phase doc `crates/runtime-main/
  src/audit.rs` vs the actual `crates/runtime-main/src/audit/writer.rs`
  path). Resolution at C.fix was a pre-red `AskUserQuestion` that
  dropped the AuditWriter integration; tracking forward as a candidate
  protocol step (`STAGE-PROMPT-PROTOCOL.md` v1.9 `<execution_step
  name="phase_doc_claims_verification">` before `red_phase_write_tests`).
- **Phase-doc-semantic-inversion class** (NEW in this cycle, the
  highest-severity recurring miss). C.fix caught a labeling inversion
  (legacy↔canonical schema) that would have produced data-corruption
  if implemented literally. The phase doc was authored against the
  abstract structure (legacy → canonical migration is right) but the
  concrete labels were guessed and wrong. The build machine's deeper
  inspection caught it pre-red; the maintainer dropped Sub-fix 2 + 🔴
  #7 routed forward to M08.6. **Pattern bit:** phase-doc authoring
  sessions must grep-verify every concrete claim (schema content,
  migration order, API signature, file path) against the actual
  codebase. The stage prompt's `<phase_doc_inventory_audit>` block
  helps (catches file-presence claims) but doesn't catch semantic
  inversions. Candidate: every code-stage prompt opens with a
  `<phase_doc_claims_verification>` step whose deliverable is a brief
  diff between phase-doc claims and codebase truth.
- **S4 time-box under-run on code stages (B + C), over-run on A.**
  B.fix ~1 h vs 4–7 h estimate (~0.15×); C.fix ~1.5 h vs 5–8 h
  estimate (~0.2×). A.fix ~10 h vs 6–9 h (~1.1–1.7×). The pattern:
  small-surface code stages with strict-TDD ship in ≤1/4 of estimate
  (the M08.5 cycle's calibration miss persists); large-surface
  infrastructure stages with cross-stack integration ship at or
  slightly over estimate. **Forward implication:** M08.6 estimates
  should distinguish "small-surface code stage" (halve M08.5-class
  rule-of-thumb) from "large-surface infrastructure stage" (keep
  M08.5-class rule-of-thumb).
- **CI-as-verifier divergence on every code stage.** B.fix and C.fix
  both cited the same divergence in their G6 CI-parity table: did NOT
  run frontend gates (no JS/TS source changes), did NOT run
  per-package safety gates for crates whose source the stage didn't
  touch, did NOT run `tauri-driver` E2E (no built Tauri binary in the
  headless coding session). The divergence is structural and the
  citation language is standardized — same M08.5 cycle's established
  pattern.

### Pattern-level wins

- **Strict v1.8 two-commit TDD held on both code stages** (B + C).
  Red commit landed standalone, surfaced for approval, impl commit
  did not modify the test files (the `git diff <red>..<impl> --
  '<tests>'` EMPTY invariant proven in both stages' commit bodies +
  retrospective evidence). C.fix landed without an amend (B.fix's
  doc-markdown amend lesson absorbed). The two-commit cadence is now
  routine for build-machine code stages.
- **Pre-red `AskUserQuestion` for design conflicts is established.**
  B.fix surfaced the conditional-vs-unconditional wrap design conflict
  before any code; C.fix surfaced TWO scope-correction decisions
  before any code (Sub-fix 2 schema inversion + AuditWriter API
  surprise). Both saved hours of writing-then-undoing. **Pattern bit:**
  when the stage's first hour is spent inspecting + verifying the
  phase doc rather than writing code, that's load-bearing — surface
  what you find, ask the maintainer to choose, then write. The cycle
  proved this pattern's ROI twice.
- **ADR-flip discipline held.** Three ADRs flipped Proposed → Accepted
  in their respective stage's impl commit (A.fix: 0024 gitleaks +
  0025 env-var override; B.fix: 0023 Windows cmd.exe wrapper). No
  ADR shipped without a code-stage acceptance behind it.
- **Path-agnostic persistence archetype consumed without thought.**
  C.fix's `stray_db_cleanup.rs` is the third use of CLAUDE.md §9's
  path-agnostic-module + Tauri-shell-resolves-directory archetype
  (after M05.D `tier::persistence` and M05.E `audit::file_path`). The
  archetype shape is now standard reflex; no design surfacing needed.
- **Canonical gate ordering (§6 v1.6 + v1.7 strict-TDD + v1.8
  assembled-app-regression) caused zero protocol-drift events
  through B + C.** B.fix's red-commit clippy-doc-markdown amend was
  the only drift across the code stages; C.fix's red-phase clippy
  gate (the B.fix lesson learned) caught the test-file clippy issues
  pre-commit, eliminating the amend entirely.

### Surprises across the parent cycle

- **Phase-doc inversion is the new highest-leverage protocol miss.**
  Prior cycles' biggest misses were time-box (M08.5) or root-cause
  framing (M03 PR #47); this cycle's biggest miss is the C.fix
  schema-direction inversion. The inversion shape is structurally
  distinct from time-box (a calibration issue) and root-cause framing
  (a static-vs-runtime issue) — it's a concrete-claim-vs-codebase
  inversion that a `<phase_doc_claims_verification>` step would catch.
  Surfaces as the M08.5.5 cycle's load-bearing recommendation for the
  M08.6 phase doc author.
- **Stage A.fix's ADR-0025 env-var override exposed a latent M08.6
  bug for the first time.** With `ANTHROPIC_API_KEY` inherited from
  the shell, smoke #6 (`reload reconstructs the graph from persisted
  signals`) actually ran for the first time — and **failed** with
  `[data-testid^="agent-node-"] still not displayed after 15000ms`
  after `browser.reloadSession()`. This is M08-IRL 🔴 #5 (Builder
  Canvas state not persisted across app restart), already routed to
  M08.6 Stage A intake — but Stage A.fix's IRL re-verify is the
  first programmatic confirmation it reproduces deterministically in
  the e2e gate (previously only manual repro on Windows). Adjacent-bug
  triage: confirmed M08.6 routing, no new finding.
- **`tokio::process::Command::raw_arg` is an inherent method (not a
  trait import from `CommandExt`).** B.fix's phase doc + ADR-0023
  wrote it as `CommandExt::raw_arg`; clippy `unused_imports` was the
  surfacing. Forward implication: phase docs that prescribe verbatim
  Rust code blocks should either be Rust-stdlib-aware or compile-checked
  inline at phase-doc authoring time.

### Hard gate violations across the cycle

  (do-not-commit-until-approved) held. Every stage's G2 (no
  Severity-5 friction) held (max severity 4 — the C.fix phase-doc
  inversion, surfaced + resolved pre-red). Every stage's G3 (protocol

- [ ] **Candidate gotcha (§15)** — cmd.exe `/?` rule 2 strip-first-
      and-last requires outer quoting on multi-arg invocations. "When
      invoking a `.cmd`/`.bat` shim via `cmd.exe /C <command line>`
      with multiple arguments (and thus multiple quoted segments in
      the inner command line), wrap the inner command line in an
      OUTER pair of quotes: `cmd.raw_arg(format!(\"/C \\\"{inner}\\\"\"));`.
      cmd.exe's `/?`-documented rule 2 (the 'old behavior' fallback
      when rule 1's 'exactly two quote characters' condition fails)
      strips the first and last quote characters of the rest of the
      command line; without the outer quotes, the inner's first/last
      quotes get stripped, mangling the resolved program name. The
      symptom is `The filename, directory name, or volume label
      syntax is incorrect.` — same string class as pre-B.fix
      BatBadBut. Reference: Microsoft cmd /? quote-handling rules.
      Fixed in M08.5.5.B2.fix; ADR-0023 carries the full rationale
      with the in-place amendment per explicit user override of §11."
