### Did the SSE state machine surface edge cases the prompt didn't anticipate?

- [x] **`signature_delta` is a real wire event that's intentionally silent.** The Anthropic API emits `signature_delta` for thinking blocks (verifier payload). My `SseDelta::SignatureDelta` variant captures it but `state.translate` returns `None` — verified by `signature_delta_is_silent` unit test + `thinking_delta_passthrough` wiremock test (which asserts the signature does NOT surface as a TextDelta). C.3 was correct on this; recording for spec/Stage-D awareness.
- [x] **Tool input can arrive as full-`input`-in-start OR as partial-JSON-deltas.** C.3 acknowledged this ("partial_json may be empty if Anthropic emitted full input in the ContentBlockStart's `input` field. Stage C accepts either form; in practice the API uses InputJsonDelta."). My implementation honors this — empty buffer falls through to an empty `Object`. The `input` field on `ContentBlockStart::ToolUse` is currently parsed but ignored. Stage D's SDK should NOT need to change this; if a future API behavior shift starts using the full-`input`-on-start form for tool calls without `input_json_delta` follow-ups, the current code emits an empty object instead of the actual input. Track in carry-forward but don't fix now (spec is explicit that `InputJsonDelta` is the live behavior).
- [x] **Server-emitted `error` events arrive AFTER `message_start` but can arrive WITHOUT a closing `message_stop`.** My `error_event_emits_provider_error` wiremock test exercises this — the stream ends when the byte source closes, with the `Error` ProviderEvent surfaced; no `MessageStop` is required. Stage D's SDK loop must NOT assume `MessageStop` is the only terminator; `Error` + EOF is also a terminal state.

### Wire-format quirks worth carrying into the spec

- [x] **`message_stop` carries no payload; `message_delta` carries the `stop_reason`.** This is counterintuitive — the natural assumption is that `message_stop` would carry the reason. Stage D's SDK + the spec §2c documentation should be explicit: emit `MessageStop` from `message_delta.delta.stop_reason`, ignore `message_stop` for stop-reason purposes. C.3 had this right; spec §2c may not — flagging for the Stage F gap-analysis review.
- [x] **`signature_delta` is a thinking-mode-specific wire event** (per <https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking>). It's parsed but silently dropped. spec §2c does not mention this event. Worth a one-sentence add to spec §2c when the Stage F PR opens.
- [x] **`ping` events are valid anywhere in the stream** — keep-alive heartbeats. My state machine treats them as no-ops. spec §2c should call this out.
- [x] **HTTP errors come BEFORE the SSE stream opens** — 401/403/429 are headers-level, not in-stream events. The `Error` SSE event is an *in-stream* server-side error after the stream has already opened. The two are distinct error paths and both must be handled. C.3 + Stage D must keep this distinction clear.
