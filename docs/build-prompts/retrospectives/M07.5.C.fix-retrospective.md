## [SPECIAL LOG] Stage-required confirmations (per `<retrospective_requirements>`)

- **Wire param names pinned from the A.fix-shipped `commands.rs` (not assumed).** Read `src-tauri/src/commands.rs`: `complete_import_artifact(app, pending_review_id: String, registry, pending) -> Result<ImportOutcome, CmdError>` and `cancel_pending_import(pending_review_id: String, pending) -> Result<(), CmdError>`. The single renderer-supplied arg on each is the snake_case Rust `pending_review_id`; Tauri auto-converts it to the camelCase JS key **`pendingReviewId`**. Both `ipc.ts` wrappers (`completeImportArtifact` / `cancelPendingImport`) and the `ipc.test.ts` assertions use `{ pendingReviewId }` — matching the phase-doc `<wire_signature_audit>` `phase_doc_assumed` exactly (no drift). `import_artifact` is the unchanged three-arg `{ sourceKind, location, artifactKind }`; only its return type became the discriminated `ImportOutcome` (`#[serde(tag = "status", rename_all = "snake_case")]` → `status: 'pending' | 'installed'`).
- **The regression asserts the BACKEND invoke fires — confirmed.** `reject_invokes_cancel_pending_import` (vitest, `ImportPanel.test.tsx`) drives the real `ImportPanel` + store, opens the review modal, clicks Reject, and asserts `invoke` was called with `('cancel_pending_import', { pendingReviewId: 'pri-1' })`. It does **not** rely on a store-record-deleted check — that store-only assertion is the prior `reject_dismisses_the_import_record`, the precise M07.V #1 Stage-V blind spot. The shipped Reject path (`ImportPanel.tsx:154` `onReject → dismissImport`) called no Tauri command, so the assertion is unsatisfiable by shipped code — a genuine red→green closure. `install_invokes_complete_import_artifact` is the symmetric Install assertion.
- **`confirmImport` / `dismissImport` stayed pure — confirmed.** Both store actions are byte-unchanged from `main` (`graphStore.ts` `confirmImport` flips `phase` to `'installed'`; `dismissImport` deletes the record). The IPC lives in the component — `handleInstall` `await`s `completeImportArtifact` then calls the pure `confirmImport`; `handleReject` `await`s `cancelPendingImport` then calls the pure `dismissImport` (the existing `handleSubmit` IPC-in-component pattern). No store action became `async`. `BlockedRow`'s Remove button stays a pure `dismissImport` — a `'blocked'` record has no pending import.
- **Vitest result — confirmed.** Full suite: 31 files / 350 tests passed; all-files coverage 97.64% line / 93.08% branch / 95.67% function / 97.64% statement (renderer ≥80 gate met with margin). `ipc.ts` 100%; `graphStore.ts` 96.83%; `ImportPanel.tsx` 94.73% — every line of the new `handleInstall` / `handleReject` / wrappers / `recordImport` discrimination is exercised (remaining `ImportPanel.tsx` gaps — lines 59/60/68/69/220-223/332-342 — are all pre-existing `handleSubmit` / `BlockedRow` / `describeProvenance` code, untouched by C.fix).
- **Playwright result — confirmed.** Full e2e suite: 29 tests passed, including the 3 `import_panel.spec.ts` tests — `injected_review_record_renders_the_tier_gate_review_modal` now drives the discriminated `pending` wire through `recordImport` → `'review'` record → modal renders.
