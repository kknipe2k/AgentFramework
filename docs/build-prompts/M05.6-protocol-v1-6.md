## M05.6.1 Problem Statement

v1.5 protocol slots caught phase-doc-vs-codebase drift cleanly at stage-execute time (zero stages shipped wrong code because of it), but each catch cost 5–30 minutes of surface-and-resolve cycles. Specific friction patterns surfaced across M05:

1. **Phase-doc-vs-codebase drift recurred at every work stage** (A, B, C1, C2, D, E, F). File paths, `$defs` references, method names, and struct fields drifted between authoring time and execute time. v1.5's `<phase_doc_inventory_audit>` + `<schema_audit>` caught the file-existence class; method/struct claims have no dedicated slot.

2. **Stage V's fresh-context bias-guard hid intentional descopes from V's read-list** (M05.V Findings #1 + #2). Stage B's authorized SDK wire-up descope was documented only in M05.B-retrospective.md — exactly the file V is forbidden to read. The waiver-as-ADR machinery (ADR-0008/ADR-0009) absorbed the finding correctly, but the structural fix is a `<scope_change>` slot in the phase doc itself that V CAN read.

3. **Cross-stack examples needed concrete authoring guardrails.** M05.F's Zustand v5 `useShallow` for derived-array selectors, M04.C + M05.F's Vite cold-start Playwright timeout, M05.C2's windows-sys feature-flag-by-parameter-type-module gotcha, M05.E's `common.v1.json#/$defs/NonEmptyString` ref-to-nonexistent-$def — all are cross-stack authoring traps that need their own slot.

4. **Stage G closeout has no recurring refactor checkpoint.** `simplify`-class consolidations (extracting duplicated patterns, splitting modules that grew across stages, dead-code removal) have to be requested out-of-band per milestone. Maintainer feedback: every milestone needs a structural refactor opportunity baked into G, not an ad-hoc ask.

5. **Coverage-gate regex prose is brittle.** M05.C1's "plumbing files" phrase took 4 attempts to land the correct `--ignore-filename-regex` argument; M05.C2 hit the same prose-vs-regex gap when lifting `ipc.rs` into the gate. A `<coverage_gate>` slot that names the regex explicitly closes this class of friction.

6. **CLAUDE.md §6 quality-gate ordering** (`cargo fmt --all` → `cargo clippy --fix --allow-dirty -p <crate>` → `cargo clippy --workspace -- -D warnings`) is a recurring 5-stage-of-M05 friction. Gotcha #64 documents it; M05 confirms it should be a per-stage `<execution_steps>::implement` discipline, not a per-stage gotcha citation.

**Success criterion:** STAGE-PROMPT-PROTOCOL.md version header bumps v1.5 → v1.6; validator passes against all existing stage prompts AND new optional tags (no required-tag promotions in v1.6); all 12 candidates have concrete XML shape + recommended placement + example usage documented; CLAUDE.md + RETROSPECTIVE-TEMPLATE.md + gotchas.md updates land in the same commit.

### Candidate 1 — `<coverage_gate>` slot (NEW)

**Problem:** M05.C1's "plumbing files" prose-vs-regex gap took 4 attempts to land the correct `--ignore-filename-regex` argument; M05.C2 hit the same when lifting `ipc.rs` into the gate. Recurred at 7 stages of M05 (A/B/C1/C2/D/E/F).

**XML shape:**

```xml

  <gate scope="workspace" target_lines="80" ignore_filename_regex="src.main\.rs|generated"/>
  <gate scope="package" name="runtime-sandbox" target_lines="95" ignore_filename_regex="src.main\.rs|generated|src.lib\.rs"/>
</coverage_gate>
```

**Placement:** Between `<gates>` and `<self_correction_budget>` in work-stage order.

**Validator:** Optional. No required attributes beyond `scope` + `target_lines`. Validator passes through new optional tag.

### Candidate 2 — `<schema_ref_audit>` slot (NEW)

**Problem:** M05.A caught `mcp_missing` as a phase-doc factual error (claim of "three existing variants" when two existed); M05.E caught `common.v1.json#/$defs/NonEmptyString` as a ref to a `$def` that doesn't exist. **Validator:** Optional. No required cross-check.

### CLAUDE.md §6 update — quality-gate ordering (recurring 6-stage friction)

Add to §6 "Quality gates" preamble after the existing "Before any commit lands…" sentence:

> **Quality-gate execution ordering.** Run gates in this order at every stage's `<execution_steps>::implement` step:
> 1. `cargo fmt --all`
> 2. `cargo clippy --fix --allow-dirty -p <touched-crate>` (mechanical first-pass autofix)
> 3. `cargo clippy --workspace --all-targets -- -D warnings` (final verification)
> 4. Remaining gates (test, doc, audit, deny, llvm-cov, …)
>
> The autofix step before final verification eliminates 6–15 mechanical lints per stage (`too_long_first_doc_paragraph`, `doc_markdown` unbackticked, `missing_const_for_fn`, `manual_let_else`, `items_after_statements`, `option_if_let_else → unnecessary_map_or` cascade). M03 graduated gotcha #34 documents the pattern; M05 confirmed at 6 stages of recurrence. v1.6 makes the ordering canonical.

### CLAUDE.md §9 update — path-agnostic persistence pattern

Add new bullet to §9 (or to the linked `docs/style.md`):

> **Path-agnostic persistence + Tauri-shell-resolves-directory archetype.** Persistence modules accept `path: &Path` parameters; the Tauri shell layer resolves `AppHandle::path().app_local_data_dir().join("<file>")` and passes it in. No workspace dep on `dirs` (already transitive via Tauri). Demonstrated at M05.D `tier::persistence` + M05.E `audit::file_path`; forward-applicable to any new persistence module.

### docs/gotchas.md graduations

4 new entries (next available numbers; build CLI picks sequentially per existing pattern):

1. **typify oneOf-enums with non-`Copy` variant payloads don't derive `PartialEq`/`Eq`** (M05.B + C1 recurrence). Forward-applicable to any future wire format nesting typify-derived `oneOf` types whose variants wrap non-Copy newtypes. Mitigation: compare via serde round-trip (re-serialize the deserialized value and check string equality) rather than `==`.

2. **Zustand v5 derived-array selectors require `useShallow`** (M05.F). Naive `filter`/`map`/`find` selectors trigger `Maximum update depth exceeded` infinite-loops; wrap with `useShallow` from `zustand/react/shallow`.

3. **windows-sys feature flags gate function bindings by ALL parameter type modules, not just function's home module** (M05.C2). `CreateJobObjectW` needs `Win32_Security` (`SECURITY_ATTRIBUTES`) in addition to `Win32_System_JobObjects`. Forward-applicable to any future Windows FFI; pair with WEBCHECK discipline (gotcha #32).

4. **`tokio::io::duplex` buffer must be smaller than payload to surface write-failure branches** (M05.C2). The 2KB default absorbs typical response payloads; reduce to ≤8 bytes when testing peer-write-failure error mapping.
