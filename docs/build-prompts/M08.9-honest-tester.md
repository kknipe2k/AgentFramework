# M08.9 — Honest, drill-able Tester results

> **Protocol version:** v1.8 (per `STAGE-PROMPT-PROTOCOL.md` — strict two-commit TDD on each code stage; `<wire_signature_audit>` + `<construction_reachability_check>`; the v1.8 assembled-app-regression mandate; the v1.9 5th `assembled_execution` verifier pass).
>
> **The Tester stops lying.** Today a Novice run that's blocked from writing reports **"PASS" with no file on disk** (the maintainer hit this IRL): `fold_outcome` folds an L2 `CapabilityViolation` but drops an L4 `TierViolation` into `_ => {}`, so a tier-blocked run returns `passed=true` with an empty failures list, and `TesterModal` renders a green **PASS**. M08.9 makes the verdict **truthful** (Pass / Fail / **Tier-limited**) and the run **drillable** (verdict → each tool call's input/result → raw), reusing the disclosure primitives that already exist. This is the re-cut's (ADR-0031) honest-Tester milestone and the **trust prerequisite for M09's vertical slice** — the slice's run is only meaningful if its result tells the truth.

---

## Background — the grounded state at entry

**The lie (grounded, file:line).** `fold_outcome` (`crates/runtime-main/src/builder/tester.rs:127-162`) matches `CapabilityViolation` → `capability_failures`, `TokenUsage` → `token_spend`, `ArtifactHashMismatch` → `integrity_blocked`, and **everything else → `_ => {}` (`:150`)** — including `AgentEvent::TierViolation`. `passed = capability_failures.is_empty() && !integrity_blocked` (`:155`). So an L4 tier block produces `passed = true`, `capability_failures = []`. `TesterModal.tsx` renders the binary verdict from `outcome.passed` (`:39/:43/:51`) → a tier-blocked run shows **PASS** with nothing surfaced (TD-047, IRL-confirmed: a Novice run "said passed" and wrote no file).

**The signal already exists.** `AgentEvent::TierViolation` (`crates/runtime-core/src/event.rs:668-676`) carries `{ agent_id, tier: TierRef, capability_kind: CapabilityKindRef, attempted_action: String }` and lands in `outcome.trace` — `fold_outcome` simply never reads it. The fix is to fold it, not to detect anything new.

**The disclosure primitives already exist** — they're just not applied to the Tester result: `ValidationCard` (plain → fix → **Show-raw** disclosure, B.fix2); the M08.8.A **Output rail + tool-payload** inspection (TD-034 — click a tool node, see its input/result); the Tester graph pane (`TesterGraphPane`). M08.9 *composes* these, it doesn't build new ones.

**Shapes (no schema change).** `TestOutcome` (`tester.rs:54-73`) is a **hand-written** Rust struct (`serde::Serialize`), NOT schema-derived; its TS mirror is hand-written at `src/lib/ipc.ts:615`. So adding a field is a Rust-struct + TS-interface change — **no `schemas/` change, no ADR** (gotcha #94: the TS mirror is producer-driven — keep it in lockstep, tested by serializing the real type).

**Why it's the next milestone.** Per ADR-0031, M08.9 (honest Tester) precedes M09 (the vertical slice): a slice that "runs and writes a file" is only trustworthy if a *blocked* run reports the block instead of a green PASS. M08.9 is small (the signal exists; the primitives exist) and high-leverage (every future Tester run tells the truth).

## Scope

**In scope:**
- **M08.9.A — Truthful verdict (closes TD-047).** `fold_outcome` folds `TierViolation` into a new `tier_blocks` field; a derived **verdict** (`Pass` / `Fail` / `TierLimited`); `TestOutcome` (Rust + TS) gains the field + verdict; `TesterModal` renders the verdict (3-state) instead of binary PASS/FAIL, with TIER-LIMITED surfacing the blocked action + a "Promote?" affordance.
- **M08.9.B — Run drill-down.** `TesterModal` renders the run as **verdict → per-tool-call (input/result) → raw**, reusing the Output-rail payload surface + `ValidationCard`'s Show-raw disclosure over `outcome.trace`.

**Out of scope:** TD-046 (`vdr.rs:98` phantom `passed` field — owner M9, before a verify producer); any new execution primitive (sub-agents / plans / hooks → M12); the budget bar / gap-resume / save-path (superseded D/E/F → M11/M13); MCP-data wiring (M11). **No `fold_outcome` pass/fail-semantics change beyond adding the tier signal** — a tier block is a *user-setting limit*, NOT a framework defect (ADR-0030), so it must NOT flip `passed` to false or be reported as a Fail.

**Locks:** no `schemas/` change (`TestOutcome` is hand-written); the verdict is **UI-truth** layered on the existing `passed` bool (kept for back-compat = framework-defect-free); reuse the existing disclosure primitives (no new ones); strict v1.8 two-commit TDD; real-app `tauri-driver` IRL is the close (ADR-0021).

## Stage M08.9.A — Truthful verdict (closes TD-047)

### A.1 Problem statement
`fold_outcome` (`tester.rs:127-162`) drops `AgentEvent::TierViolation` at `_ => {}` (`:150`), so a tier-blocked run returns `passed=true` (`:155`, since `capability_failures` is empty and nothing is integrity-blocked) and `TesterModal` renders **PASS** (`:39/:43`). The L4 block — `TierViolation { agent_id, tier, capability_kind, attempted_action }` (`event.rs:668`) — is sitting in `outcome.trace`, unread. A blocked run must report **TIER-LIMITED**, not a clean PASS, **without** being mislabeled a framework Fail (a tier block is the user's tier setting, not a defect — ADR-0030).

### A.2 Files to change
**`<wire_signature_audit>` — pin before pseudocode:** `AgentEvent::TierViolation` fields (`event.rs:668-676`: `agent_id: String`, `tier: TierRef`, `capability_kind: CapabilityKindRef`, `attempted_action: String`); `fold_outcome` match + `passed` calc (`tester.rs:131-155`); `TestOutcome` Rust struct (`tester.rs:54-73`, `serde::Serialize`); the TS `TestOutcome` interface (`ipc.ts:615`, hand-written mirror — keep in lockstep, gotcha #94); `TesterModal`'s verdict render (`TesterModal.tsx:39/43/51`); `capability_kind_label` (the existing label helper used for `CapabilityFailure`).
- `crates/runtime-main/src/builder/tester.rs` — add `TierBlock` struct + `TestVerdict` enum; `TestOutcome` gains `tier_blocks: Vec<TierBlock>` + `verdict: TestVerdict`; `fold_outcome` folds `TierViolation` + computes the verdict. `passed` **unchanged**.
- `src/lib/ipc.ts` — extend the TS `TestOutcome` interface (`:615`) with `tier_blocks` + `verdict` (mirror the Rust serde shape exactly).
- `src/components/builder/TesterModal.tsx` — render `outcome.verdict` (3-state) instead of `outcome.passed ? 'PASS' : 'FAIL'`; TIER-LIMITED shows the blocked action(s) + a "Promote?" link to the tier control.
- Tests: a Rust unit (producer-driven — serialize a real `TierViolation` → `fold_outcome` → assert `tier_blocks` + `verdict=TierLimited` + `passed` still `true`); a vitest on the verdict render; `tests/e2e-tauri/tester_verdict.e2e.ts`.

### A.3 Detailed changes
1. **`TierBlock` + `TestVerdict` (Rust).**
   ```rust
   /// One L4 tier block observed in the test session — the user's tier
   /// forbade the action (NOT a framework defect; cf. CapabilityFailure
   /// which is an L2 scope violation = a defect). Surfaced as TIER-LIMITED.
   #[derive(Debug, Clone, PartialEq, Eq, serde::Serialize)]
   pub struct TierBlock {
       pub agent_id: String,
       /// The coarse capability kind the tier's allowlist excluded.
       pub kind: String,
       /// Plain-English description of what the agent attempted.
       pub attempted_action: String,
   }

   /// The Tester's truthful, UI-facing verdict. Distinct from `passed`
   /// (the framework-defect bool, kept for back-compat): a `TierLimited`
   /// run has `passed == true` (the framework is fine — the user's tier
   /// blocked it), but must NOT read as a clean PASS.
   #[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize)]
   #[serde(rename_all = "snake_case")]
   pub enum TestVerdict {
       Pass,
       Fail,
       TierLimited,
   }
   ```
2. **`TestOutcome` gains the fields** (`tester.rs:54-73`): `pub tier_blocks: Vec<TierBlock>,` + `pub verdict: TestVerdict,`. `passed` stays.
3. **`fold_outcome` folds `TierViolation` + computes the verdict.** Add to the match (replacing the silent drop for this variant):
   ```rust
   AgentEvent::TierViolation { agent_id, capability_kind, attempted_action, .. } =>
       tier_blocks.push(TierBlock {
           agent_id: agent_id.clone(),
           kind: capability_kind_label(*capability_kind).to_string(),
           attempted_action: attempted_action.clone(),
       }),
   ```
   then after the loop:
   ```rust
   let passed = capability_failures.is_empty() && !integrity_blocked;
   let verdict = if !passed {
       TestVerdict::Fail
   } else if !tier_blocks.is_empty() {
       TestVerdict::TierLimited
   } else {
       TestVerdict::Pass
   };
   TestOutcome { passed, verdict, capability_failures, tier_blocks, token_spend, timing, vdr, trace }
   ```
   `passed` is unchanged; `verdict` is the new UI truth. A tier-blocked-but-otherwise-clean run = `passed: true, verdict: TierLimited`.
4. **TS mirror (`ipc.ts:615`).** Add `tier_blocks: TierBlock[]` + `verdict: 'pass' | 'fail' | 'tier_limited'` (matching the `snake_case` serde) + the `TierBlock` interface.
5. **`TesterModal` renders the verdict** (`:39-52`). Replace `outcome.passed ? 'PASS' : 'FAIL'` with a 3-state badge keyed on `outcome.verdict`: `pass`→PASS (ok), `fail`→FAIL (bad), `tier_limited`→**TIER-LIMITED** (warn). For `tier_limited`, render the `tier_blocks` (agent · attempted action) + a "This was blocked by your tier — Promote?" link to the Settings tier control. The metric cards + token line stay.

## Stage M08.9.B — Run drill-down

### B.1 Problem statement
Even with a truthful verdict (A), the Tester result still doesn't let you **see what happened**: `TesterModal` replays the trace into a graph pane but offers no verdict → per-step → raw drill-down. When a run fails or is tier-limited, you can't expand the offending tool call to its input/result, nor the raw event. The primitives exist (the M08.8.A Output-rail tool-payload surface; `ValidationCard`'s Show-raw) — they're just not applied to `outcome.trace` in the Tester.

### B.2 Files to change
**`<wire_signature_audit>` — pin before pseudocode:** `outcome.trace: AgentEvent[]` (the run's events); the M08.8.A Output-rail payload component (the tool input/result renderer — pin its name + props); `ValidationCard` (plain/fix/raw — pin its props); `TesterModal`'s result layout (`:39-76`); the `ToolInvoked`/`ToolResult` (and `TierViolation`/`CapabilityViolation`) event shapes in `outcome.trace`.
- `src/components/builder/TesterModal.tsx` — under the verdict, render a **trace list**: each tool call (invoke + result) is a row that expands to its input/result payload (reuse the Output-rail payload surface) → raw (reuse `ValidationCard`'s Show-raw). Tier-limited / capability rows link to their explainer.
- A small extraction if needed: lift the Output-rail payload renderer into a shared component if it's currently coupled to the live graph (mirror A's `InspectorPanel` parameterization precedent).
- Tests: a vitest (the trace renders rows; a row expands to payload + raw) + `tests/e2e-tauri/tester_drilldown.e2e.ts`.

### B.3 Detailed changes
1. **Trace → rows.** Map `outcome.trace` to a list of run steps (agent text, tool invoke→result pairs, violations/tier-blocks). Each tool step is collapsed by default, showing name + status.
2. **Expand → payload.** A row expands to its input/result via the Output-rail payload surface (the same component A's tool-node click uses — reuse, don't reimplement).
3. **Raw disclosure.** Each row's raw `AgentEvent` rides a `ValidationCard`-style **Show-raw** disclosure (the B.fix2 pattern), so the literal event JSON is one click away but not in your face.
4. **Tie to the verdict.** A `fail` row (capability violation) and a `tier_limited` row (tier block) are visually distinct + link to their explainer (the capability disclosure / the tier control).
