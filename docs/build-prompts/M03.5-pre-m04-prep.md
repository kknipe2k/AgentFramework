## Background and Design Decision

**Problem.** Two distinct pieces of prep work need to land before M04 prompt authoring begins. (1) M03's gap-analysis + retros surfaced ~21 doc-level items (M03 spec polish + still-open M02 spec carry-forward + 8 gotchas graduated from per-stage scratch space) that should land before M04's first stage runs — otherwise M04 starts atop stale spec text and a stale gotchas list. (2) STAGE-PROMPT-PROTOCOL.md v1.3 was identified in the M03 closeout as an authoring-discipline iteration informed by M01–M03 friction; five candidate tags address recurring failure modes (sequencing slips, schema drift, rename fan-out, dependency-feature gaps, OS-platform-command differences). Authoring v1.3 before M04 means M04's stage prompts can use the new tags.

**Solution.** One mini-milestone, two stages on one feature branch (`claude/m03.5-pre-m04-prep`), each a fresh Claude Code session per the v1.2 XML stage-prompt protocol. Stage A is the combined doc PR — 22 distinct edits across 3 existing files plus 1 new schema file. Stage B authors the v1.3 protocol — 5 new tags added to STAGE-PROMPT-PROTOCOL.md with descriptions, semantics, validator rules, two worked examples, anti-patterns, and a v1.3 changelog entry. PR drafts at the end of Stage B and pushes once Stage B's commit is approved.

**Why one PR for the parent milestone.** Same logic as M01–M03 — two sequential commits on one branch gives incremental discipline (each stage is reviewable; each stage retrospective surfaces friction early) without the overhead of two PR reviews for one logical prep effort. Consistent with the per-milestone-as-PR pattern in `docs/build-prompts/README.md`.

**Why two stages, not one prompt.** Stage A's 22 edits + verbatim text run ~500–700 prompt-lines on its own. Stage B's v1.3 protocol authoring (5 tags + 2 worked examples + validator updates + anti-patterns + changelog) runs another ~400–600. Combined exceeds the 250-prompt-line per-stage budget and would dilute attention across two unrelated surfaces (spec/gotchas vs. protocol). Per the scope-split rule in `docs/build-prompts/TEMPLATE.md`, split.

**Why no Phase Closeout / gap-analysis entry.** M03.5 is doc/protocol-only prep work, not a §20-bound parent milestone in the M01–M11 sense. There is no code-vs-spec product surface to audit; the entire deliverable IS doc/spec/protocol movement. CLAUDE.md §20's append-only ledger is reserved for milestones that ship code. M04's Phase Closeout (Stage G) absorbs any M03.5 carry-forward where relevant; M03.5 itself does not append to `docs/gap-analysis.md`. The append-only invariant on M01–M03 entries is preserved untouched.

**Why now (after M03, before M04).** M03 just merged (PR #47); M04 starts with carry-forward absorption in its own Stage A1 + A2. Several M04-blocking spec gaps (Plan model field shapes, `count_tokens` real-endpoint contract, `error.v1.json` schema, long-lived `events()` reconnect semantics, decision-extractor migration) are explicitly flagged in M03's gap-analysis as needing pre-M04 spec edits. Doing this prep on a separate branch keeps M04's branch focused on the M04 deliverable surface.

**Key constraints.**
- **Doc-only.** No source-code changes in either stage. No `cargo` or `npm` source changes; only `agent-runtime-spec.md`, `docs/gotchas.md`, `CLAUDE.md`, `STAGE-PROMPT-PROTOCOL.md`, `CHANGELOG.md`, `docs/build-prompts/README.md`, plus one new schema file (`schemas/error.v1.json`). Stage B may also touch `scripts/validate-stage-prompts.py` if v1.3 validator rules require code (additive only).
- **Append-only ledger preserved.** `docs/gap-analysis.md` is NOT touched by either stage. M01/M02/M03 entries remain immutable per CLAUDE.md §20. M03.5's outputs become inputs to M04 Stage A1's Carry-forward absorption.
- **v1.2 protocol for stage prompts.** Both Stage A and Stage B X.5 CLI prompts are v1.2 XML stage_prompt blocks. Stage B's *deliverable* is v1.3, but the *prompt* driving Stage B's build agent is v1.2 (since v1.3 doesn't exist until Stage B commits).
- **Reference-first STRICT (v1.2).** Both stages' content-heavy tags (`<deliverable>`, `<acceptance_criteria>`, `<scope_locks>`, `<commit_message>`) use `ref="..." section="..."` form pointing at this Phase doc's X.3 / X.4 / Key constraints / X.6 sections. No inline content in those tags.
- **No new ADR triggers.** This milestone touches no §0a Capability Matrix primitives, no schemas in the source-of-truth sense (`error.v1.json` is *new*, additive; not a modification to v1 of an existing schema), no LLMProvider, no capability enforcement, no IPC protocol, no v0.1/v1.0 scope. Per CLAUDE.md §11, no ADR required.

**License.** Apache 2.0; DCO sign-off (`git commit -s`) on every commit.

**Existing patterns to mirror.**

- PR #48 archetype for doc-only follow-up post-milestone: commit `10d6b98` (small focused doc PR off main, single-commit, surface-then-approve flow).

## Implementation Workflow

Each stage runs through this exact cycle:

```
1. /clear                     — fresh context (only between stages)
2. Paste CLI Prompt below     — XML <work_stage_prompt> pasted into a fresh
                                Claude Code session
3. WEBCHECK pass              — Stage A: none required (no fast-moving tooling
                                touched). Stage B: protocol authoring is
                                self-contained; cite STAGE-PROMPT-PROTOCOL.md
                                v1.2 + M01-M03 retros only
4. Read prior stage retros    — Stage B reads M03.5.A retro [END] Decisions
                                section, applies decisions BEFORE code
5. (No code phase)            — both stages are doc/protocol-only; no failing
                                tests to write
6. Implement                  — Claude makes the doc/protocol changes per
                                X.3 Detailed Changes
7. Verify                     — Select-String / Test-Path checks for each
                                edit landed verbatim; prettier on JSON; YAML
                                /JSON parse checks where applicable
8. Fill in retrospective      — docs/build-prompts/retrospectives/M03.5.<X>-retrospective.md
9. Commit (no push)           — exact commit message provided per stage X.6
10. User reviews + approves   — Claude does NOT push without approval
11. Push (Stage B only)       — Stage B push gates the M03.5 PR draft
```

**Rule:** Doc-only stages do not have a "failing tests first" red phase per CLAUDE.md §5. The verification step (Select-String / JSON parse) replaces the test phase. Per `docs/gotchas.md` #34, run `npx prettier --check schemas/error.v1.json` (Stage A) as the first verify step.

**Rule:** Stages are sequential. Stage B does not start until Stage A's commit is on the feature branch (locally is sufficient; push is optional). The parent-milestone PR pushes only at the end of Stage B.

**Rule per CLAUDE.md §8:** Claude does not commit without user approval. After verifications pass + retrospective filled, Claude surfaces the diff stat + retrospective + draft commit message. User approves; Claude commits.

**Rule per CLAUDE.md §19:** Each stage produces a retrospective; the final stage also produces an `M03.5-summary.md` aggregating across stages.

**Rule per CLAUDE.md §20:** No gap-analysis entry produced by this milestone (per the Background "Why no Phase Closeout" section). M01/M02/M03 entries in `docs/gap-analysis.md` remain immutable; M03.5 outputs become M04 Stage A1 carry-forward inputs.

```markdown
#### SQL inspector query validation (v0.1, M03.E shipped)

The renderer-side SQL inspector accepts user-typed queries and proxies them via the Tauri `query_session_db` command. v0.1 validates queries lexically rather than via `Connection::open_in_memory().prepare()`-style probes:

- Empty input → reject
- Compound `;`-separated statements → reject (single statement only)
- Lowercase `select` allowlist on the leading non-whitespace token (case-folded)
- `pragma` rejected explicitly

The lexical pass is intentionally narrow. An in-memory probe DB has no schema and would reject every legitimate `SELECT signal_id FROM signals`, so prepare()-based validation was rejected. v0.1 is single-user single-session; v1.0 multi-session tightens via parameter binding + read-only connection enforcement.
```

Explicit declaration of the OS the build agent is expected to run on for this stage, plus platform-specific command variants where the prompt body contains commands that differ across OS (e.g., `Select-String` vs `grep`, `Test-Path` vs `test -f`). Addresses CRLF warnings (PR #48), PowerShell-vs-bash command differences in stage prompts, and the macOS-unsupported caveats (gotcha #23 tauri-driver, gotchas #25-#27 various Vite/test patterns).

Self-closing form with `os="..."` attribute, OR child `<command>` elements with `os="..."` and `cmd="..."` attributes for platform-specific commands.

Attributes:
- `os` — one of `windows | linux | macos | any`. Default `any`.
- `note` — optional inline rationale for the OS pin (e.g., "Tauri-driver requires Linux or Windows; macOS unsupported per gotcha #23").

Validator behavior (v1.3 lean): structural — error if the tag appears outside a work-stage prompt; warning if `os` attribute is missing on the root tag.

Schema: work-stage only.

Examples:

Single-OS:

```xml
<runtime_environment os="windows" note="Build agent runs on Windows 11 per the established M01-M03 pattern; Select-String is the assumed grep equivalent throughout the prompt"/>
```

Multi-OS with command variants:

```xml
<runtime_environment os="any">
  <command os="windows" cmd="Select-String -Path schemas/error.v1.json -Pattern 'CmdError'"/>
  <command os="linux" cmd="grep -n 'CmdError' schemas/error.v1.json"/>
  <command os="macos" cmd="grep -n 'CmdError' schemas/error.v1.json"/>
</runtime_environment>
```

When this tag appears in a stage prompt, the agent runs only the commands matching the current `$RUNNER_OS` (or local equivalent). Authors using inline OS-specific commands (e.g., a `Select-String` command in the prompt body) should pair them with this tag so the validator can flag missing variants in v1.4+.
```

#### Updates to §13 Anti-patterns — three new entries

```markdown
**v1.3 anti-patterns (new):**

`<pre_flight_check>` used as a substitute for `<read_first>`. Pre-flight checks are environmental verifications (branch, prior-commit, env vars, deps); they are NOT for orientation reads. If your "check" is "agent reads CLAUDE.md", that's `<read_first>`. If your "check" is "git rev-parse HEAD matches expected branch", that's `<pre_flight_check>`. The discriminator is whether the check fails imperatively (command exit non-zero) vs informationally (read the file).

`<fan_out_grep>` enumerating the change set instead of finding consumers. Fan-out grep is for finding consumers of a name BEFORE the rename — to verify nothing is missed. If your `<grep>` patterns are the same as the files in `<deliverable>`, you've conflated "files I'll edit" with "files that reference the thing I'm renaming." The grep should find files OUTSIDE `<deliverable>` that need coordinated updates (or confirm none exist).

`<runtime_environment>` declared without command variants when the prompt body contains OS-specific commands. If the prompt body has `Select-String` commands, declare `os="windows"` (single-OS pin) or provide `<command os="...">` variants for the supported OSes. Authors who pin to one OS without rationale in the `note` attribute are setting a trap for future authors who fork the prompt to a different platform.
```
