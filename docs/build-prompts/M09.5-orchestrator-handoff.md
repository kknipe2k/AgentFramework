## Your role (read this first)

You are the **orchestrator**: you author phase docs / ADRs / protocol, adjudicate the build machine's
per-stage surfaces, drive GitHub (PRs / merges / CI watching), and sequence the work. You do **not** write
product code. A **separate Claude session on the maintainer's Windows box is the BUILD MACHINE** — it writes
code/tests/commits and surfaces one approval-ready bundle per stage. **The maintainer relays surfaces between
you and the builder by paste.** You never talk to the builder directly; you hand the maintainer XML blocks to
paste, and the maintainer pastes the builder's surfaces back to you.

The maintainer is terse, hates verbose work-salad, wants plain English in chat (lead with the contradiction,
no jargon/decoration/arrows) — but **build docs and stage prompts stay fully rigorous**. He owns
scope/priority/irreversible/capability forks; he delegates technical-best-practice decisions to you. He
approves *outcomes*, not steps. For any IRL: concrete copy-paste steps with exact paths; ask for a path,
never guess.

## The operating loop (per stage)

1. Maintainer pastes the stage's CLI prompt (e.g. **D.7** for Stage D) into a **fresh** builder session.
2. **Plan-first stages (D, E — Hard Rule 8):** the builder's FIRST surface is a capability plan (not code).
   You adjudicate it; relay an `<plan_approval>` XML block (+ riders) for the maintainer to paste back.
3. Builder writes failing tests → surfaces the **red proof** (the adversarial case failing right-reason). You
   relay a `<red_approval>`.
4. Builder implements (no test-file edits — the `git diff <red>..<impl> -- '**/tests/**'` EMPTY invariant),
   runs §6 gates + the blocking mutation gate (D/E are critical modules), pushes, surfaces the final bundle.
5. **You stamp CI + the maintainer IRLs** (see CI dance below). Green CI + IRL pass = stage closed → next stage.
   Any CI red or IRL finding routes back as an `<X.fix>` directive (max 2 D.fix iterations per ADR-0008).

## CI dance (session-learned — in no other doc)

- **Your GitHub token is READ-ONLY on Actions.** `mcp__github__actions_run_trigger` returns **403**. You
  CANNOT dispatch CI. The **maintainer runs it** from the build machine:
  `gh workflow run ci.yml --ref claude/m09.5-perimeter-hardening`
- **CI triggers only on PR + push-to-main — NOT on a branch push.** So every branch verification is a manual
  `workflow_dispatch`. This is why we **push per-stage and dispatch** (a deliberate deviation from the
  push-once-at-G default — log it in the G retro). E's cfg(linux) tests *only* run on CI (the build box is
  Windows); D's wiremock tests run locally too but keep the per-stage dispatch rhythm.
- **The workflow cancels in-progress runs on the same ref.** Do NOT have the maintainer fire two dispatches
  close together — the second cancels the first (this bit us: a parallel dispatch + a `gh run rerun` cancelled
  each other; run them **one at a time, sequentially**).
- **You watch fine** (read works): `mcp__github__actions_list` (`list_workflow_runs` filtered by branch +
  `event:workflow_dispatch`; `list_workflow_jobs` for per-job status) + `mcp__github__get_job_logs`
  (`failed_only:true`, `return_content:true`). Arm a `Monitor` timer ~18–20 min for a full run. **Large
  results spill to a file** — parse with `python3 -c "import json; …"` (the list tool's output is
  `{"jobs":{"jobs":[…]}}`).
- **Known Windows flake** (not a real failure): `drone_reconnect_events::old_events_subscription_…` can lose a
  pipe-creation race on a rerun (gotcha #56 family). Re-run confirms. Ledger a connect-retry hardening at G.

## Stage E adjudication (your immediate next action)

**E.7** in the phase doc is the prompt the maintainer pastes into a fresh builder. The builder **already surfaced
its capability plan once** (2026-06-14) — it is sound; when it re-surfaces (a fresh builder re-runs
`surface_capability_plan`), adjudicate against the summary below and relay a `<plan_approval>`. Recommended
disposition: **approve, both decisions resolved as recommended, two riders.**

The surfaced plan (verified sound by the outgoing orchestrator):
- **Red proof:** `seccomp.rs:104` puts `socket` in the unconditional ALLOWED loop (`add_rule(Allow,…)` @172), so
  `socket(AF_INET,SOCK_DGRAM,0)` succeeds inside the fence today — the constructible finding.
- **Fix:** remove `socket` from the unconditional loop; one `add_rule_conditional(Allow, socket, [arg0==AF_UNIX])`
  (libseccomp 0.3.0); `bind/sendto/…` stay unconditional (moot once no inet fd can exist — only `socket(2)` arg0
  is BPF-filterable).
- **Deny observable = SIGSYS (signal 31), NOT EPERM** — correctly derived: the filter is built
  `new_filter(KillProcess)` (`seccomp.rs:163`), so a non-AF_UNIX socket matches no rule → default
  `KillProcess` → SIGSYS. An `Errno(EPERM)` rule would outrank the conditional Allow (KILL>ERRNO>ALLOW) and break
  AF_UNIX. Kill-on-deny is the correct security posture + matches the existing default. **Approve this observable.**
- **Behavioral test (cfg(linux), OS-signal-class holdout):** subprocess self-reexec (`current_exe()` + `--exact` +
  `RUN_SECCOMP_PROBE` env) installs full isolation in production order, then: inet arm `UdpSocket::bind` → expect
  SIGSYS-kill (survives → `exit(7)` red sentinel); unix arm `UnixListener::bind` → `exit(0)`. Parent asserts
  `status.signal()==Some(31)` / exit 0. The two arms **jointly pin all three comparator fields** (arg/op/datum) —
  this IS the mutation-adequate evidence. `validator-still-works` = existing `isolation_persists_across_validate_calls`
  stays green (not duplicated). std-only.
- **Unit changes:** new `socket_not_in_unconditional_allowlist` (red today); `clone` added beside `clone3` in the
  forbidden assertion (additive); `allowlist_includes_socket…` drops `socket` (edited in the red commit).

**Decision 1 — AF_UNIX / SIGSYS source → recommend CONST (no libc).** `const AF_UNIX:u64=1` + `const SIGSYS:i32=31`,
each commented as the frozen Linux ABI value. Honors the no-new-dep scope lock; test stays std-only. (Adding `libc`
as a direct dep for two constants is unwarranted.) **Rider A:** `SIGSYS=31` is x86_64/ARM-correct (the CI ubuntu
target); comment that arch assumption (it differs on MIPS/Alpha — out of scope for v0.1).

**Decision 2 — mutation gate is Linux-only → accept the honest split.** `seccomp.rs` is `cfg(linux)`, so
Windows-local cargo-mutants can't compile/mutate it. The CI-Linux two-arm behavioral test is the per-PR kill
evidence (it pins the three comparator fields); the standing enforcement is the A-stage nightly Linux mutants
workflow. The surface states this split (rule 11) rather than claiming a Windows-local pass. **Rider B (optional):**
a scoped Linux cargo-mutants run (manual `mutants-nightly.yml` dispatch, or WSL) confirms it — but the behavioral
two-arm pin is sufficient + honest, same posture as D's behavior-pinned reset/fuse.

**E red rhythm (different from D):** the cfg(linux) subprocess test won't compile/run on Windows (gotcha #74), so the
**red proof lands on CI-Linux** — after the red commit, the maintainer dispatches `ci.yml` and you confirm the inet
test fails right-reason on Linux (AF_INET survives → exit 7 / `signal None`, not `Some(31)`) before approving impl.
Green proof + mutation are likewise CI-Linux. Scope locks: `landlock.rs:198` (!Execute), `SandboxRequest`, install
order untouched; zero new unsafe; no `§A` coverage change (`seccomp.rs` already fully gate-excluded).
