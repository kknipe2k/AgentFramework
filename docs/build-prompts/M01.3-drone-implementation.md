## Problem statement

M01.3 implements **Phase 1: The Drone** per spec §1. The drone is the survival layer — every later milestone depends on it being able to:

1. Run as a child process spawned by the (future, M02+) main process
2. Maintain a heartbeat to SQLite every 5 seconds
3. Accept `DroneCommand` messages on its IPC channel and emit `DroneEvent` messages
4. Take snapshots on demand and store them append-only
5. Survive `SIGTERM` / `SIGINT` with an emergency snapshot before exiting
6. Initialize SQLite with the four required pragmas in the right order

This is the failure-recovery foundation. If a session crashes, the drone's snapshots are how it gets restored. If the drone is broken, every later milestone inherits brokenness.

**100% line coverage on `runtime-drone`** is required. Every safety-critical branch (signal handler, snapshot writer, IPC error path) needs a test. Per `CLAUDE.md` §5, drones are safety primitives.

## Scope

### In scope

**`crates/runtime-drone/src/`:**
- `main.rs` — binary entry: clap CLI args (`--session-id`, `--db-path`, `--ipc-socket`), tokio multi-thread runtime, top-level orchestration
- `lib.rs` — library re-exports for testing
- `db.rs` — SQLite open + WAL pragmas in correct order + schema init for all five tables (`sessions`, `snapshots`, `signals`, `vdr`, `token_usage`, `skills`); only `sessions` and `snapshots` actively used in M01.3
- `ipc.rs` — Unix domain socket on Linux/macOS + Windows named pipe; framed JSON-newline via `tokio_util::codec::LinesCodec` or `LengthDelimitedCodec`; encode `DroneEvent` / decode `DroneCommand` from `runtime-core`
- `heartbeat.rs` — tokio task: 5-second `tokio::time::interval`; writes heartbeat row to SQLite; emits `DroneEvent::Heartbeat`
- `snapshot.rs` — snapshot writer; computes `state_hash` (SHA-256 of `state_json`); writes append-only to `snapshots` table; emits `DroneEvent::SnapshotWritten`
- `shutdown.rs` — graceful shutdown handler + SIGTERM/SIGINT/CTRL_BREAK_EVENT emergency snapshot path; uses `tokio::signal`
- `command_handler.rs` — dispatches `DroneCommand` variants: `SnapshotNow`, `GracefulShutdown`, `RevertToSnapshot`, `SpawnProcess` (stub — returns NotImplemented), `StopProcess` (stub), `SetActivityTimeout`

**Tests in `crates/runtime-drone/tests/`:**
- Subprocess-spawn integration tests (test #16 below)

**Manual smoke** documented in `crates/runtime-drone/README.md`:
```bash
cargo run --bin runtime-drone -- \
  --session-id smoke --db-path /tmp/d.sqlite --ipc-socket /tmp/d.sock
```

### Out of scope

- **`SpawnProcess` and `StopProcess` actual implementations** — the drone owns process lifecycle in spec, but at M01.3 we only need the IPC stubs (return a NotImplemented variant of `DroneEvent`). Real process management is M02+ when the main process actually wants to spawn things via the drone.
- **Multi-session.** Single drone, single session per spec §0d v0.1.
- **Recovery flow** — §1b spec describes resume semantics, but actually *replaying* a session is M04+ work. M01.3 just stores snapshots; M04 reads them.
- **Cross-OS testing in CI** — happens, but the M01.3 implementer doesn't need to manually verify Windows; CI does that. Focus on getting Linux/macOS solid first; iterate Windows via CI.

## TDD plan

### Unit tests (Rust — `cargo test`)

In `crates/runtime-drone`:

1. **`wal_pragmas_set_in_correct_order`** — open DB via drone's setup, query `PRAGMA journal_mode`, `synchronous`, `busy_timeout`, `foreign_keys`; assert `WAL`, `NORMAL`, `5000`, `1`. The order matters per CLAUDE.md gotcha #13.
2. **`heartbeat_fires_at_interval`** — `tokio::time::pause()` + `advance(Duration::from_secs(5))`; assert exactly one `DroneEvent::Heartbeat` produced per 5-second window. **`pause()` must be called before any `tokio::time::*` resource is constructed** (CLAUDE.md gotcha + M01-spec §3 of self-correction).
3. **`snapshot_now_writes_correct_row`** — `DroneCommand::SnapshotNow { reason: "test" }` with state blob; query `snapshots` table; assert row has correct `session_id`, non-zero `timestamp`, `reason: "test"`, `state_json` matches input, `state_hash` is SHA-256 of `state_json`.
4. **`snapshot_state_hash_is_sha256`** — for known input, assert hash matches `sha256` standalone computation.
5. **`snapshot_appends_does_not_update`** — write two `SnapshotNow` with same state; assert two rows exist (append-only, not upsert).
6. **`graceful_shutdown_flushes_within_timeout`** — given pending writes, `DroneCommand::GracefulShutdown { timeout_ms: 1000 }` exits 0 within 1s AND all pending writes present in SQLite.
7. **`graceful_shutdown_kills_after_timeout`** — pending writes that exceed timeout get killed; drone exits non-zero.
8. **`sigterm_triggers_emergency_snapshot`** — spawn drone subprocess, send SIGTERM (Unix) or `CTRL_BREAK_EVENT` (Windows); assert (a) drone exits within grace period and (b) snapshots table has row with `reason: "sigterm"` (or `"ctrl_break"` on Windows).
9. **`revert_to_snapshot_returns_blob_no_filesystem_mutation`** — given snapshot exists, `DroneCommand::RevertToSnapshot { snapshot_id }` returns `DroneEvent` containing the original `state_json`. Verify no files modified outside the SQLite database.
10. **`revert_to_unknown_snapshot_returns_alert`** — `DroneCommand::RevertToSnapshot` with bad ID emits `DroneEvent::Alert { level: Critical, message: "snapshot not found: <id>" }`.
11. **`unknown_command_returns_alert`** — malformed/unknown command emits Alert; drone keeps running.

### Property tests

12. **`drone_event_codec_round_trip_proptest`** — for any `DroneEvent` (proptest strategy), encode through framed codec, decode, assert identity. Drives IPC framing correctness.
13. **`drone_command_codec_round_trip_proptest`** — same for `DroneCommand`.
14. **`framed_codec_handles_concatenated_messages`** — encode N messages back-to-back, decode all; assert all N retrieved in order.

### Fuzz harness

15. **`fuzz_target_drone_command_decode`** — fuzz the IPC frame decoder with arbitrary bytes; must not panic, must not deserialize to a `DroneCommand` that bypasses validation. Run 30 seconds in CI per PR.

### Integration tests (`cargo test --features integration`)

16. **`drone_lifecycle_end_to_end`** — spawn drone subprocess with `tempfile`-backed db + socket; connect to the IPC socket; send `SnapshotNow` → assert `SnapshotWritten` received; send `RevertToSnapshot` with that ID → assert state_json returned; send `GracefulShutdown { timeout_ms: 1000 }` → assert clean exit.

### Coverage target

- `crates/runtime-drone`: **100% line coverage**. Every branch (including error paths) must be tested. Use `cargo llvm-cov report --package runtime-drone --fail-under-lines 100`.

### CLI args

```rust
#[derive(clap::Parser)]
struct Args {
    /// Session identifier this drone serves.
    #[arg(long)]
    session_id: String,

    /// Path to the SQLite database file.
    #[arg(long)]
    db_path: PathBuf,

    /// Path to the IPC socket / named pipe.
    #[arg(long)]
    ipc_socket: PathBuf,
}
```

### WAL pragma order (mandatory per CLAUDE.md gotcha #13)

```rust
fn init_db(path: &Path) -> Result<Connection, DbError> {
    let conn = Connection::open(path)?;
    conn.pragma_update(None, "journal_mode", "WAL")?;
    conn.pragma_update(None, "synchronous", "NORMAL")?;
    conn.pragma_update(None, "busy_timeout", 5000)?;
    conn.pragma_update(None, "foreign_keys", "ON")?;
    init_schema(&conn)?;
    Ok(conn)
}
```

### Platform-specific IPC

```rust
#[cfg(unix)]
mod ipc_impl {
    use tokio::net::UnixListener;
    pub async fn bind(path: &Path) -> Result<UnixListener, IpcError> { /* ... */ }
}

#[cfg(windows)]
mod ipc_impl {
    use tokio::net::windows::named_pipe::{NamedPipeServer, ServerOptions};
    pub async fn bind(path: &Path) -> Result<NamedPipeServer, IpcError> { /* ... */ }
}
```

Don't try to abstract them as identical. Document the differences in `runtime-drone/README.md`.

### What NOT to write

- **Real `SpawnProcess` / `StopProcess` logic.** Stub them with `DroneEvent::Alert { level: Warn, message: "spawn_process not implemented at M01.3 — see M02+" }`.
- **Custom `Drop` impl on the SQLite connection.** rusqlite handles cleanup.
- **Wall-clock-dependent tests.** Use `tokio::time::pause()` always.
- **`println!` for logging.** Use `tracing` crate. stdout/stderr are reserved for log capture; the IPC channel is what main reads from.

## Self-correction guidance

| Failure | Likely cause | Fix |
|---|---|---|
| Heartbeat test hangs | `tokio::time::pause()` not in scope OR `interval` constructed before `pause()` | Test setup order — `pause()` first, then construct `interval` |
| Snapshot test sees no row | Transaction not committed | `crates/runtime-drone/src/snapshot.rs` — confirm explicit commit; verify `synchronous = NORMAL` |
| WAL pragma test fails | Pragmas in wrong order | Order: `journal_mode` first, then `synchronous`, `busy_timeout`, `foreign_keys` |
| SIGTERM test exits 1 | Emergency snapshot panicked | Wrap snapshot in `if let Err(_) = ... { tracing::error!(...) }`; exit 0 anyway — emergency is best-effort |
| Windows-only IPC failure | Named pipe path format | Windows uses `\\.\pipe\name`; Unix sockets use filesystem path; the `--ipc-socket` arg should normalize |
| Integration test #16 flaky | Race between drone start and client connect | Drone should signal "ready" by accepting the socket; client retries connect with timeout |
| Coverage <100% on drone | A branch (likely SIGTERM emergency-snapshot path or unknown-command Alert) untested | Open HTML coverage report; identify red lines; add targeted test |

## Sub-milestone-specific gotchas

1. **WAL pragma order — non-negotiable.** Spec §1c gotcha #13. Wrong order silently degrades concurrency behavior.
2. **`tokio::time::pause()` placement.** Must be in scope BEFORE constructing any `tokio::time::Interval`. Tests fail mysteriously otherwise.
3. **SIGTERM ≠ Windows.** Use `tokio::signal::unix::SignalKind::terminate()` on Unix and `tokio::signal::windows::ctrl_break()` on Windows. Test #8 needs both code paths.
4. **Unix socket file vs Windows named pipe.** Unix socket is a filesystem object (mode 0600); Windows named pipe is a kernel object (security descriptor). Different cleanup semantics. On Unix, the socket file persists; on Windows, the pipe is destroyed when the server closes. Document in README.
5. **`tempfile`-backed test paths.** Integration test sockets must use `tempfile::tempdir()` so concurrent test runs don't collide on shared paths.
6. **Doc-test compile failures.** Every `pub` function in `runtime-drone` needs a doc comment with at least one example. The example must compile. Use `# fn main() -> Result<(), Box<dyn std::error::Error>> { ... # Ok(()) }` for examples that need fallible setup.
7. **`signal-hook` vs `tokio::signal`.** Use `tokio::signal` — the official tokio integration. `signal-hook` is more flexible but adds a dependency we don't need.
8. **Snapshot append-only.** Test #5 verifies this. Don't accidentally write `INSERT OR REPLACE`; that would deduplicate by primary key.
9. **`state_hash` is computed at write time, not read time.** Per spec §1c gotcha #14. Stored as a column, not a derived field.
10. **rusqlite blocking inside async.** Single-threaded synchronous access via `Arc<Mutex<Connection>>` is fine for M01.3's throughput. Don't reach for `tokio::task::spawn_blocking` or async pools — overkill for ~1 op/sec.
