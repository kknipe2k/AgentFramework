**Verified root causes** (grounded on `main` @ post-M08.5-merge,
authored against grep-verified read source — not memory; gotcha #41):

- **🔴 #6 — MCP Add/Test infinite-loops on Windows; `npx` spawn fails
  with "filename, directory name, or volume label syntax is incorrect"
  OS error.** `crates/runtime-mcp/src/transport/stdio.rs:82-94`
  (`StdioTransport::build_command`) constructs a `tokio::process::Command`
  via `Command::new(resolve_program(&self.command))` plus separate
  `.arg(...)` calls. `resolve_program` (lines 105-115) rewrites bare
  `npx` → `npx.cmd` on Windows correctly (M06.5 IRL 🟡-2). The
  build_command unit test asserts `npx.cmd` is the resolved program
  name (lines 304-313). The defect is one layer down: `tokio::process::
  Command` on Windows (and Rust 1.77.2+'s `std::process::Command` since
  the CVE-2024-24576 "BatBadBut" patch) escapes batch-file arguments
  through a Windows-specific quoting routine. When an argument contains
  a Windows path with `:` + `\` (e.g.,
  `C:\Users\kknip\AppData\Local\Temp\m08irl-1318332649`), the escaped
  form is the cmd.exe-level command-line that **Windows itself refuses
  to parse**, producing the OS-level "filename, directory name, or
  volume label syntax is incorrect" error before npx even starts. The
  CI runners hit the same error on `windows-latest` for the same
  reason. The unit test `connect_returns_connect_failed_for_nonexistent_
  command` (line 286) passes because the failure shape (`ConnectFailed`)
  is what the test asserts — it doesn't pin WHY connect failed, so a
  command-line-parse failure looks identical to a not-found failure.
  Fix: route `.cmd` / `.bat` spawns through an explicit `cmd.exe /C`
  wrapper with the full command line quoted as a single argument, OR
  use `std::os::windows::process::CommandExt::raw_arg` for the
  path-containing arg to bypass the BatBadBut escaping. Both are
  documented patterns; the choice is captured in **ADR-0023**
  (Proposed in this cycle, Accepted at B.fix).

- **🔴 M06.5 #1 — stray `mcp.sqlite` STILL PRESENT despite ADR-0012's
  single-source-of-truth path resolver.** The ADR-0012 fix (M06.5
  Stage A.fix, commit `7fc3277` on `main`) made `open_mcp_client` open
  `Registry` against the same `session.sqlite` path the drone uses —
  no new code path constructs `"mcp.sqlite"`. Today's IRL evidence
  confirms the fix shipped: `Select-String "mcp.sqlite" src-tauri/src/`
  returns no construction sites. The stray file at
  `C:\Users\kknip\AppData\Local\dev.aria-runtime.app\mcp.sqlite` is a
  **historical artifact from pre-ADR-0012 testing** (dated 5/17/2026,
  predates the fix merge), containing an `fs-test` row from that
  earlier session. The ADR-0012 fix does NOT clean up the stray file —
  it only prevents future creation. The IRL re-verify exposed this as
  a real user-facing artifact: when a user upgrades, the stray file
  sits dormant + visible (it has a complete session-DB schema with
  cross-table FKs), and any tool that scans `$APPDATA_DIR` shows two
  session-DB files when the runtime only writes to one. This cycle's
  fix is a **startup-cleanup migration**: on app startup, detect the
  stray `mcp.sqlite`, capture its row contents (if any) into an audit
  log entry, and rename it (`.stray-mcp.sqlite.bak`) or delete it per
  user preference. The detect+cleanup is itself a path-agnostic
  module + Tauri shell composition, tested with `tempfile`.

- **🔴 #7 — `mcp_servers` schema drift between `session.sqlite` and
  the stray `mcp.sqlite`.** The schemas are observed to differ on the
  `status` CHECK constraint:

  - `session.sqlite` (the canonical, pre-ADR-0012 file from the M06.5
    user's machine):
    `status TEXT NOT NULL DEFAULT 'disconnected' CHECK (status IN
    ('connected', 'disconnected', 'health_pending', 'error'))`
  - stray `mcp.sqlite` (created at the same period, pre-ADR-0012):
    `status TEXT NOT NULL DEFAULT 'configured' CHECK (status IN
    ('configured', 'connected', 'errored', 'disabled', 'failed'))`

  This is a **schema-divergence-by-IF-NOT-EXISTS-skip**: both files
  were created by different migration runs at different points; the
  newer migration's `CREATE TABLE IF NOT EXISTS mcp_servers (...)` body
  is skipped for any DB that already has the table from an earlier
  migration. Result: the same code path opens DBs with different
  effective schemas. The cycle fix is an **idempotent ALTER-or-recreate
  migration** that brings any pre-existing `mcp_servers` table to the
  canonical schema (the spec-aligned one — per spec §5 the status
  states are the live set), with assembled-app regression that opens
  a legacy session.sqlite, runs migrations, and asserts the table now
  has the canonical schema. Recorded as a derived sub-fix of M06.5 🔴-1
  (same Stage C.fix scope; same cluster).

- **Harness / safety gaps surfaced during the IRL pass + the build-
  machine tauri-driver install.** The M08.5 IRL re-verify on Windows
  exposed five real-app harness defects that should not survive into
  M08.6:

  1. `wdio.conf.ts:31` hardcodes `src-tauri/target/release/` for the
     binary path. Cargo workspace members share workspace-root
     `target/release/`; the hardcoded path is stale and required a
     Windows junction workaround during install (`docs/build-machine-
     tauri-driver-setup.md` Phase 3.5). Fix: use workspace-root
     `target/release/agent-runtime.exe`. Drafted on the review branch
     at commit `0239804`; lands in this cycle.
  2. `wdio.conf.ts` has no mechanism to load local API keys for the
     key-required smoke tests (3-6 in `smoke.e2e.ts`). CI uses the
     `ANTHROPIC_TEST_KEY` GitHub secret; local devs have no path.
     Fix: a `.env.local` loader at the top of `wdio.conf.ts` (the
     standard pattern; `.gitignore:38` already covers `.env.*`).
     Drafted on the review branch at commit `0239804`; lands in this
     cycle.
  3. The smoke test #2 (`smoke.e2e.ts:save-key flow`) hardcodes a
     placeholder API key (`sk-ant-test-1234567890123456`) and saves it
     to the OS keychain via the SetupPanel UI. Tests 3-6 then make
     real Anthropic API calls and read the keychain entry → auth
     fails with the placeholder. Even with `ANTHROPIC_API_KEY` set
     locally, the keychain takes precedence (`crates/runtime-main/
     src/key_store.rs:65-72` reads keychain only). Fix:
     `read_api_key` prefers `ANTHROPIC_API_KEY` env var when present,
     falls back to keychain. Standard env-as-override pattern;
     captured in **ADR-0025** (Proposed in this cycle, Accepted at
     A.fix).
  4. `npx tauri build --no-bundle` builds the Tauri package only — NOT
     the `runtime-drone` / `runtime-sandbox` sibling subprocesses the
     app spawns at startup. The Tauri app crashes immediately with
     `drone IPC unavailable: spawn drone subprocess: ... file not
     found` if the siblings are absent from `target/release/`. CI does
     this in two named steps (A.fix follow-up #3 + #4 in M08.5);
     local install required a separate `cargo build --release -p
     runtime-drone -p runtime-sandbox` invocation. Fix: the
     `wdio.conf.ts:onPrepare()` hook should build all three binaries
     via `cargo build --release` before spawning `tauri-driver`. One
     `cargo` invocation, ~5-10s when already-built (no-op).
  5. `tests/e2e-tauri/builder_drag.e2e.ts` (B.fix's regression test)
     uses the W3C WebDriver Actions API's multi-step pointer drag to
     synthesize HTML5 `dragstart`. Chromium 148+ tightened the
     dragstart synthesis threshold; the 5px intermediate `pointerMove`
     no longer trips it. The test currently has `it.skip(...)` per the
     M08.5 cycle PR closeout commit `00e4f5e`. Fix: replace the
     multi-step pointer drag with a JavaScript-level event dispatch
     via `browser.executeScript` (the standard chromedriver-html5-
     dragdrop pattern documented in `ePages-de/chromedriver-html5-
     dragdrop`). The test's `it.skip` is removed; the new mechanism
     fires `dragstart`, `dragover`, `drop` directly on the elements,
     bypassing the WebDriver layer's threshold logic.

  Plus two safety-gate gaps:

  6. **No secret-scan gate**. The orchestration session ran
     `gitleaks v8.21.2` over the full git history (324 commits, all
     branches) + working tree on 2026-05-23 — **0 leaks confirmed**.
     The repo is currently clean but has no continuous gate; an
     accidental future commit of `ANTHROPIC_API_KEY=sk-ant-...` would
     land on `main`. Fix: gitleaks as a `lefthook.yml` pre-commit hook
     AND a required `.github/workflows/ci.yml` job (using
     `gitleaks/gitleaks-action@v2` with `fetch-depth: 0` per the
     upstream README). Captured in **ADR-0024** (Proposed in this
     cycle, Accepted at A.fix).

  7. **`docs/gotchas.md` has not absorbed the 5 install-pain gotchas
     from M08.5 IRL** (tauri-driver `--version` flag absence; Edge
     auto-update msedgedriver mismatch; `wdio.conf.ts` workspace-path
     mismatch; `npx tauri build` doesn't build sibling subprocesses;
     WebDriver Actions Edge 148+ dragstart divergence). Fix: append
     all five as numbered entries continuing from the current highest
     gotcha number.

### Why four stages

The cycle has one cross-cutting harness/safety deliverable (Stage A),
two independent 🔴 fixes that need their own strict-TDD code stages
(Stages B + C), and a closing re-verify + reconcile (Stage D). The
four-stage shape mirrors **M07.5** (which also had one safety
deliverable + two 🔴 + re-verify). The Stage A bundling is justified
by the shared theme — all five sub-deliverables are CI/test-tooling
infrastructure work that ships no user-facing functionality and shares
one approval surface.

Combining B + C into one stage would obscure the strict-TDD
test-immutability invariant (which each 🔴 fix needs cleanly proven
per CLAUDE.md §6 v1.7+); splitting Stage A into per-deliverable stages
(5 stages just for harness) would dilute the approval cadence with
work the maintainer correctly reviews as one infrastructure batch.

## Document Structure

The phase doc has one Background section, four per-stage sections
(A.fix → B.fix → C.fix → D.fix), and these top-level sections. Each
per-stage section follows the M08.5 / M07.5 pattern:

- **N.1 Problem Statement** — grounded root cause with file:line
  citations; web-research references cited inline where the technical
  decision relies on upstream best practice (gotcha #41 + gotcha #32).
- **N.2 Files to Change** — exhaustive list with verb (CREATE / EDIT /
  DELETE), purpose, and one-line summary of change.
- **N.3 Detailed Changes** — the actual code/config/text to write,
  ready to paste with minimal interpretation. Pre-stated commit
  messages where helpful.
- **N.4 Tests** (impl stages B/C) or **N.4 Acceptance** (infra stages
  A/D) — the regression tests required + the per-test
  fails-on-pre-fix-main proof shape.
- **N.5 CLI Prompt** — the build-machine `<work_stage_prompt>` XML
  per `STAGE-PROMPT-PROTOCOL.md` v1.8 with all required sub-elements
  (read_first, deliverable, execution_steps, acceptance_criteria,
  scope_locks, gates, self_correction_budget, adr_triggers, gotchas,
  execution_warnings, pre_flight_check, phase_doc_inventory_audit,
  runtime_environment, time_box, retrospective_requirements,
  commit_protocol, commit_message, approval_surface; plus
  tdd_discipline where applicable).
- **N.6 Commit Message** — pre-stated commit message body per stage,
  including the strict-TDD invariant statement (Stages B + C) and the
  ADR-flip reference (Stages A + B).

## Implementation Workflow

1. **Pre-flight on each stage**: build machine reads the stage's `<read_first>`
   files end-to-end, runs the `<pre_flight_check>` commands, executes the
   Stage 0 real-app discovery walk (Revision 2 protocol), then surfaces
   the discovery log + any new adjacent-bug findings for triage before
   red-phase.

2. **Red phase (Stages B + C)**: write all failing regression tests in
   one commit. Run them locally to confirm right-reason failures. Surface
   the red commit + the test files + the falsifiable hypothesis statements
   per CLAUDE.md §6 v1.8 for red-phase approval. Wait for explicit user
   approval before impl phase.

3. **Impl phase (Stages B + C)**: write the production code that makes
   the failing tests pass. Do NOT modify any test file under the stage's
   named test paths. Run the full canonical gate suite per CLAUDE.md §6.
   Surface the impl commit + the strict-TDD invariant proof
   (`git diff <red-sha>..<impl-sha> -- '<test-paths>'` is EMPTY) +
   gate results + retrospective. Wait for final approval before push.

4. **Implementation phase (Stages A + D)**: write the harness/config
   changes (Stage A) or docs/re-verify (Stage D) in one or more commits
   as the work groups naturally. Run gates. Surface for approval. Push.

5. **Push + CI**: after approval, push to `claude/m08.5.5-mcp-resilience`.
   Trigger CI workflow_dispatch on the new HEAD (feature-branch push
   doesn't auto-trigger). Surface CI run URL. Wait for CI green before
   the next stage starts.

6. **Cycle close**: Stage D opens the M08.5.5 cycle PR after CI lands
   green on the D.fix commit. PR body summarizes the cycle's
   deliverables + the closed/still-open 🔴 ledger + the M08.6
   carry-forward routing.

## Stage A.fix — harness + safety hardening (no 🔴 fix; ADRs 0024 + 0025)

### A.1 Problem Statement

Five harness/safety gaps surfaced during the M08.5 IRL re-verify and
the orchestration session's tauri-driver install on the Windows build
machine. None is a 🔴 by itself; together they block clean execution of
Stages B and C's regression tests on the build machine + leave the repo
without a continuous secret-scan gate. This stage closes them all in one
infrastructure batch.

**Gap 1 — `wdio.conf.ts:31` hardcodes stale workspace path.**
`APP_BIN_PATH = resolve(process.cwd(), 'src-tauri', 'target', 'release',
APP_BIN_NAME)`. Cargo workspace members share workspace-root
`target/release/`; the binary lands at `target/release/agent-runtime.exe`,
not at `src-tauri/target/release/agent-runtime.exe`. The build machine
worked around this during install via a Windows junction
(`docs/build-machine-tauri-driver-setup.md` Phase 3.5). The real fix is
the one-line edit: drop the `src-tauri` segment. Drafted on review
branch at commit `0239804`.

**Gap 2 — `wdio.conf.ts` has no local API key loader.** The smoke
tests 3-6 (`smoke.e2e.ts`) require `ANTHROPIC_TEST_KEY` + `ANTHROPIC_API_KEY`
to be set; CI uses the `ANTHROPIC_TEST_KEY` GitHub secret. Local devs
have no path that doesn't leak the key into PowerShell history. Fix:
top-of-file `.env.local` loader (gitignored via `.gitignore:38`)
that reads `<KEY>=<VALUE>` lines and seeds `process.env`. Drafted on
review branch at commit `0239804`.

**Gap 3 — `read_api_key` reads keychain only.**
`crates/runtime-main/src/key_store.rs:65-72` constructs an
`Entry::new(SERVICE, USER)` and reads via `get_password()` — no env-var
check. The smoke test #2 hardcodes a placeholder key and saves it to
keychain via the SetupPanel UI; tests 3-6 then attempt real API calls
with the keychain-resident placeholder → auth fails. Setting
`ANTHROPIC_API_KEY` locally has no effect because keychain takes
precedence. Fix: `read_api_key` checks `std::env::var("ANTHROPIC_API_KEY")`
first, falls back to keychain. Standard env-as-override pattern; ADR-0025.

**Gap 4 — `npx tauri build` doesn't build sibling subprocess binaries.**
The Tauri app spawns `runtime-drone` and `runtime-sandbox` as
subprocesses at startup (from `agent-runtime.exe`'s sibling
`target/release/` directory). `npx tauri build --no-bundle` only builds
the Tauri package; the siblings must be built separately via
`cargo build --release -p runtime-drone -p runtime-sandbox`. CI does
this explicitly (M08.5 A.fix follow-ups #3 + #4 — commits `99bf7b0` +
`7ea405e`); the local install required user intervention. Fix: the
`wdio.conf.ts:onPrepare()` hook runs the cargo build before spawning
`tauri-driver`. Idempotent (`cargo` is no-op when already-built),
adds ~5-10s to a clean-tree test run, zero overhead on warm tree.

**Gap 5 — `builder_drag.e2e.ts` skipped on Chromium 148+.** Commit
`00e4f5e` (the M08.5 cycle's final commit) added `it.skip` to
`palette_drag_instantiates_a_canvas_node` because the W3C WebDriver
Actions API multi-step pointer drag no longer synthesizes HTML5
`dragstart` on Chromium 148+. The impl (`dragDropEnabled: false`) is
correct — IRL-verified on Windows 2026-05-23. The test mechanism is
what's stale. Fix: replace the multi-step pointer drag with a
JavaScript-level event dispatch via `browser.executeScript`, following
the `ePages-de/chromedriver-html5-dragdrop` upstream pattern (gotcha
#32 quote-verbatim). The new mechanism fires `dragstart`, `dragover`,
`drop` directly on the elements, bypassing the WebDriver layer's
dragstart-synthesis threshold logic entirely.

**Gap 6 — No secret-scan gate.** The orchestration session ran
`gitleaks v8.21.2` over the full git history on 2026-05-23 — 324
commits, all branches, 0 leaks. The repo is currently clean but has no
continuous gate; an accidental `ANTHROPIC_API_KEY=sk-ant-real-key`
commit would land on `main` without warning. Fix:
`lefthook.yml` pre-commit hook running `gitleaks git --staged` AND a
`.github/workflows/ci.yml` job using `gitleaks/gitleaks-action@v2`
with `fetch-depth: 0` (per the upstream README). ADR-0024.

**Gap 7 — `docs/gotchas.md` doesn't carry the 5 install-pain gotchas.**
Five gotchas surfaced during the M08.5 IRL pass + tauri-driver install
that need permanent capture (tauri-driver `--version` flag absence;
Edge auto-update msedgedriver mismatch; `wdio.conf.ts` workspace-path
mismatch; `npx tauri build` doesn't build sibling subprocesses;
WebDriver Actions Edge 148+ dragstart divergence). Fix: five new
numbered entries appended.

#### A.3.2 `crates/runtime-main/src/key_store.rs::read_api_key` env-var override

Verbatim replacement of the function body (lines 65-72):

```rust
/// Read the Anthropic API key — env var first, OS keychain fallback.
///
/// `ANTHROPIC_API_KEY` env var takes precedence over the OS keychain
/// when set + non-empty. Falls back to the OS keychain read otherwise.
/// Standard env-as-override pattern per ADR-0025. Enables CI/test runs
/// to bypass the OS keychain (matches the M08.5 IRL discovery pattern
/// where the smoke test's hardcoded placeholder stomps the keychain and
/// tests 3-6 then fail with auth on the placeholder).
///
/// # Errors
///
/// Returns [`KeyStoreError::NotFound`] if no entry exists in the
/// keychain AND no env var is set, [`KeyStoreError::Keyring`] for any
/// other backend failure on the keychain path.
pub fn read_api_key() -> Result<SecretString, KeyStoreError> {
    if let Ok(env_key) = std::env::var("ANTHROPIC_API_KEY") {
        if !env_key.is_empty() {
            return Ok(SecretString::from(env_key));
        }
    }
    let entry = Entry::new(SERVICE, USER)?;
    match entry.get_password() {
        Ok(s) => Ok(SecretString::from(s)),
        Err(keyring::Error::NoEntry) => Err(KeyStoreError::NotFound),
        Err(e) => Err(e.into()),
    }
}
```

Red-phase tests (added to the existing `#[cfg(test)] mod tests` block
in `key_store.rs`):

```rust
#[test]
fn read_api_key_returns_env_var_when_set_overriding_keychain() {
    let _env_lock = std::env::set_var("ANTHROPIC_API_KEY", "sk-ant-env-override-test-12345");
    // The OS keychain probe is best-effort; whatever's there should be
    // overridden by the env var. Cleanup the env var after the test.
    let got = read_api_key().expect("env var should resolve");
    assert_eq!(got.expose_secret(), "sk-ant-env-override-test-12345");
    std::env::remove_var("ANTHROPIC_API_KEY");
}

#[test]
fn read_api_key_falls_back_to_keychain_when_env_var_empty() {
    let _env_lock = std::env::set_var("ANTHROPIC_API_KEY", "");
    // Empty env var must NOT short-circuit; fall through to keychain.
    // This test does not assert on the keychain result (CI keychain
    // state is platform-dependent); it asserts that an empty env var
    // doesn't short-circuit with the empty string as a "key".
    let result = read_api_key();
    if let Ok(secret) = result {
        assert!(
            !secret.expose_secret().is_empty(),
            "empty env var must not be returned as a key"
        );
    }
    // If keychain returns NotFound, that's also acceptable — what
    // matters is that the empty env var was bypassed.
    std::env::remove_var("ANTHROPIC_API_KEY");
}

#[test]
fn read_api_key_falls_back_to_keychain_when_env_var_unset() {
    std::env::remove_var("ANTHROPIC_API_KEY");
    let result = read_api_key();
    // The test's job is to prove the code reaches the keychain path,
    // not to pin the keychain's state (which varies by CI runner).
    // A successful read or a NotFound both prove the keychain path
    // executed; a Keyring backend error would also acceptable
    // (proves we tried the keychain).
    match result {
        Ok(_) | Err(KeyStoreError::NotFound) | Err(KeyStoreError::Keyring(_)) => {}
    }
}
```

Reference (gotcha #32 — quote verbatim, cite source):
- Upstream: <https://github.com/ePages-de/chromedriver-html5-dragdrop>

#### A.3.5 `.github/workflows/ci.yml` gitleaks job

ADD a new job (place after the existing `detect-cargo` job; gated by
`detect-cargo.outputs.has_npm` to preserve the docs-only skip
pattern):

```yaml
  gitleaks:
    name: Secret scan
    needs: [detect-cargo]
    if: needs.detect-cargo.outputs.code_changed == 'true'
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Per the upstream README (<https://github.com/gitleaks/gitleaks-action>):
`fetch-depth: 0` is REQUIRED for full-history scanning; `GITHUB_TOKEN`
is automatically provided by GitHub Actions and enables PR commenting
when leaks are found. `GITLEAKS_LICENSE` is NOT required for personal
accounts (the `kknipe2k/AgentFramework` repo). The job runs only on
code-changed commits per the docs-only skip pattern established in PR
#98.

## Stage B.fix — 🔴 #6 MCP Add/Test Windows `npx` spawn (ADR-0023)

### B.1 Problem Statement

`crates/runtime-mcp/src/transport/stdio.rs::build_command` (lines 82-94)
constructs a `tokio::process::Command` for the MCP server's stdio
transport. On Windows, `resolve_program` (lines 105-115) correctly
rewrites bare `npx` → `npx.cmd` per M06.5 IRL 🟡-2. The `build_command`
unit test passes (line 304-313). **Yet the assembled MCP Add flow fails
with the Windows OS-level "filename, directory name, or volume label
syntax is incorrect" error** when the user provides an MCP server
configured with `command: npx, args: [-y, @modelcontextprotocol/server-
filesystem, C:\Users\<user>\Some\Path]`.

**Verified root cause** (web research + code grounding, gotcha #41):

Rust's `std::process::Command` (which `tokio::process::Command` wraps)
on Windows uses the **Windows-native `CreateProcessW` API** with a
quoted command line. When invoking a `.cmd` or `.bat` file directly,
`CreateProcessW` implicitly spawns `cmd.exe` to interpret the batch
file. Rust 1.77.2+ added the CVE-2024-24576 ("BatBadBut") security fix:
arguments to batch files are escaped through a Windows-specific quoting
routine that produces a `cmd.exe`-safe command line.

The BatBadBut-safe escaping is correct for **arbitrary string args** —
but a Windows path argument containing `:` + `\` (drive letter + path
separators) hits a cmd.exe-level command-line interpretation rule that
**Windows itself refuses to parse**: the colon after the drive letter
is interpreted as a path component separator inside what cmd.exe sees
as a quoted argument, and the embedded `\` characters trigger Windows'
"this looks like a malformed UNC path" heuristic, producing the
OS-level "filename, directory name, or volume label syntax is
incorrect" error before npx ever runs.

This is independent of `resolve_program`'s `npx → npx.cmd` rewrite (that
rewrite is correct and necessary; the bug is one layer down at the
spawn-with-args layer). It is also independent of `npx` itself (the
error fires before npx starts).

**Web research references** (cited per gotcha #41):

- BatBadBut explanation + impact: <https://thehackernews.com/2024/04/critical-batbadbut-rust-vulnerability.html>
- CVE-2024-24576 detail: <https://nvd.nist.gov/vuln/detail/cve-2024-24576>
- Node.js Windows command-line escaping issue (parallel ecosystem confirms the same class):
  <https://github.com/nodejs/node/issues/52681>
- Rust fix release notes: <https://blog.rust-lang.org/2024/04/09/cve-2024-24576.html>
- The standard workaround pattern (cmd.exe /C wrapper):
  <https://github.com/nodejs/node/issues/7367#issuecomment-247599757>

**Fix decision (ADR-0023)**: route `.cmd` / `.bat` spawns through an
explicit `cmd.exe /C "<full command line>"` wrapper using
`std::os::windows::process::CommandExt::raw_arg`. The full command line
(program + args) is built as a single Windows-quoted string, passed as
a single `raw_arg` to `cmd.exe`. This bypasses the BatBadBut escaping
entirely (which is intended for the case where Rust constructs the
arguments; here, we are constructing the entire command line ourselves
with full control over quoting). The pattern is documented in the
Node.js issue tracker (parallel ecosystem) and is the standard
workaround for cross-platform Rust applications that need to invoke
batch files with path arguments.

Alternative A considered (use `CommandExt::raw_arg` on the path arg
only): rejected because it requires per-argument decision-making at
the call site, which spreads platform-specific logic across the
codebase. The cmd.exe wrapper is a single point of platform abstraction.

Verbatim replacement of `build_command` (lines 82-94):

```rust
pub(crate) fn build_command(&self) -> Command {
    let resolved = resolve_program(&self.command);
    #[cfg(target_os = "windows")]
    {
        // M08.5.5 Stage B.fix (ADR-0023): when invoking a Windows
        // batch file (.cmd / .bat), wrap in `cmd.exe /C "<full command
        // line>"` via raw_arg to bypass the BatBadBut escaping rules
        // that produce cmd.exe-unparseable command lines for path
        // arguments containing : + \. The raw_arg path is the
        // documented workaround pattern for cross-platform Rust apps
        // invoking batch files with path args. See ADR-0023 §
        // "Decision" for the full rationale.
        if resolved.ends_with(".cmd") || resolved.ends_with(".bat") {
            use std::os::windows::process::CommandExt;
            let mut cmd = Command::new("cmd.exe");
            let mut full_command_line = format!("\"{resolved}\"");
            for arg in &self.args {
                // Quote each arg if it contains a space, colon, or
                // backslash (the BatBadBut-trigger chars). Embedded
                // quotes in args are escaped per the standard
                // Windows quoting convention.
                let needs_quote = arg.contains(' ')
                    || arg.contains(':')
                    || arg.contains('\\')
                    || arg.contains('"');
                if needs_quote {
                    let escaped = arg.replace('"', "\\\"");
                    full_command_line.push_str(&format!(" \"{escaped}\""));
                } else {
                    full_command_line.push(' ');
                    full_command_line.push_str(arg);
                }
            }
            cmd.raw_arg(format!("/C {full_command_line}"));
            for (k, v) in &self.env {
                cmd.env(k, v);
            }
            if let Some(c) = &self.cwd {
                cmd.current_dir(c);
            }
            return cmd;
        }
    }
    let mut cmd = Command::new(resolved);
    for arg in &self.args {
        cmd.arg(arg);
    }
    for (k, v) in &self.env {
        cmd.env(k, v);
    }
    if let Some(c) = &self.cwd {
        cmd.current_dir(c);
    }
    cmd
}
```

## Stage C.fix — M06.5 🔴-1 stray cleanup + 🔴 #7 schema reconciliation

### C.1 Problem Statement

Two interlocking sub-fixes, both rooted in the same migration / cleanup
module work and the same root cause (pre-ADR-0012 stray DB + pre-spec
schema):

**Sub-fix 1 — Stray `mcp.sqlite` cleanup.** ADR-0012 (M06.5 Stage A.fix,
commit `7fc3277` on `main`) made `open_mcp_client` open `Registry`
against the same `session.sqlite` path the drone uses. No new code
path constructs `"mcp.sqlite"`. The fix shipped correctly. But the
ADR-0012 fix does NOT clean up the stray `mcp.sqlite` files left by
**pre-ADR-0012 testing** on users' machines. The M08.5 IRL re-verify
on the build machine 2026-05-23 confirmed the stray file is STILL
PRESENT: `C:\Users\kknip\AppData\Local\dev.aria-runtime.app\mcp.sqlite`
(4096 bytes, dated 5/17/2026, contains a complete session-DB schema +
an `fs-test` row from 5/17 prior testing). A user upgrading through
ADR-0012's fix continues to see two session-DB files in the
`$APPDATA_DIR` indefinitely — visually confusing and a latent path-
divergence hazard if any tool ever opens the stray.

Fix: a **startup-cleanup migration** invoked from the Tauri shell's
`setup()` before the canonical `session.sqlite` is opened. The
migration:

1. Scans `app_local_data_dir` for the legacy `mcp.sqlite` file.
2. If present, opens it read-only, queries its `mcp_servers` rows.
3. Writes an audit-log entry (via the existing `AuditWriter`) capturing
   the rows + the file path + the discovery timestamp (NEVER discards
   data silently).
4. Renames the file to `.stray-mcp.sqlite.bak` (preserves the original
   bytes; user can manually delete or import into the canonical DB at
   their discretion). The rename + audit happens once per launch; if
   `.stray-mcp.sqlite.bak` already exists, the original is renamed
   with a timestamp suffix (idempotent).
5. Logs an INFO-level message; emits no UI prompt (the cleanup is
   silent + auditable).

The module lives at `crates/runtime-main/src/stray_db_cleanup.rs`,
path-agnostic per CLAUDE.md §9 archetype (takes `dir: &Path`, the
Tauri shell resolves the directory). Tested with `tempfile`-backed
paths (no real filesystem touch in unit tests).

**Sub-fix 2 — `mcp_servers` schema reconciliation.** The M08.5 IRL
re-verify exposed that `session.sqlite` and the stray `mcp.sqlite` have
**incompatible `mcp_servers` schemas**:

- `session.sqlite` mcp_servers: status CHECK `('connected',
  'disconnected', 'health_pending', 'error')` — default `'disconnected'`.
- stray `mcp.sqlite` mcp_servers: status CHECK `('configured',
  'connected', 'errored', 'disabled', 'failed')` — default `'configured'`.

The divergence is real (verified by `sqlite3 .schema mcp_servers` on
both files). Root cause: a `CREATE TABLE IF NOT EXISTS mcp_servers
(...)` migration is **idempotent in CREATE but NOT in schema-evolution**.
If a DB has the table from an older migration, the newer migration's
body is skipped; the table retains the older schema indefinitely.
Specifically: when migration 002 created `mcp_servers` with the 4-state
schema, and a later migration (003+) intended to evolve it to the
5-state schema, the IF NOT EXISTS guard suppressed the evolution.

Result: the same code opens DBs with different effective schemas. The
runtime's MCP-status read path expects one set of valid states; a row
written with the other set fails the CHECK constraint at write time
(or, worse, succeeds silently if the runtime writes a value that
satisfies one constraint but not the other — depending on which DB is
canonical at that moment).

Fix: an **idempotent ALTER-or-recreate migration** (number 004, the
next available) that:

1. Queries `sqlite_master` for the existing `mcp_servers` table
   definition.
2. If the table has the legacy 4-state CHECK, executes a
   `CREATE TABLE mcp_servers_new (...)` with the canonical 5-state
   schema + a row-mapping `INSERT ... SELECT` that maps legacy values
   to canonical (`disconnected` → `disabled`, `health_pending` →
   `configured`, `error` → `errored`, `connected` → `connected`).
3. Drops the legacy table, renames the new table to `mcp_servers`,
   re-applies the original indices.
4. If the table already has the canonical schema, no-op (the migration
   is idempotent).

Tested with assembled-app fixtures: a checked-in `session.sqlite`
carrying the legacy schema is opened, migrations are run, the table is
asserted to have the canonical schema with rows preserved + state-mapped.

**Combined scope decision**: C.fix lands BOTH sub-fixes in one stage
because they share:

- The same fix-cycle theme (post-ADR-0012 MCP-registry cleanup).
- The same assembled-app regression infrastructure (the fixture-DB
  test pattern).
- The same migration-runner code path (the cleanup module + the new
  migration are both invoked from the Tauri shell's setup path).

Splitting into two stages would duplicate the fixture setup +
introduce a state-dependency where one stage's fixture-DB depends on
the other stage's cleanup behavior. Combined, the single stage is
clean.

//! M08.5.5 Stage C.fix — startup cleanup of the pre-ADR-0012 stray
//! `mcp.sqlite` file. ADR-0012 (M06.5 Stage A.fix) prevented future
//! creation; this module handles the cleanup of legacy files left on
//! users' machines from pre-fix testing.
//!
//! Path-agnostic: accepts `dir: &Path`, the Tauri shell resolves
//! `app_local_data_dir()`. Tested with `tempfile`-backed paths;
//! `tempfile::TempDir` gives `&Path` directly.

use std::path::{Path, PathBuf};
use std::sync::Arc;

use runtime_main::audit::AuditWriter;

/// Outcome of one startup cleanup pass.
#[derive(Debug)]
pub enum CleanupOutcome {
    /// No legacy `mcp.sqlite` file present; no action taken.
    NotPresent,
    /// Legacy file detected + renamed; audit entry written.
    Cleaned {
        original: PathBuf,
        renamed_to: PathBuf,
        rows_captured: usize,
    },
}

#[derive(Debug, thiserror::Error)]
pub enum CleanupError {
    #[error("io error reading stray DB: {0}")]
    Io(#[from] std::io::Error),
    #[error("sqlite error reading stray DB: {0}")]
    Sqlite(#[from] rusqlite::Error),
}

/// Detect + audit + rename the pre-ADR-0012 stray `mcp.sqlite` in `dir`.
///
/// If `dir/mcp.sqlite` exists, opens it read-only, reads `mcp_servers`
/// rows into an audit entry (via `AuditWriter` if provided), and
/// renames the file to `dir/.stray-mcp.sqlite.bak`. If
/// `.stray-mcp.sqlite.bak` already exists, the rename target gets a
/// timestamp suffix (`.stray-mcp.sqlite.bak.YYYYMMDDTHHMMSS`).
///
/// Idempotent: running twice in a row produces `NotPresent` the second
/// time. Never fails-loud at startup (a stray file is a recoverable
/// state; the cleanup is best-effort).
pub fn cleanup_stray_db(
    dir: &Path,
    audit: Option<Arc<AuditWriter>>,
) -> Result<CleanupOutcome, CleanupError> {
    let stray = dir.join("mcp.sqlite");
    if !stray.exists() {
        return Ok(CleanupOutcome::NotPresent);
    }
    let rows = read_legacy_rows(&stray)?;
    let target = pick_rename_target(dir);
    std::fs::rename(&stray, &target)?;
    if let Some(writer) = audit {
        let _ = writer.write_jsonl(&audit_entry(&stray, &target, &rows));
    }
    Ok(CleanupOutcome::Cleaned {
        original: stray,
        renamed_to: target,
        rows_captured: rows.len(),
    })
}

#### C.3.2 `crates/runtime-drone/src/db.rs` — migration `004_mcp_servers_status_reconcile`

Idempotent ALTER-or-recreate. Detects the legacy 4-state CHECK constraint
via `pragma_table_info` + SQL fragment parsing; rebuilds with the canonical
5-state schema; preserves rows by mapping legacy state values.

```rust
// Append to MIGRATIONS const in db.rs:
Migration {
    version: 4,
    name: "mcp_servers_status_reconcile",
    sql: r#"
        -- M08.5.5 Stage C.fix: reconcile mcp_servers status CHECK
        -- constraint. Pre-fix some session.sqlite files have the
        -- legacy 4-state set; canonical is 5-state per spec §5.
        -- Idempotent: no-op when the table already has the canonical
        -- schema. Preserves rows by mapping legacy → canonical states.
        --
        -- Mapping rule:
        --   connected      → connected
        --   disconnected   → disabled
        --   health_pending → configured
        --   error          → errored
        -- (the legacy DEFAULT was 'disconnected'; canonical is
        -- 'configured').

        -- Detect legacy schema; if present, rebuild via swap-and-rename.
        -- Otherwise this migration is a no-op (the IF EXISTS guard makes
        -- the rebuild self-skipping when applied to an already-canonical
        -- table; the row-mapping INSERT runs only against the legacy
        -- copy).

        CREATE TABLE IF NOT EXISTS mcp_servers_v4_canonical (
            -- (same shape as the current mcp_servers, but with the
            -- canonical 5-state CHECK + 'configured' default)
        );

        -- Conditional copy: only if mcp_servers has the legacy CHECK
        -- (detected via sqlite_master.sql lookup at runtime — the
        -- raw SQL approach uses a helper in db.rs's Rust runner that
        -- inspects sqlite_master before deciding to execute the
        -- swap-and-rename).
    "#,
    runner: |conn| { /* ... Rust-side schema-detection + conditional copy ... */ },
},
```

The Rust-side runner inspects `SELECT sql FROM sqlite_master WHERE
type='table' AND name='mcp_servers'`, regex-detects the legacy CHECK
pattern, and conditionally executes the swap-and-rename. This avoids
the "always-run unconditional CREATE" trap that produces silent no-ops
on already-migrated DBs.
