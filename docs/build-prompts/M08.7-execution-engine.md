## Background and Design Decision

**Problem.** M08.6 IRL surfaced the load-bearing observation that drove this milestone (`docs/M08.6-irl-findings.md` + `ORCHESTRATOR.md` §9): the runtime *paints* most of what it claims. **Only provider-streaming and MCP-tool dispatch actually execute.** A built-in `ToolUse` for `Read` emits `ToolInvoked` and stops — no file is read, no result feeds back. Sub-agents emit `AgentSpawned` and stop — no child runs. `LoadSkill` has no handler. `request_capability` has a 314-line handler with no production caller. `drive_plan` advances the FSM but its own docstring says "runs no tasks." Hooks/rails are defined in framework JSON with no firing engine. The Builder ships green; ARIA loads and renders; the v0.1 acceptance criteria depend on a runtime that runs, and at M08.6 end-of-milestone it doesn't.

**Solution.** M08.7 builds the execution layer as an **eval-first ladder of clusters** (`docs/cluster-pattern.md` §1) — each rung isolates one execution function against a minimal hand-authored framework so a break is diagnosable, each rung closes only when the assembled thing runs and is IRL-watched, and ARIA is rung N (the integration test, not the unit-of-work). M08.7a (rungs 1–5) ships the single-agent execution primitives — built-in tool execution (Stage A — already built), capability-on-live-tool (Stage B), skill load (Stage C), gap detection (Stage D), and budget enforcement at the run loop (Stage E). M08.7b (rungs 6–7) ships sequential multi-agent + plan-driven tasks; M08.7c (rungs 8–9, N) ships hooks/rails, MCP fix, and ARIA. M08.7b/c are structured here and authored to full X.1–X.6 detail at each sub-phase entry — because rung 6+ specs depend empirically on what M08.7a's executor + feedback wiring actually look like (e.g., whether per-child execution reuses `drive_stream` as-is or needs a re-entrant variant is a rung-6 question, not an authoring-time guess).

**Why one PR for the parent milestone.** Per the M01–M08 precedent: stages A–E commit sequentially on the parent-milestone branch (`claude/m08.7-execution-engine`); Stage V runs in a fresh CLI session; closeout authors the summary + the immutable `docs/gap-analysis.md` entry; the milestone PR drafts at closeout with every stage commit + every retrospective + the gap-analysis entry surfaced together (three-artifact review per CLAUDE.md §20). Rungs accumulate on one branch; no per-rung PR.

**Why six stages (A as-built + B–E + V + closeout) here, not "one stage per rung" or "one mega-stage."** Stage A is rung 1 (the executor + the multi-turn feedback contract every later rung mirrors) and is **already shipped on `claude/m08.7-execution-engine`** — committed `f7f9b7e..f9b02bf`, IRL-observed (real Anthropic model invoked `Read` → executor read `Cargo.toml` → model quoted contents back), mutation gate 0-missed, retro + IRL addendum committed. B–E are sized 2–4 h each — rung 2 is largely verification of an already-tested denial path; rung 3 is the only true greenfield (no `LoadSkill` handler exists); rungs 4 and 5 are wires (handler + budget enforcer both exist as unit-tested primitives — production callers are absent). Splitting per-rung gives each cluster its own §X.5 prompt + its own IRL gate + its own retro + zero-propagation triage in-rung. Stage V runs the protocol v1.9 5th pass (`assembled_execution`) and is the structural guard against another M08.6-class escape.

### What this milestone produces

1. **Built-in tool executor** (Stage A — shipped) — `crates/runtime-main/src/sdk/builtin_tools.rs` (195 lines, new); the `drive_stream` built-in branch routing a `Read`/`Write` `ToolUse` through `CapabilityEnforcer::check` against `file_access`, executing in-process, emitting `ToolInvoked`/`ToolResult`, and returning a result that joins `feedback.dispatched` exactly as the MCP path's `DispatchedTool` does. **The execution + multi-turn-feedback archetype every later rung mirrors.**
2. **Capability enforcement gates a live tool** (Stage B — rung 2) — a Promoted-tier `Write` outside `file_access.write` scope emits `CapabilityViolation(Write)` (the scope-gate denial, not the L4-tier denial that rung 1 already covers) and the target file does NOT exist on disk; an in-scope Write creates the file with its content, through the assembled loop. Closes the BDD `Scenario: A write outside file_access scope does not run` + `Scenario: A write inside scope does run`.
3. **Skill load** (Stage C — rung 3) — a `LoadSkill` handler that reads the resolved skill markdown (the M08.6 loader already resolves skill companions to bodies) and injects it into the agent's context for subsequent turns, emitting `SkillLoaded` (the variant exists at `crates/runtime-core/src/event.rs:279` — schema-defined, never emitted; `replay.rs:146` knows how to project it). The grounded acceptance is that **skill content changes observable agent behavior** (the BDD: a skill saying "always reply in ALL CAPS" makes the agent reply in all caps), not the `SkillLoaded` event alone.
4. **Gap detection wired into the run loop** (Stage D — rung 4) — a model `request_capability` `ToolUse` routes from `drive_stream` into `handle_request_capability` (`crates/runtime-main/src/sdk/request_capability.rs:88` — exists with full unit tests, no production caller), emits the appropriate `*Missing` gap event (`event.rs:479-527`), and suspends the session cleanly per the gap flow (spec §4 — the project's signature behavior). Grounded acceptance: the session **suspends, recoverable, does not crash/continue** — observed, not inferred.
5. **Budget enforcement at the run loop** (Stage E — rung 5) — the budget primitive is **further along than the original M08.7 doc claimed**: `crates/runtime-main/src/budget/` has `cost.rs` + `enforcer.rs` + `hook.rs` + `mod.rs`; `warn_at_percent` is wired in `enforcer.rs:73,116`; the opus→sonnet→haiku ladder is real (`hook.rs:6-11`); every provider call already produces a `CostBreakdown` (`providers/anthropic.rs:147,292`). The gap is the **run-loop wire**: spend produced per turn does not drive `warn` / `downshift` / `hard_stop` on the live session. Stage E wires the existing enforcer into `drive_stream`'s per-turn loop. Grounded acceptance: a session with a tiny `session_usd_cap` exceeds the cap → the session stops with a budget-stop reason → no further provider turns are issued, observed.
6. **Two ADRs.** ADR-0026 — **The built-in tool execution contract** (Stage A's archetype frozen as the multi-turn feedback contract every rung mirrors; the file_access-scope-is-the-boundary decision; the in-process-vs-sandbox split with `Bash` deferred to a separate ADR-class rung). ADR-0027 — **Skill-into-context injection model** (Stage C; what *section* of the assembled prompt the loaded skill body lands in; whether it persists across turns; how a second `LoadSkill` composes — these touch the §0b system-prompt-assembly job that proposals `0001-interaction-layer-roadmap.md` + `harness-review-takeaways.md` Item 2 flagged as v0.1-load-bearing).
7. **Stage V** five-pass verifier (Inventory + Wire + Behavior + Multi-call invariants + the v1.9 **`assembled_execution`** 5th pass — V runs the assembled app / integration tests and OBSERVES each rung execute, not just confirms tests exist).
8. **Closeout** — M08.7a summary + immutable `docs/gap-analysis.md` entry + `<simplify_pass>` + `<coverage_policy_reconciliation>` + the M08.7b/c sub-phase entry handoff.

### What's not in scope

- **`Bash` + any OS-spawning or out-of-scope op** — separate ADR-class rung (the sandbox has no `RunCommand` variant — `crates/runtime-sandbox/src/protocol.rs:29` is `ValidateArtifact | Shutdown` only; adding one is an IPC-protocol change → §11 ADR + CODEOWNERS-flagged). M08.7a is in-process file tools only. If a stage finds it cannot proceed without the sandbox exec path, it STOPS and surfaces — that's the trigger to split, not to widen.
- **Concurrent multi-agent orchestration** — v1.0 (sequential is the v0.1 floor that proves the engine; parallel is the v1.0 follow-on per the multi-agent research). M08.7b rung 6 ships *sequential* spawn — child runs, returns a summary, parent continues.
- **Mode router** — STANDARD stays hardcoded (§0d; gotcha #3). The mode router is v1.0.
- **New provider** — Anthropic-only per §0d. The capability **grant-right vs use-right split** (AWS PassRole / macaroon pattern — let a coordinator be the source of a delegated capability while attenuating its own use) is the correct v1.0 ADR; it still requires the orchestrator to hold a grant-right and is NOT a v0.1 capability-model change. Strict child ⊆ parent narrowing stays (validated against object-capability theory + AI-agent delegation best practice — you may only delegate authority you hold; attenuation only narrows).
- **The Builder authoring surface** — that's M08.6.7's track, not this one. M08.7 fixtures are hand-authored JSON loaded from disk, so M08.7 does not depend on Builder correctness. M08.6.7 absorbs the 32 M08.6 IRL findings as clusters after M08.7c closes (sequencing: M08.7 → Claude Design pass → M08.6.7 → M09 mentor).
- **No new third-party deps** without `cargo deny`. No telemetry. No `unsafe` outside `runtime-sandbox`. No schema change unless a rung proves one is needed (then: new `*.v2.json` + ADR).

### Why the ladder framing is preserved on top of M08-style staging

The M08.7 framing introduced in PR #109 (commit `f15f916`) was right and stays: each rung is a cluster; rungs accumulate on one branch; M08.7a/b/c are sub-phase boundaries with clear endings. The change in this rewrite is to apply the **M08 phase-doc shape** (per-stage X.1–X.6, single Document Structure table, single Implementation Workflow, single Pre-existing legacy file inventory, Summary Table + Verification Checklist at the end) so the build CLI consumes each stage's prompt from a doc that matches every prior milestone's authoring pattern — same parser, same retro template, same closeout shape. Rungs = stages; the ladder is the milestone's deliverable map; the stage shape is the build-CLI contract.

### Key constraints (all stages)

- **Cluster-gate close discipline** (`docs/cluster-pattern.md` §1 step 4). Every stage's BDD acceptance + assembled test is authored before the build (red), runs the *real* path, and the stage does not close until the maintainer IRL-watches it run. The §X.5 prompt's `<close_gate>` carries the assembled-run output + IRL-required + mutation-gate + triage children; the `<approval_surface>` carries the **closure proof** (assembled-run observed + IRL result — NOT "tests green").
- **Grounded claims** (§4 rule 11). Every rung surface states what was run and what was NOT. "Tool executes" requires the assembled app (or assembled integration test) to have run the tool and observed the result — not a unit test that injects a `ProviderEvent::ToolUse` and asserts the emitted event (the trap that hid the M08.6.7 #32 + the built-in-tools-don't-execute gap).
- **Strict v1.8 two-commit TDD on every code stage.** Red commit standalone (`test(M08.7.X): ...`) → surface for red approval → impl commit (`feat(runtime): M08.7.X — ...`) whose body states `git diff <red>..<impl>` over the test paths is EMPTY. No amend dances; no Co-Authored-By.
- **Acceptance-assertion rule** (cluster-pattern.md §4). Assertions on user-observable behavior / observable side effects, never on internal events alone. "A `ToolInvoked` event is emitted" licenses "the event is emitted" — NOT "the tool ran." "The file exists on disk with the written content" / "the agent's reply is in all caps" / "no further provider turns are issued after the cap" are the grounded assertions.
- **Mutation gate** (cluster-pattern.md §5). `cargo-mutants` on every Rust cluster; a surviving viable mutant on the cluster's logic blocks close or is justified inline. **Windows-local mitigation:** `cargo mutants --in-place` bypasses the os-1314 symlink-privilege block (the rung-1 finding; the §5 ledger captures it as a Phase-0a addendum the M08.7.B opening docs commit applies). Stryker on TS is build-side-pending — not needed for M08.7a (Rust-only).
- **CI-parity is a hard rule** (CLAUDE.md §6). Local gate `✓` requires the exact CI command. Gotcha #56 Windows `--test-threads=1` documented per CLAUDE.md §6; any other divergence is cited inline + filed as a CI-fix follow-up.
- **Sandbox boundary** (Hard Rule 8 + §10). Built-in tools that touch the OS (`Bash`, and `Write`/`Read` *outside the workspace* — though the workspace scope is the agent's `file_access`, not the project dir) execute through `crates/runtime-sandbox/`, never directly in main. The sandbox has no exec path today; rung 1's scope lock holds for B–E. If a stage cannot satisfy its rung without the sandbox, STOP and surface — that's the signal to split a separate ADR-class rung, not to widen the current one.
- **No schema change** unless a rung proves one is needed. ADR-0026 (Stage A's archetype) and ADR-0027 (Stage C — skill injection) are the two ADRs already in scope; additional ADRs per CLAUDE.md §11 if a stage surfaces a trigger. ADRs file `Proposed` at their stage, flip `Accepted` in the M08.7 PR before merge.
- **The cluster owns its findings — zero-propagation** (cluster-pattern.md §2). Every finding from the in-cluster IRL has three legal dispositions: **fixed in this cluster** (observed resolved in the running system), **a new cluster** (opened now, scheduled in this phase or M08.6.7), or **explicit ADR scope-out** (a one-line waiver-ADR stating it is v1.0+ and *why*, with the observable v0.1 line documented). "Route to next milestone Stage A" is **banned**.

## Implementation Workflow

1. **Read first** — each stage's `<read_first>` block; Stage B+ reads the prior stage's retrospective `[END] Decisions` and applies them (CLAUDE.md §19 rule 1). The M08.7.A retrospective + IRL addendum are mandatory reading for Stage B (`docs/build-prompts/retrospectives/M08.7.A-retrospective.md`).
2. **Branch.** Develop on `claude/m08.7-execution-engine` — the branch already carries rung 1's 8 commits. Stage B opens with the carry-forward docs commit (the M08.7.A retro's 4 Decisions: cargo-mutants `--in-place` gotcha + cargo-test `--no-fail-fast` gotcha + the rungs-3–9 ground-at-red phase-doc edit already applied here + the `.gitignore` scratch-artifacts entries). The docs commit lands FIRST as a standalone `docs(M08.7): apply M08.7.A retro carry-forwards` commit, then the strict-TDD red+impl sequence for the stage's deliverable.
3. **Per-stage flow** (the cluster-gate cycle):
   - `ground_at_red` (resolve grounding sub-steps before writing tests; surface findings)
   - `write_failing_tests` → `red_phase_clippy` → `red_phase_commit` → **`surface_for_red_approval`** (HITL: maintainer relays orchestration adjudication; nothing proceeds without approval)
   - `implement` → `verify_gates` → `mutation_gate` (cargo-mutants `--in-place` on touched files) → `green_phase_commit`
   - **`assembled_run_irl`** (the close gate — assembled test runs + maintainer IRL-watches; the rung does not close on tests-green)
   - `fill_retrospective` → **`surface_for_final_approval`**
4. **Adjudicate the surface.** Each surface carries the cross-machine state (`git log --oneline main..HEAD` + retro listing) + the strict-TDD invariant proof (`git diff <red>..<impl>` over test paths = empty) + the closure proof (assembled-run output observed + IRL result) + gate results + the retro's three-axis table + an explicit "what was NOT exercised" statement (§4 rule 11). The orchestration session (me) adjudicates against the §X.5 prompt + this phase doc + the cluster-gate contract; you (the user, HITL) approve or request changes.
5. **Build CLI is cleared between stages.** Each new stage is a fresh session reading the §X.5 prompt + the per-stage `<read_first>` files. State lives in artifacts (this phase doc · the retros · the gap-analysis · git) — not session memory.
6. **v1.6 canonical gate ordering** (CLAUDE.md §6): `cargo fmt --all` → `cargo clippy --fix --allow-dirty -p <crate>` → `cargo clippy --workspace --all-targets -- -D warnings` → test/doc/audit/deny → `cargo llvm-cov clean --workspace` (gotcha #81) → the llvm-cov gates → frontend gates (n/a for M08.7a — Rust-only). CI-parity is a hard rule; cite any divergence inline with a gotcha reference.
7. **Stage V in a fresh CLI session** (the bias guard). V's `<read_first>` deliberately omits every `*-retrospective.md` / `M08.7-summary.md` / prior-gap-analysis but DOES consume this phase doc's `<construction_reachability_check>` blocks + the v1.9 5th `assembled_execution` pass (V runs the assembled rung — not just reads its tests). The verifier's 🔴 findings → scoped D.fix (max 2 iter) per ADR-0008; 🟡 → in-cluster fix-now / new cluster / explicit scope-out per cluster-pattern.md §2; 🟢 → `docs/tech-debt.md`.
8. **Closeout** runs `<simplify_pass>` (M08.7.A..HEAD diff) + the v1.8 `<coverage_policy_reconciliation>` (sync `docs/coverage-policy.md` §B/§C + CLAUDE.md §5/§6 + `codecov.yml` for any gate change). The M08.7 PR drafts at closeout. ADR-0026 + ADR-0027 flip `Proposed → Accepted` in the closeout commit, before push.
9. **ADRs.** Two expected (CLAUDE.md §11) — ADR-0026 (Stage A's archetype frozen) and ADR-0027 (Stage C skill injection). Additional ADRs per §11 if a stage surfaces a trigger.

## Stage C — Skill load (rung 3) — `LoadSkill` handler

### C.1 Problem Statement

No `LoadSkill` handler exists (grep-confirmed). `AgentEvent::SkillLoaded` is schema-defined (`crates/runtime-core/src/event.rs:279`) and `replay.rs:146` knows how to *project* it from persisted signals — but **no code emits it**. A skill in an agent's `allowed_skills` is never injected into the agent's context; the agent cannot follow skill instructions. This is the only true greenfield in M08.7a — rungs 2/4/5 are wires over existing primitives; rung 3 builds the handler + the context-injection mechanism from nothing.

The grounded inputs (verified): the M08.6 loader already resolves skill companions to bodies (a skill `{id, path}` reference resolves to the skill markdown body at load time — `framework_loader` + the M08.6 ADR-0022 single-resolution-boundary). So rung 3 does **not** re-resolve skill files — it reads the already-resolved skill body. The open design question (ADR-0027): **where in the assembled prompt does the skill body land, and how does it persist across turns?** `AgentConfig` (`providers/mod.rs:288`) carries the per-turn message history + `system_prompt`; the §0b system-prompt-assembly job (flagged load-bearing by proposals `0001-interaction-layer-roadmap.md` + `harness-review-takeaways.md` Item 2) is the natural home — a loaded skill is an instruction set that should persist for subsequent turns, which means it lands in a stable section of the assembled context, not a one-shot tool result.

**Scope lock:** the skill body injection is the deliverable; the broader three-tier system-prompt-assembly refactor (stable/context/volatile per the Hermes blueprint) is NOT — rung 3 adopts the *minimal* injection that makes a skill change behavior, and ADR-0027 records the forward path. No schema change (`SkillLoaded` already exists). NO Bash/sandbox.

### C.3 Detailed Changes

#### C.3.0 Ground at entry (required, before red)

Read the skill model in `crates/runtime-core` (`generated/skill.rs` — the `skill.md` frontmatter schema, "Skills are context-loaded instruction sets, NOT callables. They are loaded via the runtime-injected LoadSkill tool") + how the M08.6 loader resolves skill companions to bodies (`framework_loader` + the resolved `Framework`'s skills) + `AgentConfig`'s shape (`providers/mod.rs:288` — `system_prompt: Option<String>` + `messages`). State: (a) how a resolved skill body is reached from the run loop (the loaded framework is in scope at `run_test_session_with`); (b) the load policy — does the agent invoke `LoadSkill` as a tool (the schema says "loaded via the runtime-injected LoadSkill tool"), or does the framework's `allowed_skills` auto-load? The schema text says tool-invoked; confirm against the spec §0b + §4b. This finding shapes ADR-0027.

#### C.3.1 The `LoadSkill` handler (`load_skill.rs`)

A pure fn `load_skill(skill_name, agent, resolved_skills) -> Result<LoadedSkill, LoadSkillError>` that: checks `skill_name` is in the agent's `allowed_skills` (the capability gate — analogous to the executor's `file_access` check; a skill not in `allowed_skills` is denied, mirroring rung 1's scope denial); looks up the resolved skill body from the loaded framework's resolved skills (no re-resolution — the M08.6 loader already produced the body); returns `LoadedSkill { name, body }` for the `SkillLoaded` event + the context injection. `tempfile`/in-memory testable (the resolved skills are in-memory `Framework` data).

#### C.3.2 The `drive_stream` `LoadSkill` branch + context injection (ADR-0027)

Insert a `LoadSkill` branch in `drive_stream` analogous to the rung-1 built-in branch. On a `LoadSkill` `ToolUse`: call `load_skill`, emit `SkillLoaded { agent_id, skill_name }` (the event at `event.rs:279`), and **inject the skill body into the agent's context for subsequent turns** — the ADR-0027 decision point. The grounded minimal injection: append the skill body to the agent's `AgentConfig.system_prompt` (or a dedicated loaded-skills section of it) so every subsequent turn's assembled prompt carries it — a loaded skill is a *persistent instruction set*, not a one-shot tool result, so it must outlive the turn it was loaded in. ADR-0027 records: the injection target (system_prompt section vs a synthetic message); persistence (across all subsequent turns of this agent); composition (a second `LoadSkill` appends, does not replace — multiple skills compose); and the forward path to the three-tier assembly (stable/context/volatile) the harness-review proposals flagged. **The grounded acceptance is behavioral** — the skill content reaching the model is proven by the agent's reply changing, not by the `SkillLoaded` event.

#### C.3.3 ADR-0027 sketch (filed at this stage)

- **Title:** Skill-into-context injection model.
- **Status:** Proposed (→ Accepted in the M08.7 PR).
- **Context:** `LoadSkill` must make a skill's instructions reach the model for subsequent turns. Skills are context-loaded instruction sets (§0b), not callables. The §0b system-prompt-assembly job is partial (harness-review Item 2); rung 3 needs the minimal injection that makes a skill change behavior without committing to the full three-tier refactor.
- **Decision:** the skill body is appended to a dedicated loaded-skills section of the agent's assembled context (system_prompt-adjacent), persists for all subsequent turns of that agent, and composes additively (a second `LoadSkill` appends). The grounded close is behavioral (the agent's reply reflects the skill).
- **Consequences:** forward-compatible with the three-tier (stable/context/volatile) assembly — loaded skills are stable-tier content; when that refactor lands (v1.0 / a later rung), the loaded-skills section moves into the stable tier wholesale. No schema change. Capability-gated by `allowed_skills`.
- **Alternatives considered:** one-shot tool-result injection (rejected — a skill must persist across turns, not vanish after the turn that loaded it); framework-level auto-load of all `allowed_skills` at session start (rejected for v0.1 — the schema says tool-invoked; auto-load is a forward option ADR-0027 notes but does not adopt).

## Stage D — Gap detection (rung 4) — `request_capability` wired into the run loop

### D.1 Problem Statement

`handle_request_capability` (`crates/runtime-main/src/sdk/request_capability.rs:88`) exists, is fully implemented (314 lines), and is unit-tested (`:201`, `:216`, `:239`, `:291`) — but has **no production caller** (grep-confirmed; only its own test fns call it). A model `request_capability` `ToolUse` today falls through `drive_stream` to `pipeline.next_event`, which treats it as an ordinary (unknown) tool — emits `ToolInvoked` for the name and stops. **The project's signature gap-detection flow never fires.**

The grounded shape (verified from the handler): `handle_request_capability(invocation, emitter) -> Result<RequestCapabilityResult, _>` emits the appropriate `*Missing` event (`ToolMissing` / `SkillMissing` / `McpMissing` / `AgentMissing`) with `requested_via: GapSourceRef::RequestCapability` + `severity: GapSeverityRef::Requested`, and returns `RequestCapabilityResult::Pending`. The module docstring (`:1-11`) is explicit about the wire rung 4 builds: "The SDK does NOT route this to an LLM tool call: it intercepts the invocation inline, emits the appropriate `*_missing` event... and returns `Pending` to the caller. The HITL seam (`on_gap` trigger, wired in M04 Stage E)... routes the gap event to the user; **the SDK turn loop awaits resolution before resuming**." So rung 4 = (1) the `drive_stream` interception branch for a `request_capability` `ToolUse` (analogous to the rung-1 built-in branch — it routes BEFORE `pipeline.next_event` treats it as unknown); (2) the **suspend-at-the-awaiter** wire (the turn loop awaits the gap resolution rather than continuing or crashing).

**Scope lock:** the v0.1 outcome is `Pending` → suspend → recoverable (the gap flow per spec §4). NOT the full grant/install/decline resolution UI (that's the HITL surface). Rung 4 proves the session **suspends cleanly and is recoverable** — the observable gap behavior. **Resolve-and-resume (the resume half) is pulled forward into v0.1 as a scheduled gap-resume rung per ADR-0029 — NOT M09-deferred; rung 4 is the suspend half.** No schema change (the `*Missing` events + `requested_via` + `severity` all exist). ## Stage E — Budget at the run loop (rung 5)

### E.1 Problem Statement

The budget primitive is **built and unit-tested, not painted** (the grounded correction to the original M08.7 doc's "ground at entry — budget may be painted" framing — verified at authoring time). `crates/runtime-main/src/budget/` ships `cost.rs` + `enforcer.rs` + `hook.rs` + `mod.rs`. `BudgetEnforcer::record_spend(...) -> Vec<ThresholdAction>` returns the threshold actions that just crossed (`Warn` / `Downshift` / `Suspend` / `HardStop`, each with scope + spent + cap + percent), idempotent per (cap, percent), tightest-cap-wins (`enforcer.rs:1-124`). The downshift ladder (`opus→sonnet→haiku`) is real (`hook.rs:6-11`). Every provider call already produces a `CostBreakdown` (`providers/anthropic.rs:147,292`). The enforcer's own module docstring states the gap precisely: "The caller (**SDK turn loop**) translates actions into events + side effects" — and **that caller does not exist on the live run loop.** Per-turn spend is produced but never fed to `record_spend`; the thresholds never fire on a live session.

So rung 5 is the **run-loop wire**, not a build: feed each turn's `CostBreakdown` (cost in USD via `estimate_cost`) into a session-scoped `BudgetEnforcer::record_spend` at the per-turn boundary in `drive_stream`'s loop, and dispatch the returned `ThresholdAction`s — `Warn` → emit `budget_warn` + notifier; `Downshift` → invoke the downshift hook (swap the model via the existing `hook.rs` ladder) + emit `budget_downshift`; `Suspend` → the HITL flow; `HardStop` → stop the session (drone `StopProcess`) + emit `budget_exceeded`. The grounded acceptance for the v0.1 close gate: a session with a tiny `session_usd_cap` exceeds the cap → the session **stops with a budget-stop reason → no further provider turns are issued**, observed.

**Scope lock:** wire the existing enforcer + ladder + events. Do NOT rebuild the budget model. The full four-threshold dispatch is the deliverable, but the **load-bearing IRL gate is `HardStop`** (the safety primitive — spend control); `Warn`/`Downshift`/`Suspend` are dispatched + unit-asserted, and the assembled+IRL gate centers on the hard-stop (the observable "the run halts and issues no more turns"). No schema change (the `budget_*` events + `BudgetScope` + `ThresholdAction` all exist). NO Bash/sandbox. **M08.6.7 owns the budget-cap *persistence* bug** (M08.6-IRL #22 — cap doesn't survive restart); rung 5 is the *enforcement-at-the-run-loop* wire, a different surface — do not conflate (zero-propagation: #22 is an M08.6.7 cluster, not rung 5's).

#### E.3.1 The per-turn `record_spend` wire

In `drive_stream`'s multi-turn loop, after each turn's `provider.stream` completes and the `CostBreakdown` is known, compute the turn's USD cost (`provider.estimate_cost(&breakdown, model)`) and call `session_budget.record_spend(cost)`. The returned `Vec<ThresholdAction>` is dispatched in firing order (the enforcer returns lowest-percent-first for deterministic dispatch).

#### E.3.2 The ThresholdAction dispatch

- `Warn { scope, spent, cap, percent }` → emit `budget_warn` event + a notifier dispatch (the `NotifierDispatched` path).
- `Downshift { … }` → invoke the downshift hook (`hook.rs` — swap opus→sonnet→haiku via the ladder), update the session's model for subsequent turns, emit `budget_downshift`.
- `Suspend { … }` → the `on_budget_threshold` HITL flow (the same awaiter family as rung 4's gap suspend — the Tester-vs-live-session triage applies; on the Tester path it may be suspend-and-record).
- `HardStop { … }` → stop the session (drone `StopProcess` on the live path; a loop-break with a budget-stop `TestOutcome` reason on the Tester path) + emit `budget_exceeded`. **No further provider turns are issued** — the load-bearing observable.

#### E.3.3 The assembled close — tiny cap stops the run

`run_test_session_with` over a framework with a tiny `session_usd_cap` (e.g. `$0.0001` — below one turn's cost); the `ScriptedToolStub` is scripted to keep producing turns (or the single turn's cost exceeds the cap). Assert: the run stops with a budget-stop reason in the `TestOutcome` AND no further provider turns are issued after the cap is crossed (count the stub's `stream` invocations — the rung-1 `seen` Arc<Mutex<Vec<AgentConfig>>> pattern records each turn's config; assert the count stops growing after the hard-stop). This is the grounded "the run halts" assertion — a counted side effect, not the `budget_exceeded` event alone.
