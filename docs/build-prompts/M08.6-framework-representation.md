## Background and Design Decision

The post-M08 IRL pass (`docs/M08-irl-findings.md`) and the follow-on
review established that **the M08 Builder cannot load the archetype
frameworks.** Loading `examples/aria/` into the Builder Canvas produces
a pile of unconnected nodes — no edges, no workflow.

ADR-0022 settles the architecture. There are two un-reconciled
framework-on-disk representations:

- **The archetype / canonical model** — `examples/aria/` and
  `examples/ralph/`: a `framework.json` whose agents / tools / skills
  are `{id, path}` references into subdirectories (`agents/orchestrator.md`,
  `tools/git_checkpoint.md`, `skills/planning.md`). Each `.md` is YAML
  frontmatter (the `agent.v1.json` / tool / skill shape) plus a markdown
  body. This is the universal agent-definition idiom and what makes
  artifacts shareable.
- **The Builder model** — `crates/runtime-main/src/builder/persist.rs`
  (`load_framework` / `save_framework`) + `src/lib/builderStore.ts`:
  a `framework.json` with **inline** agents plus flat `<name>.agent.md`
  companion files at the directory top level.

`load_framework` parses `framework.json` and scans only the top
directory for `*.{agent,skill,tool}.md`; against `examples/aria/` it
finds zero companions and never resolves the `{id, path}` references.
`projectCanvasEdges` (`builderStore.ts`) then skips every non-inline
agent, so no edges are projected. M08 shipped the Builder for the
inline model only; the archetype frameworks were never in the test loop.

**ADR-0022's decision** (Accepted at Stage B): the modular multi-file
markdown-frontmatter representation is **canonical on disk**; references
resolve at **one loader boundary** (`load_framework`) into a fully
reference-resolved in-memory `Framework`; **save re-splits** back to the
modular form. Everything downstream — the canvas projection, the
Tester, the runtime — already speaks the inline form, so localizing
resolution to the loader is one seam, not a stack-wide change.

**Why this is a milestone, and why before M09.** M09 wires the LLM
generators ("Generate Tool / Skill / Agent") into the Builder — they
write into `builderStore.framework` (ADR-0020's write target). M09 can
only build on a Builder that can **hold, render, and persist a real
framework**. If the Builder cannot load / render / save the archetype,
M09 has no working base. M08.6 makes the base work; M09 builds on it.

**The acceptance bar (the maintainer's criterion).** Load `examples/aria/`
into the Builder and see ARIA's real workflow — the eight agents wired
by their `spawns` / `allowed_*` relationships, laid out as a graph, with
the Palette surfacing ARIA's artifacts as reusable items; and a
load → save → load round-trip that is byte-stable. "Build ARIA from
scratch in the tool" is an M09-and-later criterion (it needs the
generators to author agent bodies / system prompts — M08.6 ships no
prompt-authoring surface); the M08.6 bar is **load + render + persist**.

### What M08.6 produces

1. **Stage A — intake + carry-forward absorption + loader groundwork.**
   Absorb the M08.5 carry-forward (the routed 🟡 #4/#5/#6 + the
   framework-representation gap); triage 🟡 #5 / #6 (M08.6-fix vs M09);
   author the `<construction_reachability_check>` for the loader
   resolution wiring (the M08 Stage-A→F1 groundwork pattern).
2. **Stage B — the loader resolution (ADR-0022 core).**
   `runtime_main::builder::load_framework` walks the framework
   directory, reads each `{id, path}`-referenced `.md`, parses its YAML
   frontmatter into an inline `Agent` / tool / skill, and returns a
   `LoadedFramework` whose `framework` is fully reference-resolved.
   ADR-0022 flips `Proposed → Accepted`.
3. **Stage C — save re-split.** `save_framework` writes the modular
   form back — `framework.json` with `{id, path}` references + the
   companion `.md` files in their subdirectories — so a load → edit →
   save round-trip preserves the modular structure.
4. **Stage D — the canvas renders the resolved framework + auto-layout
   on load.** Once Stage B resolves, `projectCanvasEdges` already
   projects edges for the (now-inline) agents; D wires
   `src/lib/layout.ts`'s dagre `layoutGraph` into the load path so a
   loaded framework lays out as a graph instead of stacking at {0,0}
   (closes 🟡 #4 + the "disconnected pile" observation).
5. **Stage E — the Palette populated from the loaded framework.**
   `src/components/builder/Palette.tsx` surfaces a loaded framework's
   agents / tools / skills as reusable Palette items by type (closes
   the "defined agents are not shareable" observation).
6. **Stage V — the fresh-context four-pass verifier.**
7. **Stage F — closeout.** The M08.6 parent-milestone summary, the
   immutable M08.6 `docs/gap-analysis.md` entry, the `<simplify_pass>`,
   the `<coverage_policy_reconciliation>`, the PR description draft.

### What's not in scope

- **Building a framework from scratch in the tool to equal ARIA.**
  M08.6 ships no agent-body / system-prompt authoring surface — that
  content is what M09's generators produce. The M08.6 bar is load +
  render + persist of an existing framework, not authoring one.
- **A multi-agent execution engine.** M08.6 makes the canvas *render* a
  loaded framework's agent topology and makes the loader *resolve* it;
  it does not build per-agent orchestration. The runtime's single-agent
  `run_agent` loop is unchanged. (Under ADR-0022 the loader resolves to
  inline, so the Tester's existing `spawn_framework_subagents` will now
  see a path-ref framework's sub-agents — a free improvement — but
  M08.6 builds no new execution path.)
- **Applying an agent's `system_prompt_template` / `.md` body at run
  time.** Stage B's loader *captures* the agent `.md` body in the
  resolved model (per `agent.v1.json`); *applying* it at run time is the
  M09 carry-forward deferred from M08.5 (decision 1). M08.6 captures;
  M09 applies.
- **🟡 #6 (the UI does not disclosure-gate by tier).** Carried from
  M08.5; Stage A triages it — it is not framework-representation work
  and routes to M09 unless Stage A's triage finds it trivially
  co-located with a D/E surface. Default: M09.
- **A registry / "Share It" module.** ADR-0022's modular artifacts make
  these natural v1.0 features; M08.6 builds neither.
- **`schemas/framework.v1.json` changes.** The `agents[]`
  `oneOf(inline | {id,path})` already permits both forms — ADR-0022
  decides which is canonical and where resolution happens, not the
  schema. No schema version bump, no §14 schema ADR.

### Why five work stages + V + closeout

The milestone has one intake stage, two backend stages (load resolve,
save re-split — distinct concerns, distinct red→green), two renderer
stages (canvas render + layout, Palette populate — a different language
and layer), a fresh-context verifier, and a closeout:

- **A (intake) stands alone** — it absorbs the M08.5 carry-forward,
  triages the routed 🟡, and authors the loader-resolution
  `<construction_reachability_check>` groundwork (the M08 A→F1 pattern).
  It ships minimal code; its product is the staged intake + the
  construction graph the later stages discharge.
- **B (loader resolution) is the architectural core** and cannot be
  split further — it is one coherent change to `load_framework`
  (directory walk + frontmatter parse + `{id,path}`→inline). ADR-0022
  flips here. B's red→green is "load `examples/aria/` → a fully
  resolved `Framework`."
- **C (save re-split) is a distinct concern** — the write path. Its
  red→green is "save → load → save byte-stable on a modular framework."
  It lands after B because the round-trip test consumes B's loader.
- **D (canvas render + auto-layout) is a renderer stage** — once B
  resolves, the existing `projectCanvasEdges` paints the edges; D's new
  work is wiring `layoutGraph` into the load path. Splitting D from B/C
  keeps the Rust red→green and the renderer red→green clean.
- **E (Palette populate) is a renderer stage** — a separate surface
  (`Palette.tsx`) from D's canvas, its own red→green.
- **V (verifier)** — the milestone-class fresh-context contract-fidelity
  check (ADR-0008). M08.6's central V duty: the loader genuinely
  resolves the **archetype** (`examples/aria/`), not only inline test
  fixtures — the M08-Builder escape was that the archetype was never in
  the test loop.
- **F (closeout)** — the parent-milestone summary + the immutable
  gap-analysis entry + the simplify_pass + the coverage reconciliation
  + the PR draft.

### Key constraints

- **The archetype is the acceptance fixture (load-bearing).** Every
  load / render / round-trip test MUST exercise the **real
  `examples/aria/`** framework (and, where relevant, `examples/ralph/`
  — which adds cross-framework `../aria/...` references), not only an
  inline or hand-built fixture. The M08 Builder shipped green because
  the archetype was never tested; M08.6 closes that by making ARIA the
  fixture. Stage V's central duty is to confirm this.
- **Real-app regression mandate (CLAUDE.md §6 v1.8; ADR-0021).** D and E
  are renderer behaviors — their regression tests are `tests/e2e-tauri/`
  tests driving the real Tauri app through the M08.5-revived harness
  (load ARIA → the canvas shows wired, laid-out nodes; load ARIA → the
  Palette shows ARIA's artifacts). B and C are backend — their
  assembled regressions are `crates/runtime-main/tests/` integration
  tests against `examples/aria/` on disk.
- **Strict v1.8 two-commit TDD on B + C + D + E.** Failing tests
  committed standalone (`test(M08.6.X): …`), surfaced for red-phase
  approval before implementation; the impl commit must not modify test
  files; the impl commit body states `git diff <red>..<impl> --
  '<tests>'` is EMPTY. Stage A is an intake stage (groundwork + triage
  + minimal code); it runs the standard gate flow but is not a strict
  two-commit code stage (the M08.A intake precedent).
- **ADR-0022 is the surfaced plan; resolution is one seam.** B
  implements ADR-0022 — reference resolution lives only in
  `load_framework`. No path-ref logic is added to the canvas, the
  Tester, or the runtime — they consume the resolved model. A build
  that spreads resolution across components contradicts ADR-0022.
- **No new third-party dependency.** Stage B parses YAML frontmatter
  with `serde_yaml`, which is **already a workspace dependency**
  (`Cargo.toml` workspace; consumed by `runtime-core`). Adding
  `serde_yaml = { workspace = true }` to `crates/runtime-main/Cargo.toml`
  is wiring an existing workspace dep, not a new external dependency —
  no `cargo deny` concern, no §11 dependency ADR. The frontmatter
  splitter (the `---\n…\n---` block) is a few lines of `runtime-main`
  code; grep first for any existing splitter (gotcha #41).
- **No schema change.** `schemas/framework.v1.json` / `agent.v1.json` /
  the generated types are unchanged; M08.6 resolves between two forms
  the schema already permits.
- **Path resolution is deliberate.** The loader resolves relative
  `{id,path}` values — including `../` cross-framework references
  (Ralph's `../aria/tools/…`) — against the framework directory. It
  reads only the files the loaded `framework.json` references; no glob,
  no symlink-escape. A malformed / missing referenced file is a load
  error the Builder surfaces (the existing gap-tolerant posture for a
  *partially-built* framework is preserved; a *broken reference* is an
  error).
- **No Co-Authored-By trailers.** Commit messages end with the
  session-URL footer only.
- **Windows is the v0.1 target; the real-app harness runs Windows +
  Linux** (macOS on Playwright — ADR-0021 / gotcha #23).

  authored `Proposed` alongside this phase doc. Stage B flips it
  `Proposed → Accepted` in its impl commit (the M06.5.A.fix / ADR-0012

  (CLAUDE.md §6). The `e2e-tauri-driver` real-app job (revived + gating
  since M08.5) is part of the suite. CI-parity is a hard rule.
- **Stage V in a fresh CLI session** — the clear-and-paste bias guard.
  V's `<read_first>` omits the per-stage retrospectives, the M08.6
  summary, and the gap-analysis, but DOES consume the phase doc's
  `<construction_reachability_check>` blocks.
- **Do not push between stages.** Surface each stage; the maintainer
  approves; the agent pushes on approval (CLAUDE.md §8). Cross-machine
  state (`git log --oneline main..HEAD` + retro listing) is a top-level
  surface item.

## Stage B — the loader: resolve `{id,path}` → inline (ADR-0022)

### B.1 Problem Statement

`load_framework` does not resolve `{id,path}` references; loading the
archetype yields an unusable framework.

Verified (citations on `main` @ post-M08.5):

- `crates/runtime-main/src/builder/persist.rs:93-105` — `load_framework`
  reads `dir/framework.json`, `serde_json::from_str` into `Framework`,
  and calls `read_companions(dir)`. The `{id,path}` agent entries stay
  raw; nothing reads the referenced files.
- `persist.rs:112-133` — `read_companions` scans only the top directory
  (`std::fs::read_dir(dir)`, non-recursive) for `*.skill.md` /
  `*.tool.md` / `*.agent.md`. `examples/aria/`'s files are
  `agents/orchestrator.md` etc. — subdirectories, plain `.md` — so
  `read_companions` finds **zero**. And `Inspector.tsx`'s `onLoad`
  consumes only `loaded.framework`, discarding `companions` entirely.
- `examples/aria/framework.json` — eight agents, all
  `{ "id": …, "path": "agents/…md" }`; four tools and five skills
  likewise path-referenced. `examples/aria/agents/orchestrator.md` is
  YAML frontmatter (`id` / `role` / `model` / `allowed_tools` / … —
  the `agent.v1.json` shape) between `---` delimiters, then a markdown
  body. `examples/ralph/` is the same form and adds cross-framework
  refs (`../aria/tools/…`).

Per ADR-0022 (Accepted at this stage): `load_framework` becomes the
single resolution boundary. It walks the framework directory, reads
each `{id,path}`-referenced `.md`, parses its YAML frontmatter into an
inline `Agent` / tool / skill, and returns a `LoadedFramework` whose
`framework` is fully reference-resolved — every `agents[]` entry inline.

Why this is the milestone core: every downstream consumer — the canvas
projection, the Tester, `spawn_framework_subagents` — already speaks
the inline form. Resolving at the loader is the one change that makes
the whole stack handle the archetype.

### B.3 Detailed Changes

#### B.3.1 The frontmatter splitter + the YAML parse

A `.md` artifact file is a YAML frontmatter block delimited by `---`
lines, then a markdown body. Add a small splitter to `persist.rs` —
grep first for any existing one (gotcha #41); if none, it is a few
lines: take the text between the first `---\n` and the next `\n---`,
the remainder is the body. The frontmatter parses with `serde_yaml`
into the generated inline type (`runtime_core::generated::agent::Agent`
etc.). Add `serde_yaml = { workspace = true }` to
`crates/runtime-main/Cargo.toml` — it is already a workspace dependency
(consumed by `runtime-core`), so this is wiring, not a new dependency.

The body is **captured** on the resolved artifact where the type
carries it (the agent `.md` body is the system-prompt source per
`agent.v1.json`); *applying* it at run time is the M09 carry-forward.
B captures, M09 applies.

#### B.3.2 Directory walk + reference resolution

`load_framework(dir)` resolves the framework:

1. Read + parse `dir/framework.json` into `Framework` (as today).
2. For each `agents[]` entry that is the `{id,path}` reference form
   (the build reads `crates/runtime-core/src/generated/framework.rs`
   for the exact `oneOf` enum variant names — gotcha #41), resolve
   `dir.join(path)` (relative, incl. `../`; canonicalize and confirm
   the read is a deliberate single file — no glob), read the `.md`,
   split + parse the frontmatter into an inline `Agent`, and replace
   the entry with the inline variant.
3. Repeat for `tools[]` and `skills[]` path-referenced entries
   (`tool.v1.json` / skill frontmatter → the generated inline types).
4. Return `LoadedFramework { framework, companions }` with `framework`
   fully reference-resolved. `companions` continues to carry the raw
   `.md` bodies (Stage C's re-split consumes them; the round-trip is
   byte-stable).

The resolution is the **only** new path-ref logic in the codebase
(ADR-0022). The canvas projection, the Tester, and
`spawn_framework_subagents` are untouched — they receive the resolved
inline `Framework`.

#### B.3.3 Gap-tolerance vs. broken references

The existing `load_framework` is gap-tolerant: a *partially-built*
framework (missing pieces) reloads fine and the Builder surfaces gaps
as validation errors. B preserves that for partial frameworks. A
**broken reference** — a `{id,path}` whose file is missing or whose
frontmatter does not parse — is different: it is a `BuilderError`
(B.3.4) the Builder surfaces, not a silent drop. (A reference that
points at a file is a structural promise; a missing target is an
error, distinct from an unfilled optional field.)

#### B.3.4 `BuilderError` — the resolution error variant

`error.rs` gains a variant for an unresolvable reference (missing file
/ unreadable / frontmatter parse failure), carrying the offending
`{id,path}` for a clear Builder-surfaced message.

Proposed → Accepted` in the B impl commit. No further ADR: no schema
change, no new external dependency (`serde_yaml` is an existing
workspace dep), no IPC change — the `load_framework` / `LoadedFramework`
wire shape is unchanged (only `framework` is now resolved).

## Stage D — the canvas renders the resolved framework + auto-layout on load

### D.1 Problem Statement

Even with Stage B resolving the framework, a loaded framework's nodes
all stack at {0,0} — there is no layout, so the graph reads as a pile.

Verified (citations on `main` @ post-M08.5):

- `src/lib/builderStore.ts:295-318` — `projectCanvasEdges` derives
  edges from each **inline** agent's `allowed_skills` / `allowed_tools`
  / `spawns`. Once Stage B's loader resolves `{id,path}` agents to
  inline, this projection **already** paints the edges — no change
  needed here.
- `src/lib/builderStore.ts:210-211` — `projectCanvasNodes` positions
  each node at `nodePositions[id] ?? { x: 0, y: 0 }`. A loaded
  framework has no `nodePositions` entries (only a drag-drop fills

- `src/components/builder/Inspector.tsx:76-89` — `onLoad` calls
  `loadFramework(dir)` → `replaceFramework(loaded.framework)`; nothing
  computes a layout.
- `src/lib/layout.ts` — `export function layoutGraph(nodes, edges):
  GraphNode[]` already exists: a pure dagre top-down layout (the live
  `GraphCanvas` uses it). D reuses it.

D wires `layoutGraph` into the load path so a loaded framework lays out
as a graph. Once B resolves and D lays out, loading ARIA shows the
real workflow — eight agents wired by `spawns` / `allowed_*`, ranked
top-down. This closes 🟡 #4 and the "disconnected pile" observation.

### D.3 Detailed Changes

On a framework **load** (distinct from an interactive drag-drop, which
already seeds `nodePositions`), the Builder computes a layout:

1. Project the nodes + edges from the resolved `framework`
   (`projectCanvasNodes` / `projectCanvasEdges` — unchanged; B made
   them see inline agents).
2. Run `layoutGraph(nodes, edges)` (`src/lib/layout.ts` — the existing
   dagre top-down layout) to compute positions.
3. Seed `nodePositions` from the layout result, so the canvas
   projection renders the laid-out graph. (User drag-repositioning
   after load still works — it overwrites a `nodePositions` entry, the
   existing `moveNode` path.)

The build chooses the seam — `replaceFramework` computing the layout
when it replaces the document, or the `Inspector.onLoad` path doing it
after `loadFramework` returns — but it is **one** seam, and it fires on
a *load*, not on every framework mutation (an interactive edit must not
re-layout and discard the user's manual positions). ADR-0020's
"`nodePositions` is editor-local view state" holds — auto-layout seeds
it; it is not framework data.

Edges need no new code: `projectCanvasEdges` already paints them for
inline agents, and Stage B's resolution makes a loaded framework's
agents inline.

## Stage E — the Palette populated from the loaded framework

### E.1 Problem Statement

A loaded framework's own agents / tools / skills are not surfaced as
reusable Palette items — so a framework's defined artifacts cannot be
re-used on the canvas.

Verified (citations on `main` @ post-M08.5):

- `src/components/builder/Palette.tsx:62-85` — `paletteItemsForTab`
  builds each tab's items from the runtime built-ins (`BUILTIN_TOOLS`)
  + `listInstalledArtifacts()` (the `skills.lock` reader). It has **no
  awareness of the currently-loaded framework** — a framework's own
  agents / tools / skills (the `builderStore.framework` document) never
  appear in the Palette.

The IRL observation: "if there are defined agents they do not show up
in the tools as sharable." Per ADR-0022, a loaded framework's resolved
artifacts are first-class — they should populate the Palette by type so
they are reusable drag sources.

E adds the loaded framework as a Palette item source: the
`builderStore.framework`'s agents / tools / skills appear in the
Agents / Tools / Skills tabs, alongside the built-ins and the installed
artifacts, each a drag source.

### E.3 Detailed Changes

`paletteItemsForTab(tab, installed, framework)` gains the loaded
framework as a third source. For the Agents / Tools / Skills tabs, the
items are the union of:

1. the runtime built-ins (Tools only — unchanged);
2. `listInstalledArtifacts()` results (unchanged);
3. **the loaded framework's resolved artifacts** — `framework.agents`
   / `framework.tools` / `framework.skills` (resolved inline by Stage
   B), each a drag source carrying the same `application/x-builder-node`
   payload an installed artifact does.

De-duplicate by `(kind, ref)` so an artifact that is both installed and
in the loaded framework appears once. The loaded-framework items are
distinguishable (a source label / data attribute) so the user can see
which artifacts come from the open framework — but they drag
identically.

The Palette reads `builderStore.framework` via the store (the same
`useBuilderStore` the canvas uses); a framework load (Stage D's load
path) re-renders the Palette with the loaded artifacts.
