## Background and Disposition

`docs/M06-irl-findings.md` (the M07 Stage A gating input, produced by
executing `docs/M06-irl-test-plan.md` against the running app after the
M06 merge) recorded **2 🔴 · 4 🟡 · 5 🟢**. Per that document's
disposition (§"Disposition / routing"): *"M07.A does not start until
🔴-1 and 🔴-2 are fixed and their cards re-tested green."* This cycle is
that fix-and-re-test, scoped strictly to the two 🔴.

Both 🔴 are the same class the IRL pass exists to catch (gotcha #66 —
tests-pass-but-contract-fails): M06 unit + Stage-V suites are green, yet
the **assembled running app** fails, because Stage-V verified components
in isolation (the MCP `Registry`, the drone, the SDK each tested
standalone) while the failures live in the `src-tauri` composition layer
that wires them together. Every regression test this cycle lands must
therefore exercise the **assembled running app path**, not the isolated
component that already passes — that is the precise structural gap
Stage-V could not see, and it is the load-bearing mandate of this cycle.

Verified root causes (grounded on `main` @ `71107c7`, citations in each
stage's Problem Statement — authored against read source, not memory):

- **🔴-1** — `src-tauri/src/main.rs:262` (`open_mcp_client`) opens the
  MCP registry at `app_local_data_dir().join("mcp.sqlite")` while
  `src-tauri/src/main.rs:298-301` (`resolve_db_path`) gives the drone
  `…join("session.sqlite")`. The `Registry` is correctly path-agnostic;
  the **composition layer** never decided which file is canonical. Fixed
  by ADR-0012 (single source-of-truth session DB; the registry shares
  the drone session DB via a second WAL connection).
- **🔴-2** — `crates/runtime-main/src/sdk/agent_sdk.rs`: every
  `AgentEvent` flows through `self.emit()` (`agent_sdk.rs:251,313,
  322-323`); `emit` (`agent_sdk.rs:521-526`) **only** does
  `self.event_tx.send(event)` — it never calls
  `self.drone_client.write_signal(...)`. The run loop's sole drone
  interaction is the one-shot `SnapshotNow.send` at `agent_sdk.rs:262`
  (hence `snapshots` land; `heartbeats` land via the drone's own task;
  `signals = 0` + `token_usage = 0` because the per-event signal write
  is structurally absent). Not a no-op drone (the same `drone_client`
  successfully sends `SnapshotNow`), not a path divergence — a missing
  emission. `DroneClient::write_signal` (`drone_ipc/client.rs:128`) and
  the drone-side `handle_write_signal` (`command_handler.rs:127`) both
  exist and are unit-tested in isolation; the assembled call site is
  the gap.

### Why three stages

🔴-1 (composition path resolution) and 🔴-2 (SDK run-loop signal
emission) are distinct code paths. Strict v1.7 two-commit TDD requires a
clean per-fix red→green diff, so each 🔴 gets its own stage (A.fix,
B.fix) with its own `git diff <red>..<impl> -- '**/tests/**'`-EMPTY
audit surface. Stage C.fix exists because the findings doc gates M07.A
on the cards being *re-tested green in the assembled app* — that
re-verification + the findings/CHANGELOG/carry-forward reconciliation is
real deliverable work that, bundled into B.fix, would muddy B.fix's
red→green diff. No Stage V (this is a D.fix-class remediation, not a
milestone; per ADR-0008 the D.fix loop is itself the response to V
findings — it does not recurse a V).

- **Assembled-app regression mandate (load-bearing).** Each fix's
  regression test MUST exercise the assembled running-app path (the

  `run_smoke_session_with` with a stub provider against a real drone

- **No scope creep.** Touch only what 🔴-1 / 🔴-2 require. No
  refactors, no 🟡 fixes, no protocol edits. A bug fix does not need
  surrounding cleanup (CLAUDE.md §"Doing tasks").
- **No IPC-protocol change.** 🔴-2 uses the *existing*

- **No Co-Authored-By trailers.** Project convention + model-identity
  rule (M06.B Decision, recurring). Commit messages end with the session
  URL footer only.
- **Windows is the v0.1 target; CI runs all three OSes.** The drone

  `crates/runtime-main/tests/recovery_lifecycle.rs` pattern
  (`ensure_drone_built` / `spawn_drone` / `poll_until`), which already

- **Capability adherence (CLAUDE.md §10).** Regression tests use a stub

## Stage A.fix — 🔴-1 single source-of-truth session DB path

### A.1 Problem Statement

The MCP server registry write resolves to a stray DB; an added server is
audited yet invisible to the runtime.

Verified (citations on `main` @ `71107c7`):

- `src-tauri/src/main.rs:262` — `open_mcp_client` builds
  `let path = dir.join("mcp.sqlite");` and calls `Registry::open(&path)`.
- `src-tauri/src/main.rs:298-301` — `resolve_db_path` builds
  `Ok(dir.join("session.sqlite"))` and that path is given to
  `DroneLifecycle::spawn` (the live drone DB).
- `crates/runtime-mcp/src/client/registry.rs` — `Registry::open(path)`
  delegates to `runtime_drone::db::init(path)`, which runs **all**
  migrations (incl. `002_mcp_servers.sql`; `crates/runtime-drone/src/db.rs:71-75`)
  on whatever path it is handed. Both files get the full schema; the
  defect is solely that the registry writes a different file than the
  drone (which is what the runtime reads).

ADR-0012 records the decision: the drone session DB
(`session.sqlite`) is the single source of truth; the registry shares it
via a second WAL connection from the Tauri main process (safe:
`db::init` sets `journal_mode=WAL` + `busy_timeout=5000`, `db.rs:5-7`,
and the migration runner is idempotent, `db.rs:13`, gotcha #80; registry
writes are low-frequency vs the drone's stream).

Why 🔴 for M07: M07's deliverable is the agent-with-tools loop that
reads the registry to resolve/dispatch MCP servers; it reads the live
drone session DB → would find an empty registry → MCP dispatch
non-functional. This is the registry-path-divergence risk realized in
the running app.

Stage-V could not see this: it tested `Registry` in isolation against
one path. It never asserted the **shell** hands the registry and the
drone the *same* path. That assertion is this stage's regression test.

### A.3 Detailed Changes

Introduce a single resolver as the one source of truth for the session
DB path. `resolve_db_path` already returns
`app_local_data_dir().join("session.sqlite")` (`main.rs:298-301`).
Make `open_mcp_client` obtain the registry path from that same resolver
(call `resolve_db_path`, or extract a small path-agnostic helper —
`fn session_db_path(dir: &Path) -> PathBuf` — that both `resolve_db_path`
and `open_mcp_client` call so the resolution is provably identical).
The `Registry` stays path-agnostic (CLAUDE.md §9); only the composition
layer changes. The build also greps for any `"mcp_servers.sqlite"` /
`"mcp.sqlite"` literal anywhere in `src-tauri/` + `crates/` and confirms
none remains (the field 0-byte `mcp_servers.sqlite` was a stale
removed-path artifact; no current code must construct it).

The shared resolver is the testable seam; the `src-tauri` shell
wrappers around it are the documented coverage holdout (CLAUDE.md §5
tauri-shell patch-gate 50%, the `*_with`/seam-vs-wrapper precedent).
ADR-0012's WAL + idempotent-migration rationale is the safety argument
for the second connection; the round-trip test (A.4) pins it.

### A.4 Tests

`src-tauri` assembled-path regression test (new file). The discriminator
is that it exercises the **composition layer's resolution**, not
`Registry` in isolation:

- `registry_path_equals_drone_session_db_path` — the path
  `open_mcp_client` resolves for the registry is **byte-identical** to
  the path `resolve_db_path` returns for the drone (assert via the
  shared resolver / helper, given a `tempdir` as the app-local-data
  dir). This is the assertion Stage-V structurally lacked.
- `add_server_then_list_round_trips_through_the_same_store` — open a
  `Registry` at the resolved path, `add_server`, then open a
  **separately-constructed** `Registry`/`rusqlite::Connection` at the
  **same** resolved path and assert the added server is listed. Proves
  single-file durability + cross-connection visibility (the drone-reads-
  what-the-UI-wrote contract), and exercises ADR-0012's two-connection
  invariant.
- `no_stray_mcp_sqlite_path_literal_constructed` — a guard asserting the
  resolver yields the `session.sqlite` filename (regression pin against
  re-introducing a divergent registry filename).

Acceptance: the three tests pass; full v1.6 canonical gate suite green
(exact CI commands; `cargo llvm-cov clean` first per gotcha #81);
workspace ≥80; the `src-tauri` patch-gate honored (the resolver seam +
round-trip are tested; the shell wrapper is the holdout); no
Co-Authored-By; ADR-0012 flipped to Accepted; v1.7 invariant
`git diff <red>..<impl> -- '**/tests/**'` EMPTY stated in the impl
commit body.

## Stage B.fix — 🔴-2 assembled smoke→drone signal-write wiring

### B.1 Problem Statement

The agent signal stream is never persisted to the live drone DB. After
a smoke run the renderer shows the agent node `complete`,
`tokensTotal 34` (the main→renderer path works) but the live
`session.sqlite` has `signals = 0` and `token_usage = 0` while
`heartbeats` and `snapshots` populate in the **same** DB.

Verified root cause (citations on `main` @ `71107c7`):

- `crates/runtime-main/src/sdk/agent_sdk.rs` —
  `run_agent_with_provider_stream` emits every `AgentEvent` through
  `self.emit(...)`: `AgentSpawned` at `agent_sdk.rs:251`, the run-loop
  events at `agent_sdk.rs:313`, the flush events at
  `agent_sdk.rs:322-323`.
- `agent_sdk.rs:521-526` — `async fn emit(&self, event)` does **only**
  `self.event_tx.send(event)`. It never calls
  `self.drone_client.write_signal(...)`. There is no per-event drone
  write anywhere in the run loop.
- `agent_sdk.rs:262-267` — the run loop's **sole** drone interaction is
  one `self.drone_client.send(DroneCommand::SnapshotNow{…})` at task
  start. Hence `snapshots` land; `heartbeats` land via the drone's own
  background task (`crates/runtime-drone/src/heartbeat.rs`);
  `signals = 0` + `token_usage = 0` because the per-event signal write
  is structurally absent.

Not a no-op drone (the *same* `self.drone_client` successfully sends
`SnapshotNow`), not a path divergence (it writes the correct file —
snapshots prove it). It is a missing emission. The destination side is
ready: `DroneClient::write_signal(signal_id, session_id, kind, event,
context_type, payload)` exists (`crates/runtime-main/src/drone_ipc/client.rs:128`)
and sends the existing `DroneCommand::WriteSignal`; the drone-side
`handle_write_signal` (`crates/runtime-drone/src/command_handler.rs:127`)
exists, writes the `signals` row, and runs the VDR + plan projectors in
the same transaction. Both are unit-tested in isolation;
`crates/runtime-main/tests/recovery_lifecycle.rs:126` proves the
plumbing by calling `client.write_signal()` **manually** three times.

Why 🔴 for M07: the drone is the system-of-record (ADR-0007). With
`signals = 0`, recovery/replay (M01/M04), audit, and every Stage-V
"signals landed" wire-trace are hollow in the running app — M07's loop
+ recovery build on this. Spec §11 requires every `AgentEvent` to reach
the drone signal sink; the assembled smoke path violates it.

Stage-V could not see this: every signal trace either called
`write_signal` directly or tested the drone in isolation. None drove
the assembled `run_smoke_session_with` → `AgentSdk` → `emit` path. That
is this stage's regression test.
