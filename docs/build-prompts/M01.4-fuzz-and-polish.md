## Code expectations

### Fuzz target shape

```rust
// crates/runtime-drone/fuzz/fuzz_targets/drone_command_decode.rs
#![no_main]
use libfuzzer_sys::fuzz_target;
use runtime_core::DroneCommand;

fuzz_target!(|data: &[u8]| {
    // Try to parse as if it were a framed JSON message arriving on the IPC socket.
    // Must not panic; if it deserializes, the result must be a valid DroneCommand.
    if let Ok(s) = std::str::from_utf8(data) {
        let _ = serde_json::from_str::<DroneCommand>(s);
    }
});
```

### CI fuzz step (excerpt for `.github/workflows/ci.yml`)

```yaml
fuzz-smoke:
  name: Fuzz smoke (30s)
  needs: detect-cargo
  if: needs.detect-cargo.outputs.has_cargo == 'true'
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: dtolnay/rust-toolchain@nightly
    - uses: Swatinem/rust-cache@v2
    - run: cargo install cargo-fuzz --locked
    - run: cargo +nightly fuzz run drone_command_decode -- -max_total_time=30
      working-directory: crates/runtime-drone
```

### Nightly fuzz workflow shape

```yaml
# .github/workflows/fuzz-nightly.yml
name: Fuzz Nightly
on:
  schedule:
    - cron: '0 4 * * *'  # 04:00 UTC daily
  workflow_dispatch:
permissions:
  contents: read
jobs:
  fuzz:
    runs-on: ubuntu-latest
    timeout-minutes: 70
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
      - uses: Swatinem/rust-cache@v2
      - run: cargo install cargo-fuzz --locked
      - name: Fuzz drone_command_decode for 1 hour
        run: cargo +nightly fuzz run drone_command_decode -- -max_total_time=3600
        working-directory: crates/runtime-drone
      - name: Upload corpus on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: fuzz-corpus
          path: crates/runtime-drone/fuzz/corpus
```

## Self-correction guidance

| Failure | Likely cause | Fix |
|---|---|---|
| `cargo fuzz build` fails | Missing `cargo-fuzz` install or wrong working dir | Install via `cargo install cargo-fuzz --locked`; run from `crates/runtime-drone/` |
| Fuzz panics in 30s | Real bug in the decoder | Reproduce: `cargo fuzz run drone_command_decode <crash-input>`; fix decoder; add regression unit test |
| Coverage <100% on drone after M01.3 | Untested branch (likely error handling) | Open HTML coverage report; identify; add test |
| `cargo doc` warns missing_docs | A `pub` item lacks doc comment | Add `///` with example; verify with `cargo test --doc` |
| Windows CI failure not present in Linux | Platform-specific code path untested locally | Read CI log; add platform-conditional test if applicable; iterate via push |
| `cargo-mutants` reports many surviving mutations | Test suite doesn't catch them | Address top 5 by adding behavior tests; document the rest in `CHANGELOG.md` as advisory findings |

## Sub-milestone-specific gotchas

1. **`cargo-fuzz` requires nightly.** The fuzz harness package (`crates/runtime-drone/fuzz/Cargo.toml`) is a separate package, not a workspace member. Don't add it to the workspace.
2. **Initial corpus matters.** Seed `corpus/drone_command_decode/` with valid `DroneCommand` JSON examples (one per variant). Without seeds, fuzz starts from scratch and finds shallow bugs slowly.
3. **`cargo-mutants` is opinionated.** Surviving mutations indicate test gaps but aren't necessarily real bugs. Treat findings as advisory; address the top few; document the rest.
4. **Doc examples that need `// ignore` are red flags.** If a doc example would require live network or filesystem state to compile, the API is poorly designed for users. Refactor to take owned values; use `# fn main() -> Result<...> { ... }` for fallible setup.
5. **Cross-OS CI quirks.** Windows clippy occasionally flags lints that Linux doesn't (line-ending issues, path separators in test fixtures). Address by adding platform-conditional `#[cfg(...)]` tests rather than blanket `#[allow(...)]`.
6. **CHANGELOG promotion.** Don't tag `v0.0.1` automatically; surface "ready to tag if you want" in the PR description and let the user decide.
