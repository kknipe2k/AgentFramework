## Background and Disposition

`docs/M08-irl-findings.md` — the manual real-app walk-through of
`docs/M08-irl-test-plan.md` against the merged M08 Workbench (the real
Tauri desktop app on Windows, run elevated, no mocks) — recorded
**3 🔴 · 3 🟡 · 3 🟢**. Per that document's Disposition: the three 🔴
route to *"an M08.5 fix-cycle before M09 — staged X.5 prompts, strict
red→green TDD, gates, and real-app regression tests."* This cycle is
that fix-and-re-test, scoped strictly to the three 🔴 plus the real-app
regression harness they all need.

All three 🔴 are one escape class. M08's unit + Vitest + Playwright +
Stage-V suites were green and the Verifier returned **Sound, 0🔴** — yet
the **real Tauri desktop app** fails three ways, because **no automated
gate, and not the Verifier, runs the real Tauri app.** The Playwright
`e2e` job mocks `@tauri-apps/api` and runs in a plain Chromium; the
`e2e-tauri-driver` job that *would* drive the built binary has been
`if: false` since M03 PR #47 (gotcha #54). Every regression test this
cycle lands for a renderer/shell defect must therefore exercise the
**real running Tauri app** via `tauri-driver` — that is the precise
structural gap, and reviving + gating that harness is the load-bearing
deliverable of this cycle (ADR-0021).

**Verified root causes** (grounded on `main` @ `12f0fb3`, authored
against grep-verified read source — not memory; gotcha #41):

- **🔴-1 — palette drag is dead; Tauri's native drag-drop swallows
  HTML5 DnD.** `src/components/builder/Palette.tsx:138-156` makes each
  item a native HTML5 drag source (`draggable`,
  `onDragStart` → `e.dataTransfer.setData('application/x-builder-node',
  …)`); `src/components/builder/BuilderCanvas.tsx:81-97` is the drop
  target (`onDragOver` → `preventDefault`, `onDrop` →
  `e.dataTransfer.getData(DND_MIME)` → `addNode`). The renderer DnD
  contract is correct. The defect is the Tauri shell:
  `src-tauri/tauri.conf.json` declares the `main` window with **no**
  `dragDropEnabled` key, so it takes the Tauri 2.x default `true` —
  which, per the official Tauri docs, means *"Tauri's internal drag and
  drop system is enabled, and DOM drag and drop is disabled."* On
  Windows (the v0.1 target), `dragDropEnabled: false` is **required** to
  use HTML5 DnD on the frontend. React Flow's pointer-drag node
  *reposition* works because it is not HTML5-native DnD — which is why
  the IRL pass saw repositioning work and instantiation not.

- **🔴-2 — the Tester emits the candidate's root agent as `"smoke"`.**
  `crates/runtime-main/src/sdk/agent_sdk.rs:333-390` (`session_prelude`)
  derives the runtime `agent_id` correctly — `agent_sdk.rs:339-342`,
  when `capability_wiring` is present, sets it to the framework's
  `session_root_agent` — but then `agent_sdk.rs:348-355` emits the root
  `AgentEvent::AgentSpawned` with **`agent_name: "smoke".to_string()`
  hardcoded**. The IRL pass loaded ARIA and saw a node labelled
  `"smoke"`, and reasonably concluded the Tester "runs a hardcoded
  smoke session." The grep-verified reality refines that hypothesis:
  `run_test_session_with` (`tester.rs:295`) **does** substitute the
  candidate framework into the run — agent id (the root), capability
  enforcement (`CapabilityWiring::new(.., Arc::new(framework.clone()),
  ..)`, `tester.rs:322-326`), the model
  (`test_agent_config` → `framework.model.id`, `tester.rs:259`), and
  every declared sub-agent (`spawn_framework_subagents`,
  `agent_sdk.rs:456-515`, which already names each child with its real
  `role`). The **one** thing not substituted is the *root* agent's
  display name. The fix is to derive it the same way the sub-agents
  already do — from the framework's root agent `role`. The hardcoded
  `"smoke"` mislabel is what broke MVP §M8 criterion 5's "graph runs"
  for the IRL observer.

- **🔴-3 — the "Add MCP Server" modal renders but Test / Add / Cancel
  are all non-responsive.** This is the one finding whose exact root
  cause is **not determinable by static source reading**, and the phase
  doc says so honestly (gotcha #41 — a claim that cannot be verified is
  not asserted as verified). What *is* verified: `MCPServerAddModal.tsx`
  is correct in isolation — `handleTest` / `handleSubmit` / `onClose`
  are defined and bound; Test and Cancel are `type="button"` and
  unconditionally enabled; Add is `type="submit"` gated on `nameValid`.
  `MCPServerSettings.tsx:88` mounts it as `{showAdd && <MCPServerAddModal
  …/>}`. `styles.css:1059-1067` gives `.mcp-server-add-modal-backdrop`
  `position: fixed; inset: 0; z-index: 50`. No app element stacks above
  `z-index: 50` in the Runtime view: the only other `z-index: 50`
  element (`.gap-panel`) is earlier in DOM order, and every
  post-`.graph-layout` overlay component (`HITLModal`, `HITLToast`,
  `RecoveryDialog`, `UncertaintyPrompt`) returns `null` with nothing
  pending while `SqlInspector` is a normal-flow `<section>`. The dead
  buttons are therefore a **real-runtime (WebView2) behavior the static
  renderer code does not explain** — which is exactly the bug class
  this cycle's harness exists to catch. Stage D.fix is **diagnosis-first**
  (D.3): the red `tauri-driver` test reproduces the dead button against
  the real app, the build diagnoses the actual interception with the
  live DOM, then fixes — the v1.8 "root cause is a falsifiable
  hypothesis the regression test must disprove" model, applied honestly.

### Why five stages

The cycle has one cross-stack-infrastructure deliverable (the harness),
three independent 🔴 in three different layers, and a closing
re-verification. The split is by **concern, layer, and risk**:

- **A.fix (the harness) must come first and stand alone.** It is the
  precondition for B.fix's and D.fix's real-app regression tests — they
  cannot have a `tauri-driver` test until the harness runs. It is also
  a **cross-stack-integration risk** (the WebdriverIO ↔ `tauri-driver`
  matrix is the exact thing that disabled the job at M03 PR #47):
  isolating it means a stuck harness does not entangle a 🔴 fix's
  red→green diff. It ships no 🔴 fix.
- **B.fix (🔴-1) is a one-line `tauri.conf.json` change** with a verified
  root cause — but it earns its own stage because its regression test
  is the harness's first new real-app test, and strict TDD wants a
  clean red→green isolated from the harness-revival churn.
- **C.fix (🔴-2) is a Rust SDK change** (`agent_sdk.rs`) — a different
  language and layer from B/D, with a Rust assembled regression test
  (`crates/runtime-main/tests/`), not a `tauri-driver` test:
  🔴-2 is backend logic the renderer merely displays, so the assembled
  composition for it is `run_test_session_with`, not the webview.
- **D.fix (🔴-3) is a renderer/shell defect with an undiagnosed root
  cause** — it needs the harness as a *diagnosis instrument*, not just
  a regression gate, and a diagnosis-first stage muddied into a code
  fix would have no clean red phase.
- **E.fix is the closing re-verification + reconciliation.** The
  findings doc gates the M08 demo on a clean re-run of
  `docs/M08-irl-test-plan.md`; that re-run + the contract-fidelity pass
  + the CHANGELOG / summary / CLAUDE.md-gate-list / M08.6 carry-forward
  reconciliation is real work that, folded into D.fix, would muddy
  D.fix's red→green diff.

No Stage V: this is a D.fix-class remediation, not a milestone — per
ADR-0008 the D.fix loop is itself the response to findings; it does not
recurse a V (the M06.5 / M07.5 fix-cycle precedent — no X.5 IRL fix
cycle has carried a Stage V). The verification endpoint is threefold:
the now-gating real-app `tauri-driver` harness, strict-TDD's
red-must-fail-on-`main` per 🔴, and **Stage E's hardened
contract-fidelity pass** (E.3) — which explicitly re-confirms each 🔴's
regression test fails on pre-fix `main` and was not weakened to pass,
the cross-stage guard against a large error carrying over. M08.6 (a
milestone-class build) carries the full Stage V; M08.5 (a fix cycle)
does not.

### Key constraints

- **Real-app regression mandate (load-bearing — CLAUDE.md §6 v1.8;
  gotcha #82; ADR-0021).** Each renderer/shell 🔴 fix's regression test
  MUST exercise the **real running Tauri app** via `tauri-driver`, not
  a Playwright-mocked browser and not only a Rust integration test. The
  phase-doc root cause is a **falsifiable hypothesis the regression
  test must disprove**: B.fix's drag test drives a real palette→canvas
  drag and asserts a node appears — it **fails on `main`**; D.fix's
  modal test drives a real Cancel click and asserts the modal closes —
  it **fails on `main`**. 🔴-2 is the exception that proves the rule:
  it is backend logic, so its assembled-app path is the real
  `run_test_session_with` composition (`crates/runtime-main/tests/`,
  the `tester_isolated_session.rs` harness) — a `tauri-driver` test is
  not required for it (C.4).
- **Strict v1.8 two-commit TDD on B.fix + C.fix + D.fix.** Failing
  tests committed standalone (`test(M08.5.X.fix): …`) and surfaced for
  red-phase approval **before** any implementation; the impl commit
  must **not** modify test files; the impl commit body states the
  verifiable invariant `git diff <red-sha>..<impl-sha> -- '<tests>'` is
  EMPTY. B.fix and D.fix touch `tests/e2e-tauri/**`; C.fix touches
  `crates/runtime-main/tests/**`. Net-new additive tests + mechanical
  fmt/clippy on test files go in a separate labelled follow-up commit.
  **A.fix is exempt** from strict two-commit TDD — it is CI/harness
  infrastructure revival, not a production red→green code change; its
  acceptance is the existing six smoke tests green in the gating job
  (the M06.5.C.fix / M07.5.D.fix "no `<tdd_discipline>`" precedent for
  a non-code stage).
- **Cross-stack integration: quote, do not hand-author (gotcha #32).**
  A.fix's WebdriverIO / `tauri-driver` configuration MUST be quoted
  verbatim from a verified-working upstream reference (the official
  Tauri 2.x WebDriver docs, `github.com/tauri-apps/webdriver-example`,
  or `github.com/Haprog/tauri-wdio-win-test`) — never authored from
  docs/memory. M03 PR #47 disabled this exact job by shipping a
  hand-synthesized capabilities object. A.fix carries the
  cross-stack-escalation budget below.
- **Cross-stack escalate-at-iteration-2 (ORCHESTRATOR.md §3; the M03
  PR #47 lesson).** For A.fix's harness revival specifically: if
  self-correction iteration 2 advances the failure to a *new* error
  class rather than clearing it, that is a structural signal — the
  build escalates to the maintainer immediately rather than iterating
  to the standard 3-round budget. A.3 names the primary path and the
  ordered fallbacks so an escalation has a decision already framed.
- **No scope creep.** Touch only what the three 🔴 + the harness
  require. No 🟡 / 🟢 fixes, no `system_prompt_template` wiring, no
  multi-agent engine, no protocol edits, no refactors. A bug fix does
  not need surrounding cleanup (CLAUDE.md §"Doing tasks").
- **No new Tauri-shell dependency on native drag-drop.** Grep confirms
  nothing in `src/` or `src-tauri/src/` uses Tauri's `onDragDropEvent`
  / `getCurrentWebview` / native file-drop — so `dragDropEnabled: false`
  removes no feature. B.fix re-confirms this grep before shipping.
- **No Co-Authored-By trailers.** Project convention + model-identity
  rule. Commit messages end with the session-URL footer only.
- **Windows is the v0.1 target; CI runs Windows + Linux for the real-app
  job** (macOS excluded — gotcha #23 / ADR-0021). The `tauri-driver`
  harness mirrors the existing `wdio.conf.ts` + `tests/e2e-tauri/`
  structure already in the tree.
- **Capability adherence (CLAUDE.md §10).** The `smoke.e2e.ts`
  happy-path test makes one real low-budget Anthropic call via the
  existing `ANTHROPIC_TEST_KEY` CI secret (the pre-existing M03 design);
  no other real-app test reaches a live API. C.fix's Rust assembled
  test uses a stub provider — no live Anthropic.

  gating input) and `docs/build-prompts/M08.5-irl-fix.md` +
  `docs/adr/0021-real-app-tauri-driver-regression-gate.md` (`Proposed`).
  `docs/M08-irl-findings.md` is on PR #96 at cycle-authoring time; that
  PR (plus this phase doc + ADR-0021) must merge to `main` before the
  M08.5 build machine cuts its branch. The Stage A.fix
  `<pre_flight_check>` asserts all three files are present and that the
  `e2e-tauri-driver` job is still `if: false` (the bug is still live).
- **Branch:** the build machine cuts `claude/m08.5-irl-fix` from `main`.
  One branch, five stages, commits in order. Strict v1.8 two-commit on
  B.fix + C.fix + D.fix means each of those is two commits (red, then
  impl) plus an optional follow-up; surface for red-phase approval and
  again for final approval per stage.
- **ADR:** `docs/adr/0021-real-app-tauri-driver-regression-gate.md` is
  authored `Proposed` alongside this phase doc. Stage A.fix flips it
  `Proposed → Accepted` in its impl commit (the M06.5.A.fix / ADR-0012
  precedent — the stage that implements an ADR flips it).
- **Gates:** the full canonical gate suite in the canonical ordering
  (CLAUDE.md §6: `cargo fmt` → `cargo clippy --fix -p <crate>` →
  `cargo clippy --workspace -- -D warnings` → test/doc/audit/deny →
  `cargo llvm-cov clean` then the llvm-cov gates → frontend gates).
  From Stage A.fix onward the `e2e-tauri-driver` job is part of the
  suite. CI-parity is a hard rule — exact CI commands, no divergent
  flags.
- **Do not push between stages.** Surface each stage; the maintainer
  approves; the agent pushes on approval (CLAUDE.md §8). Cross-machine
  state (`git log --oneline main..HEAD` + retro listing) is a top-level
  surface item per the retrospective protocol.
- **A.fix-stuck contingency.** If A.fix cannot clear the harness on
  Windows + Linux CI within the cross-stack escalation budget (the
  escalate-at-iteration-2 rule above), the build escalates to the
  maintainer with the error matrix. B.fix and D.fix's real-app
  regression tests depend on A.fix; a maintainer scope call (proceed
  with documented-manual IRL verification + a harness follow-up, vs.
  hold the cycle) is made at that point. The default plan is A.fix
  succeeds.

## Stage A.fix — revive + gate the real-app `tauri-driver` harness

### A.1 Problem Statement

No automated gate runs the real Tauri desktop app, so a renderer or
Tauri-shell defect ships green. All three M08-IRL 🔴 escaped through this
gap.

Verified (citations on `main` @ `12f0fb3`):

- `.github/workflows/ci.yml:594-656` — the `e2e-tauri-driver` job exists
  (matrix `ubuntu-latest` + `windows-latest`, installs `tauri-driver`,
  builds the app, runs `npm run test:e2e:tauri`) but its condition is
  `.github/workflows/ci.yml:597` — **`if: false && …`**. The job has
  never run. The comment block `ci.yml:569-593` records the M03 PR #47
  history: three iterations on `wdio.conf.ts` capabilities did not clear
  both platforms; "Upstream wdio v9 + tauri-driver 2.x compat is
  unresolved … only confirmed-working community example pins wdio@7";
  "Deferred to M04." It was never revived through M04–M08.
- `wdio.conf.ts` (repo root) — a complete, official-shape WebdriverIO
  v9 config: `runner: 'local'`, `framework: 'mocha'`,
  `specs: ['./tests/e2e-tauri/**/*.e2e.ts']`, capabilities that
  intentionally omit `browserName` with `'tauri:options': { application:
  APP_BIN_PATH }`, and `onPrepare` / `onComplete` hooks that spawn /
  kill `tauri-driver`.
- `tests/e2e-tauri/smoke.e2e.ts` — six mocha/chai tests (app launch +
  SetupPanel; save-key flow; smoke happy-path with a real Anthropic
  call; click-to-inspect; SQL inspector; reload-reconstructs-graph).
- `package.json` — `webdriverio` + `@wdio/*` at `^9.0.0`, `chai`,
  `@types/mocha`, and the `"test:e2e:tauri": "wdio run wdio.conf.ts"`
  script are all present. The harness is fully scaffolded; only the CI
  gate is dead.

Why this gates the cycle: B.fix (🔴-1) and D.fix (🔴-3) are
renderer/Tauri-shell defects whose regression tests can only be a
`tauri-driver` test driving the real app — they cannot exist until this
harness runs. ADR-0021 records the decision to make the job a required
blocking gate.

## Stage B.fix — 🔴-1 palette drag (`dragDropEnabled: false`)

### B.1 Problem Statement

Dragging a Palette item onto the Builder canvas does nothing — no
dragstart, no drop, no node instantiated. React Flow's pointer-drag node
*reposition* works; only HTML5-native drag-and-drop is dead.

Verified root cause (citations on `main` @ `12f0fb3`):

- `src/components/builder/Palette.tsx:138-156` — each palette item is a
  native HTML5 drag source: `<li … draggable onDragStart={(e) => {
  e.dataTransfer.setData('application/x-builder-node', JSON.stringify({
  kind, ref })); e.dataTransfer.effectAllowed = 'copy'; }}>`.
- `src/components/builder/BuilderCanvas.tsx:81-97` — the canvas is the
  drop target: `onDragOver` calls `e.preventDefault()` +
  `dropEffect = 'copy'`; `onDrop` reads
  `e.dataTransfer.getData(DND_MIME)`, parses the payload, and calls
  `addNode(kind, ref, screenToFlowPosition(...))`.
- The renderer DnD contract is correct end-to-end. The defect is the
  Tauri shell: `src-tauri/tauri.conf.json:12-20` declares the `main`
  window with `label` / `title` / `width` / `height` and **no
  `dragDropEnabled` key**. Tauri 2.x defaults `dragDropEnabled` to
  `true`, which — per the official Tauri docs — *"enables Tauri's
  internal drag and drop system and disables DOM drag and drop."* On
  Windows, `dragDropEnabled: false` is required for HTML5 DnD on the
  frontend. The OS handler intercepts the drag gesture before the
  webview's HTML5 DnD ever sees it.

Why 🔴 for the demo: MVP §M8 acceptance criterion 1 — "User drags an
Agent node onto empty canvas" — is the primary Builder interaction; with
drag dead, a framework cannot be composed on the canvas at all.

Why this escaped: the Playwright `e2e` job runs in a plain Chromium
where HTML5 DnD works natively and `@tauri-apps/api` is mocked — the
Tauri-shell `dragDropEnabled` default is invisible to it. Only the real
app exhibits the defect; only the Stage A.fix `tauri-driver` harness can
catch it.

## Stage C.fix — 🔴-2 Tester emits the candidate's root agent

### C.1 Problem Statement

The Builder's Tester runs the loaded framework, but the live graph
labels the framework's root agent `"smoke"` — so the run reads as a
smoke session, not the candidate.

Verified root cause (citations on `main` @ `12f0fb3`):

- `crates/runtime-main/src/sdk/agent_sdk.rs:333-390` — `session_prelude`
  runs once per session. `agent_sdk.rs:339-342` derives the runtime
  `agent_id` correctly: `self.capability_wiring.as_ref().map_or_else(||
  format!("agent_{}", Uuid::new_v4()), |w|
  w.framework.session_root_agent.clone())` — with wiring present, the
  id **is** the framework's `session_root_agent`.
- `agent_sdk.rs:348-355` — it then emits the root
  `AgentEvent::AgentSpawned` with **`agent_name: "smoke".to_string()`
  hardcoded**, with the inline comment "The smoke / streaming-only spawn
  site predates the L2a wire-up."
- `agent_sdk.rs:456-515` — `spawn_framework_subagents` already emits
  every declared *sub*-agent's `AgentSpawned` with `agent_name:
  child.role.to_string()` (`:480`). The root agent is the **only**
  agent whose name is not derived from the framework.
- `crates/runtime-main/src/builder/tester.rs:295-344` —
  `run_test_session_with` builds `CapabilityWiring::new(enforcer,
  Arc::new(framework.clone()), HitlSeam::test_defaults())`
  (`:322-326`), constructs the SDK with `with_capability_wiring`
  (`:327-328`), and runs `sdk.run_agent(test_agent_config(framework,
  task))`. The candidate framework **is** substituted into the run —
  the root agent id, the capability enforcement, the model
  (`test_agent_config` → `framework.model.id`, `tester.rs:259`), and
  every sub-agent. The IRL observer saw a `"smoke"` node and reasonably
  inferred "it runs a smoke session"; the grep-verified reality is
  narrower — the Tester runs the candidate, but `session_prelude`
  mis-labels the root agent.
- `crates/runtime-main/src/sdk/agent_sdk.rs` — when `capability_wiring`
  is `None` (the real smoke session: `run_smoke_session_with` builds the
  SDK with `AgentSdk::new`, `commands.rs:233`), `"smoke"` is the correct
  name and must stay.

The fix: in `session_prelude`, when `capability_wiring` is present,
derive the root `agent_name` from the framework's root agent `role` —
exactly as `spawn_framework_subagents` already derives sub-agent names.

Why 🔴 for the demo: MVP §M8 criterion 5 — "Click Test → … → graph
runs" — the graph that runs must visibly be the candidate framework's.
A root node labelled `"smoke"` makes the Tester appear not to run the

The fixture's `session_root_agent` is `"worker"`, but a trace carrying

### C.3 Detailed Changes

#### C.3.1 `session_prelude` — derive the root agent name from the framework

`agent_sdk.rs:339-355` currently reads:

```rust
        let agent_id = self.capability_wiring.as_ref().map_or_else(
            || format!("agent_{}", Uuid::new_v4()),
            |w| w.framework.session_root_agent.clone(),
        );
        // The smoke / streaming-only spawn site predates the L2a wire-up
        // (M06 Stage A); top-level agents have no parent grants to narrow
        // against, so `narrowed_from` is empty here. The framework-walk
        // spawn site below (`spawn_framework_subagents`) populates it for
        // every sub-agent that flowed through `narrow()`.
        self.emit(AgentEvent::AgentSpawned {
            agent_id: agent_id.clone(),
            agent_name: "smoke".to_string(),
            parent_id: None,
            session_id: self.session_id.as_string(),
            narrowed_from: Vec::new(),
        })
        .await?;
```

Derive `agent_name` the same way `agent_id` is derived — from the
framework when wiring is present, the literal `"smoke"` when not:

```rust
        let agent_id = self.capability_wiring.as_ref().map_or_else(
            || format!("agent_{}", Uuid::new_v4()),
            |w| w.framework.session_root_agent.clone(),
        );
        // The root agent's display name: with capability wiring (the
        // Tester / a framework run) it is the framework root agent's
        // `role` — the same source `spawn_framework_subagents` uses for
        // every sub-agent (agent_sdk.rs:480). Without wiring (the smoke
        // / streaming-only session) there is no framework, so the
        // literal "smoke" stays correct. M08.5 🔴-2.
        let agent_name = self.capability_wiring.as_ref().map_or_else(
            || "smoke".to_string(),
            |w| root_agent_role(&w.framework, &agent_id),
        );
        self.emit(AgentEvent::AgentSpawned {
            agent_id: agent_id.clone(),
            agent_name,
            parent_id: None,
            session_id: self.session_id.as_string(),
            narrowed_from: Vec::new(),
        })
        .await?;
```

`root_agent_role(framework, root_id)` resolves the root agent's display
name from the framework: find the entry in `framework.agents[]` whose id
is `root_id` and return its `role`. The `agents[]` items are a `oneOf`
of an inline `agent.v1.json` (carries `role`) or a `{ id, path }`
path-reference (no inline `role`). The Tester's Rust test fixtures use
the inline form; the v0.1 reference frameworks `examples/aria/` and
`examples/ralph/` both use the **path-ref** form (their agents are
`{ id, path }` refs into `agents/*.md`) — so the path-ref fallback is
the *common production case*, not an edge case. For the path-ref form,
fall back to the agent `id` (a path-ref root has no inline `role` to
read; resolving the external `.md` is M08.6's loader work — ADR-0022 —
not this fix's scope; for ARIA the Tester root then labels as
`orchestrator`, the candidate's real root, not `smoke`). The build
chooses the placement — a small private helper in
`agent_sdk.rs`, or in `framework_loader.rs` next to the existing
`inline_agents` / `parent_grants_for_agent` walkers (which already
iterate `framework.agents[]` and are the natural home). Whatever the
placement, the resolution must be a pure function unit-tested directly.

This is the entire 🔴-2 fix. `test_agent_config` (`tester.rs:258-272`)
is **not** changed: its `system_prompt: None` reflects a runtime-wide
gap (`system_prompt_template` is consumed nowhere in `runtime-main` —
"What's not in scope"), and wiring it Tester-only would be scope creep.
Stage E records the `system_prompt_template` non-application as an M09
carry-forward.

Note on `agent_sdk.rs` as a sensitive path (CLAUDE.md §4 rule 8): this
change touches the SDK run-loop prelude but **not** capability
enforcement — the enforcer, the `narrow()` calls, and `EventPipeline`
wiring are untouched; only the root `AgentSpawned` event's display-name
field changes. This phase doc is the surfaced plan.

### D.3 Detailed Changes

#### D.3.1 Diagnose first (the red phase IS the diagnosis)

The red `tauri-driver` test (D.4) reproduces the dead button against the
real running app — that failure is the reproduction. With the real app
under WebDriver, the build then diagnoses the actual interception
before writing any fix:

- Query the element at a dead button's screen coordinates
  (`document.elementFromPoint(x, y)` via `browser.execute`) — if it is
  not the button, the returned element **is** the interceptor.
- Inspect the computed `pointer-events`, `position`, `z-index`, and the
  stacking context of the modal, the backdrop, and every ancestor.
- Confirm whether the modal's buttons are within the viewport (the
  `.mcp-server-add-modal` panel has no `max-height`/`overflow`; a panel
  taller than the 800px window, flex-centered in the
  `align-items: center` backdrop, overflows top **and** bottom with no
  scroll — the bottom action row could be unreachable).

The build records the **confirmed** root cause in the Stage D retro and
the impl commit body — the v1.8 "falsifiable hypothesis the regression
test disproves" model: the diagnosis replaces the hypothesis with a fact.

#### D.3.2 Fix per the diagnosis — the likely shape

The build implements the fix the diagnosis confirms. The
best-practice modal fix, robust across the candidate causes (a trapped
stacking context, a transparent overlay, an off-viewport action row),
is the standard React portal pattern:

- Render the modal through `ReactDOM.createPortal(…, document.body)` so
  it escapes any ancestor stacking context and is a direct child of
  `<body>` — the same pattern the M07.E `import-review-modal` and any
  robust modal use. A portal makes the modal's stacking independent of
  `.graph-layout` and its sibling panels entirely.
- Give the modal layer a `z-index` above every other overlay
  (the `import-review-modal` uses `1000`; the modal layer should be
  consistent with — or above — that, and the build reconciles the
  modal `z-index` scale rather than leaving the MCP modal at the
  lowest `50`).
- Harden `.mcp-server-add-modal` with `max-height` + `overflow: auto`
  to match `.import-review-modal` (`:1223-1232`) so a tall form never
  pushes its action row off-viewport.

If the diagnosis surfaces a *different* concrete interceptor (e.g. a
specific transparent fixed element from another component), the build
fixes **that** as well — the portal + z-index + overflow hardening is
the modal's own robustness; a separate interceptor is its own fix. The
build does not ship the portal change as a guess: it ships what makes
the red `tauri-driver` test go green, and the retro states the
confirmed cause.

No §11 ADR: a renderer modal-rendering change is no schema / capability
/ IPC / dependency change.
