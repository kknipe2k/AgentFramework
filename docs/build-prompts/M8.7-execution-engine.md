> `<tdd_discipline strict="true">` two-commit on every code rung.
>
> **Governed by CLAUDE.md §4 rule 11 (grounded-claims / no-gaslighting).**

## Background — the grounded engine map

The post-M08.6 IRL walk (`docs/M08.6-irl-findings.md`) and a direct read
of the execution path establish what the runtime **executes** versus what
it **paints** (emits as events for the live graph) or **checks**
(capability gates) without executing. This is verified, not inferred:

**Executes real work (cited):**

- Single-agent multi-turn provider streaming — `AgentSdk::run_agent`
  (`crates/runtime-main/src/sdk/agent_sdk.rs:256`): prelude once, then
  `provider.stream` → `drive_stream` per turn, terminating when a turn
  dispatches no tool (`agent_sdk.rs:261`) or hits `MAX_AGENT_TURNS = 16`.
- MCP tool dispatch — `try_mcp_dispatch` (`agent_sdk.rs:576`) calls the
  injected `McpToolDispatch` seam, emits agent-correct
  `ToolInvoked`/`ToolResult`, and returns a `DispatchedTool` the
  multi-turn loop feeds back as the next turn's `tool_result`
  (`agent_sdk.rs:269-294`). This is the **only** tool path that
  executes + feeds back. (Modulo M08.6 #23 — the Windows `npx` spawn
  bug.)
- L1 capability enforcement — `EventPipeline::next_event(ToolUse)`
  (`crates/runtime-main/src/sdk/event_pipeline.rs:102`) emits
  `ToolInvoked` or, on a blocked check, `CapabilityViolation` and omits
  `ToolInvoked`; the loop routes violations through
  `await_capability_violation_hitl` (`agent_sdk.rs:539`).
- L2a narrowing **check** at spawn — `spawn_framework_subagents`
  (`agent_sdk.rs:467`) runs `narrow(parent_grants, proposed)` per inline
  child and emits `AgentSpawned` or `CapabilityViolation`.
- Event emission + drone persistence + start-of-session snapshot —
  `emit` → `persist_signal` (`agent_sdk.rs:681`,`:717`) and the
  `SnapshotNow` in `session_prelude` (`agent_sdk.rs:370`).

**Painted or checked, NOT executed (verified gap):**

- **Built-in tools** (`Read`/`Write`/`Bash`/`Glob`/`Grep`/`WebFetch`) —
  **no executor exists.** `grep` across `crates/` + `src-tauri/` for any
  built-in tool executor (`execute/run/dispatch_tool`, `read_file`,
  `run_bash`, `ToolExecutor`) returns nothing. A built-in `ToolUse`
  emits `ToolInvoked` via the pipeline and is never run; the multi-turn
  loop only feeds back MCP tools, so `feedback.dispatched` stays empty
  and the loop breaks. The file is never read; no result feeds back.
- **Sub-agents** — `spawn_framework_subagents` emits `AgentSpawned` and
  stops. No child execution context is created; no child runs its own
  `drive_stream`. Only the session root streams.
- **`LoadSkill`** — no handler (grep). Skills are never injected into
  context.
- **gap / `request_capability`** — `handle_request_capability`
  (`crates/runtime-main/src/sdk/request_capability.rs:88`) exists but is
  **not wired into the run loop** (called only from its own tests). A
  model `request_capability` `ToolUse` falls through to
  `pipeline.next_event` (emits `ToolInvoked`) — the gap flow never fires.
- **Plans** — `drive_plan` (`crates/runtime-main/src/plan/plan_loop.rs:79`)
  advances the FSM and emits `plan_approval_requested → plan_approved →
  plan_complete`, but its own docstring states it "runs no tasks"
  (`plan_loop.rs:7-9`,`:128-129`), and it has **no production caller**
  (M08.V 🟡 #2).
- **Hooks / rails** — defined in framework JSON; no firing engine.
- **Modes** — STANDARD hardcoded (gotcha #3); no router.

The summary: M1–M8 built the streaming / event / capability-check / MCP /
Builder scaffolding **around** an execution core that does
provider-streaming + MCP dispatch + checks, and paints the rest. The
*external* protocol path (MCP) executes; the *built-in* path does not.
M8.7 builds the execution layer.

## What M8.7 is (and is not)

**Is:** the milestone that makes the runtime *run* what frameworks
declare — built-in tool execution, skill loading, gap-flow wiring,
sequential multi-agent spawn, plan-driven task execution, hook/rail
firing — climbed as an eval-first **ladder** of minimal frameworks, each
run and IRL-watched, ending at *import → fix → run ARIA* (the v0.1
subset).

**Is not:** concurrent multi-agent orchestration (v1.0 — sequential is
the v0.1 floor that proves the engine; parallel is the validated
high-value follow-on per the multi-agent research); the mode router
(STANDARD stays hardcoded, §0d); a new provider (Anthropic-only); the
capability **grant-right/use-right split** (v1.0 ADR — see "Capability
model" below). The Builder authoring surface is M8.6.7's track, not
this one — M8.7 fixtures are hand-authored JSON loaded from disk, so
M8.7 does not depend on Builder correctness.

## The ladder (full definition; rungs are clusters)

Each rung isolates one function so a break is diagnosable (the
opposite of throwing ARIA at a half-built engine). Rungs 1–5 are the
single-agent execution primitives; 6–8 are coordination/policy; 9 is
the one path that already runs; N is ARIA-as-integration.

| Rung | Function | Reality (cited) | Build |
|---|---|---|---|
| 0 | smoke baseline | ✓ runs (no tools) | none — the baseline |
| 1 | built-in tool execution | ✗ no executor | **build the built-in tool executor + feedback into the multi-turn loop** |
| 2 | capability enforce on a *live* tool | partial (checks; tool never ran) | gate a *real* execution: allowed → runs, blocked → does not |
| 3 | skill load | ✗ no handler | **build `LoadSkill`** (inject skill md into context) |
| 4 | gap detection | handler exists, unwired | **wire `request_capability` into the run loop** |
| 5 | budget | hypothesis: painted | **ground at rung entry**, then make spend drive warn/downshift/stop |
| 6 | sequential spawn ⭐ | ✗ emit-only | **build per-child execution context + sequential coordination** |
| 7 | plan-driven tasks | FSM-only, no caller | **wire `drive_plan` + build the task loop** |
| 8 | hooks + rails | ✗ no engine | **build the firing engine** |
| 9 | MCP | ✓ runs | fix M08.6 #23 (Windows npx) |
| N | ARIA | — | import → fix shape → run the v0.1 subset; document the v1.0 line |

## Milestone structure — three sub-phases, each with a clear ending

Per the maintainer's "clear ending in the current phase" requirement and
the construction-reachability discipline (do not write a detailed spec
that rests on an unverified upstream outcome — under §4 rule 11 that
would itself be an ungrounded claim), M8.7 splits into three sub-phases.
**Only M8.7a is detailed below**; M8.7b and M8.7c are structured, and
their per-rung detail is authored *after* M8.7a's evidence lands —
because rung 6+ specs depend on what the rung 1–5 execution layer
actually requires (e.g., whether per-child execution reuses
`drive_stream` as-is or needs a re-entrant variant is a rung-1/rung-6
empirical question). This deferral is grounded-reason, not effort.

| Sub-phase | Rungs | Ending |
|---|---|---|
| **M8.7a** | 1–5 | a single agent **executes** built-in tools, loads skills, triggers gaps, and respects budget — IRL-watched |
| **M8.7b** | 6–7 | a **multi-agent** framework runs sequentially; a plan drives tasks — IRL-watched |
| **M8.7c** | 8–9, N | hooks/rails fire; MCP fixed; **ARIA runs** the v0.1 subset — IRL-watched |

## Key constraints (all rungs)

- **Eval-first, IRL-gated, zero-propagation.** Each rung's BDD acceptance
  + assembled test is authored and committed *before* the build (red),
  runs the *real* path, and the rung does not close until the maintainer
  IRL-watches it run. Findings from a rung's IRL are triaged in-phase
  (fix-in-rung / new rung / explicit ADR-class scope-out) — never routed
  forward silently (§4 rule 11 + the zero-propagation rule).
- **Grounded claims (§4 rule 11).** Every rung surface states what was
  run and what was not. "Tool executes" requires the assembled app (or
  assembled integration test) to have run the tool and observed the
  result — not a unit test that injects a `ProviderEvent::ToolUse` and
  asserts the emitted event (that licenses "the event is emitted," the
  exact trap that hid this gap).
- **Capability model: strict child ⊆ parent narrowing stays** (validated
  — object-capability theory + AI-agent delegation best practice: you
  may only delegate authority you hold; attenuation only narrows). The
  shipped `narrow()` is correct. M08.6 #31 (ARIA's orchestrator
  under-declares the capabilities it delegates) is fixed at rung N by
  **declaring ARIA's grantable union on the orchestrator**, not by
  weakening narrowing. The **grant-right vs use-right split** (let a
  coordinator be the source of a delegated capability while attenuating
  its own use — AWS PassRole / macaroon pattern) is the correct **v1.0
  ADR**; it still requires the orchestrator to hold a grant-right, never
  to grant from nothing.
- **Sandbox boundary (Hard Rule 8 + §10).** Built-in tools that touch
  the OS (`Bash`, and `Write`/`Read` outside the workspace) execute
  through `crates/runtime-sandbox/`, never directly in main. The tool
  executor's design (rung 1) surfaces a plan for which built-ins route
  through the sandbox vs run in-process under capability scope — this is
  a CODEOWNERS-flagged path; the rung-1 prompt requires the plan before
  code.
- **No new third-party deps without `cargo deny`.** No telemetry. No
  `unsafe` outside `runtime-sandbox`. No schema change unless a rung
  proves one is needed (then: new `*.v2.json` + ADR).
- **Strict v1.8 two-commit TDD** on every code rung; the impl commit's
  `git diff <red>..<impl>` over the test paths is EMPTY.

## M8.7a — single-agent execution primitives (rungs 1–5)

> Detailed to build standard. Rungs 1–4 are groundable from the code read
> this session; rung 5 (budget) carries an explicit **ground-at-entry**
> step because the budget-tracking surface was not read this session and
> §4 rule 11 forbids speccing it from assumption.

### Rung 1 — built-in tool execution

**1.1 Problem (cited).** A built-in `ProviderEvent::ToolUse` reaches
`drive_stream` (`agent_sdk.rs:412`), is offered to `try_mcp_dispatch`
which returns `Ok(None)` (not an MCP tool, `agent_sdk.rs:607`), and falls
through to `pipeline.next_event` which emits `ToolInvoked` and stops.
Nothing executes the tool; nothing feeds a `tool_result` back; the
multi-turn loop sees empty `feedback.dispatched` and breaks. The agent
cannot do work with a built-in tool.

**1.2 Deliverable.** A built-in tool executor that, for a `ToolUse`
naming a built-in the agent is capability-permitted to call: executes the
tool (in-process under capability scope, or through `runtime-sandbox` for
OS-touching tools per the sandbox-boundary plan), produces a structured
result, emits the agent-correct `ToolInvoked` + `ToolResult`, and returns
the result so the existing multi-turn feedback path
(`agent_sdk.rs:269-294`) re-streams it to the model — exactly mirroring
the MCP path's shape (`try_mcp_dispatch` → `DispatchedTool` → feedback),
so built-in and MCP tools converge on one feedback contract.

Scope rung 1 to the minimal, non-OS, capability-simple built-ins first
(start with `Read` against a workspace-scoped path) to isolate the
executor wiring from the sandbox-integration question; `Bash`/`Write`
follow once the executor + sandbox routing is proven.

**1.3 Detailed changes — authored at rung entry** against a fresh read of
`drive_stream`, the `ProviderEvent`/`ToolResultContent` types
(`crates/runtime-main/src/providers/`), and the capability scope model.
The construction-reachability wire to verify before coding:
`drive_stream` (the caller that today only branches MCP-or-emit) must
gain a built-in-execute branch whose result joins `feedback.dispatched`
— grounded in the actual `TurnFeedback`/`DispatchedTool` shapes
(`agent_sdk.rs:80-97`).

**1.4 Acceptance (BDD — authored before the build).**

```
Feature: A single agent executes a built-in tool
  Scenario: Read feeds file contents back into the agent's turn
    Given a one-agent framework whose agent has Read in allowed_tools
      And a workspace file fixture "hello.txt" containing "ladder-rung-1"
     When I run the framework with task "read hello.txt and quote it"
     Then a ToolInvoked(Read) event is emitted
      And the agent's NEXT turn receives the file contents as a tool_result
      And the agent's final text contains "ladder-rung-1"
```

The last two `Then`s are the grounded-claims load-bearing assertions —
they distinguish "the tool *ran* and *fed back*" from "the event was
emitted." The assembled test runs the real run loop (the
`run_test_session_with` path, `tester.rs:295`); the IRL gate is the
maintainer running the framework and watching the agent quote the file.

**1.5 CLI prompt + 1.6 Commit message** — authored at rung entry per the
cluster-gate schema (red commit `test(M8.7a.1): …`, surface for red
approval, impl commit diff-EMPTY over tests, IRL gate before close).

### Rung 2 — capability enforcement on a live tool

**2.1 Problem (cited).** `EventPipeline` enforcement
(`event_pipeline.rs:102`, `with_enforcement` at `agent_sdk.rs:391`)
already emits `CapabilityViolation` and omits `ToolInvoked` on a blocked
check — but because no built-in tool executed anyway (rung 1), "blocked"
and "allowed" were behaviorally identical (neither ran). Once rung 1
makes allowed tools *run*, rung 2 verifies blocked tools *do not run*.

**2.2 Deliverable.** With rung 1's executor in place: an allowed built-in
executes; a built-in outside the agent's capability scope is blocked
(no execution, `CapabilityViolation` emitted, HITL routed). The gate now
gates a real side effect.

**2.4 Acceptance (BDD).**

```
Feature: Capability enforcement gates real execution
  Scenario: A write outside file_access scope does not run
    Given a one-agent framework whose agent's file_access.write is ["out/**"]
     When I run a task that tries to Write to "secret/x"
     Then a CapabilityViolation event is emitted
      And NO file is created at "secret/x" on disk
      And a HITL prompt surfaces
  Scenario: A write inside scope does run
     When I run a task that writes to "out/x"
     Then the file "out/x" exists on disk with the written content
```

"NO file on disk" / "file exists on disk" are the grounded assertions —
observable side effects, not events.

### Rung 3 — skill load (`LoadSkill`)

**3.1 Problem (cited).** No `LoadSkill` handler exists (grep). A skill in
`allowed_skills` is never injected into the agent's context; the agent
cannot follow skill instructions.

**3.2 Deliverable.** A `LoadSkill` handler that, on the agent invoking it
(or per the framework's load policy — grounded at entry against the
skill model in `crates/runtime-core`), reads the resolved skill markdown
(M08.6's loader already resolves skill companions to bodies) and injects
it into the agent's context for subsequent turns, emitting `SkillLoaded`
(`event.rs:279`).

**3.4 Acceptance (BDD).**

```
Feature: A skill changes agent behavior
  Scenario: Loading a skill makes the agent follow its instruction
    Given a skill "shout.md" whose body says "always reply in ALL CAPS"
      And a one-agent framework with shout in allowed_skills
     When I run a task "say hello" that loads the skill first
     Then a SkillLoaded(shout) event is emitted
      And the agent's reply is in all caps
```

The "reply is in all caps" assertion proves the skill content reached the
model — behavior, not the `SkillLoaded` event alone.

### Rung 4 — gap detection (`request_capability`)

**4.1 Problem (cited).** `handle_request_capability`
(`request_capability.rs:88`) exists but is not called from the run loop;
a model `request_capability` `ToolUse` falls through to
`pipeline.next_event` and is treated as an ordinary (unknown) tool. The
signature gap-detection flow never fires.

**4.2 Deliverable.** Wire `drive_stream` to route a
`request_capability` `ToolUse` into `handle_request_capability`, which
emits the appropriate `*Missing` gap event (`event.rs:479-527`) and
suspends the session cleanly per the gap flow (spec §4) — the project's
signature behavior.

**4.4 Acceptance (BDD).**

```
Feature: A missing capability suspends the session cleanly
  Scenario: request_capability for an unheld tool raises a gap and suspends
    Given a one-agent framework whose agent lacks tool "deploy"
     When the agent calls request_capability(tool, "deploy", "<why>")
     Then a ToolMissing gap event is emitted
      And the session suspends (does not crash, does not silently continue)
      And the suspended state is recoverable
```

"Suspends, recoverable, does not crash/continue" are the grounded
assertions — the observable gap behavior, run.

### Rung 5 — budget

**5.0 Ground at entry (required).** The budget-tracking surface was not
read this session. Per §4 rule 11, before authoring 5.1–5.6 the rung
must read the actual budget path (token accounting, the
`budget.{warn,downshift,hitl,hard_stop}_at_percent` consumers, the
`select_cheaper_model` downshift hook) and state whether budget is
tracked-but-painted or tracked-and-enforced. The detailed spec is
authored against that finding, not assumed.

**5.2 Deliverable (shape).** Token spend drives the framework's budget
thresholds: `warn` → `NotifierDispatched`; `downshift` → the
`select_cheaper_model` hook actually swaps the model; `hard_stop` → the
session stops. Each threshold is an observable behavior, IRL-watched.

**5.4 Acceptance (BDD — shape; finalized after 5.0).**

```
Feature: Budget thresholds change runtime behavior
  Scenario: Hard-stop halts the run
    Given a one-agent framework with a tiny session_usd_cap
     When the run exceeds the cap
     Then the session stops with a budget-stop reason
      And no further provider turns are issued
```

### M8.7a ending

A single agent **executes** built-in tools (rung 1), is **gated** on real
execution (rung 2), **loads skills** that change its behavior (rung 3),
**raises gaps** that suspend cleanly (rung 4), and **respects budget**
(rung 5) — each rung IRL-watched against a minimal framework, each with a
permanent assembled regression test that runs the real path. The five
rung fixtures land in `examples/ladder/` (or `tests/fixtures/ladder/`,
decided at rung 1) as the start of the execution-regression suite.

## M8.7b — multi-agent + plan (rungs 6–7)

> **Structured, not detailed — grounded-reason deferral.** Rung 6's spec
> (per-child execution context: reuse `drive_stream` as-is, or a
> re-entrant variant; how a child's result returns to the parent's turn;
> how budget sub-allocates) depends on what M8.7a's executor + feedback
> wiring actually look like. Rung 7 (wire `drive_plan` + build the task
> loop) depends on rung 6's child-execution primitive. Authored after
> M8.7a lands; the construction-reachability wires are traced then,
> against real code, not assumed now.

- **Rung 6 — sequential spawn ⭐.** Build the per-agent execution context
  so `spawn_framework_subagents` (today emit-only, `agent_sdk.rs:467`)
  becomes: root runs → spawns child with narrowed grants → child runs its
  own loop → returns a summary to the parent → parent continues
  (sequential; concurrent is v1.0). The validated subagent contract
  (isolated context, invoked-as-tool, returns a summary to the parent who
  holds memory) is the target shape. **BDD shape:** load an
  orchestrator+1-child framework → run → the child's own work is
  observable in the live graph (events with the child's `agent_id`, not
  just an `AgentSpawned`-then-nothing) → the orchestrator continues after
  the child returns.
- **Rung 7 — plan-driven tasks.** Wire `drive_plan` (`plan_loop.rs:79`,
  currently no caller) into a production session entrypoint and build the
  task-execution loop it currently lacks (`plan_loop.rs:128-129`).
  **BDD shape:** a planner framework → plan created → HITL approve →
  tasks execute (each `TaskStarted`/`TaskCompleted` observable) →
  `PlanComplete`.

**M8.7b ending:** a multi-agent framework runs sequentially and a plan
drives real task execution — both IRL-watched.

## M8.7c — hooks/rails, MCP, ARIA (rungs 8–9, N)

> **Structured, not detailed — grounded-reason deferral.** Rung 8's

- **Rung 8 — hooks + rails.** Build the firing engine for

- **Rung N — ARIA (the opus).** Import the real `examples/aria/`, **fix
  its shape** against the engine (M08.6 #31: declare the orchestrator's
  grantable union so strict narrowing passes; reconcile the
  built-in-vs-framework tool set, #28), and **run the v0.1-supported

  skills, gaps, budget, hooks/rails. **Document the v1.0 line**
  explicitly (concurrent orchestration, the grant/use capability split,
  the mode router) so what does *not* run is stated, not implied (§4
  rule 11). **BDD shape:** load ARIA → Run → watch ARIA's agents
  actually execute the subset in the live graph.

**M8.7c ending — the milestone ending:** ARIA runs. The runtime is a
real running agentic runtime for the v0.1 scope, with a permanent
ladder regression suite proving each function executes and a documented
v1.0 line for what doesn't.
