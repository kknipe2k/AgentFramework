## M06.6.1 Problem Statement

Five recurring authoring-time gaps, each with its M06 source:

1. **Construction-graph reachability not checked** (M06.D special-log claim grep-false; M06.F over-framing — `M06.F-retrospective.md:31,77,116,121`). `<phase_doc_inventory_audit verified="true">` confirmed symbols exist but not that an "inject X into Y" claim has its constructor inputs *reachable in the assembled composition*. M06.F's "MCP dispatch end-to-end in the running app" mandate over-reached: no `impl ConnectionResolver for McpClient`, no shell ctor inputs, the only `AgentSdk` construction is the no-tools smoke path. Generalizes the M06.D `<dependency_cycle_check>` carry-forward.
2. **Wire signature / store-slot shape drift** (M06.E — `M06.E-retrospective.md:118,183`). Phase-doc renderer/IPC pseudocode drifted from the shipped wire: `currentMcpServers` modeled `Map`+`.tools`, actual `Record<…,McpServerStatusRecord>`; `mcp_test_connection {config}` vs actual `{name}`. Five such drifts at E.
3. **Wire trace authored against an ADR-superseded architecture** (M06.V Decision 6 — `M06.V-retrospective.md:76`). V.3 trace #6 expected `McpClient` to drive re-resolution; ADR-0010 (accepted, read after the trace was authored) moved the resolver into `McpDispatcher`. The inconsistency should have been caught at phase-doc-authoring, not at V.
4. **Verifier re-derives the "delivered+tested, driver-absent, root = accepted-ADR carry-forward" ruling each milestone** (M06.V Decision 6 second half). V handled trace #11/#6 correctly by reasoning from ADR-0011, but the rule (treat as 🟡-with-mandatory-carry-forward-enumeration, not 🔴, not silent) is not codified in the verifier template — it is re-derived every milestone.
5. **Mock-only Behavior pass cannot rule out rmcp wire-format correctness** (M06.V Decision 7 — `M06.V-retrospective.md:77`). The `transport/stdio.rs`+`http.rs` excluded holdout is attributed to the feature-gated `tests/integration.rs` reference-MCP-server smoke, which ran 0/0 at M06.V. By M07 a real dispatch path exists; the integration smoke must become a required Behavior-pass input from M07.V onward.

Plus the **M06.5 recorded input** (M06.5-summary.md:156-173): the assembled-app-regression mandate caught not only incomplete *code* (Stage-V's blind spot) but an incomplete *phase-doc root-cause diagnosis* (B.fix's FK/session-id second condition). Encode: "the phase-doc root cause is a falsifiable hypothesis the assembled regression test must disprove, not a premise" + the binary-crate seam-test variant of the strict-TDD invariant (M06.5.A.fix in-source `#[cfg(test)]` byte-identical-block precedent when `**/tests/**` is vacuous).

## M06.6.3 Deliverables

### A — `<construction_reachability_check>` slot (NEW)

For every phase-doc deliverable phrased as "wire / inject / construct X into/through Y", the author traces the *construction graph* at authoring time: does Y's constructor (in the assembled composition the phase doc targets) actually receive X, and are X's ctor inputs reachable at that call site? Not "the symbol exists" — "the symbol is constructible where the deliverable claims it is wired." Generalizes the M06.D `<dependency_cycle_check>` carry-forward; would have caught M06.F's "no shell ctor inputs for the concrete dispatcher" at authoring.

```xml
<construction_reachability_check>
  <wire claim="inject Arc<dyn McpToolDispatch> into AgentSdk run loop"
        constructor="AgentSdk::new (src-tauri/src/commands.rs:NNN)"
        inputs_reachable="false — no shell McpDispatcher ctor; only the no-tools smoke path constructs AgentSdk"
        resolution="scope to M07 / descope via <scope_change>"/>
</construction_reachability_check>
```

Children: `<wire>` with `claim` / `constructor` / `inputs_reachable` / `resolution`. Optional; work-stage only. Lean-validator: structural pass-through, honor-system.

### B — `<wire_signature_audit>` slot (NEW) + `<phase_doc_inventory_audit>` `shape=` extension

Before authoring any renderer pseudocode or IPC-wrapper call, pin the wrapper to the **actual** Tauri command params and the store slot to its **actual** TS shape. The `<phase_doc_inventory_audit>` `<claim>` gains an optional `shape="…"` attribute for store-slot claims (verifies shape, not just symbol). Would have caught M06.E's `Map`-vs-`Record` and `{config}`-vs-`{name}` drifts.

```xml
<wire_signature_audit>
  <wrapper ipc_command="mcp_test_connection" actual_params="{ name: string }" phase_doc_assumed="{ config }"/>
</wire_signature_audit>
```
```xml
<claim type="store_slot" path="src/state/graphStore.ts" symbol="currentMcpServers"
       shape="Record<string, McpServerStatusRecord>" verified="true"/>
```

Children: `<wrapper>` with `ipc_command` / `actual_params` / `phase_doc_assumed`. Optional; work-stage only. Lean pass-through.

### C — `<wire_trace_vs_adr_reconcile>` slot (NEW)

At phase-doc-authoring time, every Wire trace is reconciled against accepted ADRs: does any accepted ADR supersede the architecture the trace assumes? Would have caught M06.V trace #6 (authored assuming `McpClient` drives re-resolution; ADR-0010 moved it to `McpDispatcher`).

```xml
<wire_trace_vs_adr_reconcile>
  <trace id="6" assumes="McpClient drives namespace re-resolution"
         adr_checked="ADR-0010" superseded="true — resolver moved to McpDispatcher"
         resolution="rewrite trace #6 against McpDispatcher before authoring"/>
</wire_trace_vs_adr_reconcile>
```

Children: `<trace>` with `id` / `assumes` / `adr_checked` / `superseded` / `resolution`. Optional; work-stage AND verifier-relevant (V reads it). Lean pass-through.
