>
> **Why this milestone exists.** The 2026-06-09 maintainer-commissioned external review (`docs/code-design-review-2026-06-09.md`, merged PR #131) line-verified eight Criticals — **TD-050…TD-057** — that cluster in exactly the blind spot of an IRL-driven verification culture: *adversarial inputs, trust boundaries, and network failure modes*. >
> **The defining authoring rule of this cluster (maintainer-directed):** every stage's assembled regression encodes the **ADVERSARIAL case, not the happy path** — a traversal that must be refused, a symlink escape the capability check must now reject, an oversized no-newline frame that must be rejected, an idle-stalled stream that must surface cleanly. The hostile case IS the acceptance test (the red commit), because the finding class is precisely "the happy path was the only thing ever exercised."

## Background — the grounded state at entry

**What stands (rule 11).** M09 merged: the workbench authors and runs a real single-agent MCP workflow from scratch at the enforced tier (maintainer real-app IRL 2026-06-09; execution-status row flipped). The capability model's *internal* layers are strong — the review verified the L1/L4 enforcer ordering, the proptest-verified narrowing asymmetry, SSRF egress defense, end-to-end secret hygiene. Nothing in this cluster redesigns any of that.

**What is open (the review's line-verified perimeter gaps — re-verified on post-M09 `main` 2026-06-10):**

- **TD-050 — the WebView ships with no CSP, and all four Zustand stores ride `window` unconditionally.** `src-tauri/tauri.conf.json:23` → `"csp": null`; `src/App.tsx` assigns `window.__graphStore` / `__builderStore` / `__testGraphStore` / `__toastStore` at module load with no gate (the comment block at `App.tsx:~43-75` documents the exposure as intentional). Any script that reaches the WebView runs unimpeded and gets a typed write path into runtime state — including forging `AgentEvent`s the `graphStore.applyEvent` reducer trusts. **Load-bearing discovery (2026-06-10 seam audit):** the e2e harnesses *depend* on this exposure — 12 `tests/e2e-tauri/*.e2e.ts` files and 9 `tests/e2e/*.spec.ts` files drive the app through `window.__*Store`, and `e2e-tauri` drives the **built** (production) binary. A naive `import.meta.env.DEV` gate breaks the merge-blocking `e2e-tauri-driver` job. Stage A therefore ships a **shell-resolved test-mode seam**, not a bare DEV gate.
- **TD-051 — renderer-supplied filesystem paths reach `fs` unconfined.** `src-tauri/src/commands.rs:1611` `save_framework(dir: String, …)` / `:1640` `load_framework(dir: String)` pass `Path::new(&dir)` straight through; `:1433` `import_artifact` with `source_kind == "file"` passes `location` to `ImportSource::File` (`:1451`). The OS file dialog (`dialog:allow-open`, `capabilities/default.json`) is a UX affordance, not an enforcement boundary — a direct `invoke("save_framework", {dir: "../../anywhere"})` is accepted. Compounds with TD-050 (script injection → arbitrary file write).
- **TD-052 — built-in `Read`/`Write` never canonicalize the model-supplied path.** `crates/runtime-main/src/sdk/builtin_tools.rs:98` (`fs::read_to_string(path)`) / `:119` (`fs::write(path, content)`) use the raw LLM-supplied string for **both** the capability scope check (`file_decl` `:131` → `PathPattern::from_str(path)`) **and** the IO. With grant `read: workdir/**`, a symlink `workdir/link → /etc/passwd` passes the glob and reads outside the granted scope; unnormalized `..` segments are the simpler variant. This bypasses the product's central promise on the two tools that already `execute — observed` (E-01/E-02). ADR-0028 freezes the contract this stage operates inside (`file_access` scope is the boundary; dispatch→feedback shape unchanged).
- **TD-053 — unbounded `LinesCodec` on all three IPC layers; Unix sockets unrestricted; Windows pipe DACL claim untested.** 13 sites use `LinesCodec::new()` (max length `usize::MAX`): `runtime-main/src/sandbox_ipc/connection.rs:108,109,203,204`, `runtime-main/src/drone_ipc/connection.rs:102,103,236,237`, `runtime-drone/src/ipc.rs:144,175`, `runtime-sandbox/src/ipc.rs:103,104,439`. A peer writing bytes with no newline buffers unbounded memory (OOM-DoS); the `MaxLineLengthExceeded` error arms exist in the `From` impls (`drone_ipc/connection.rs:57`, `sandbox_ipc/connection.rs:63`, `runtime-sandbox/src/ipc.rs:164`) and are **unreachable** today. `runtime-drone/src/ipc.rs:82` and `runtime-sandbox/src/ipc.rs:50` bind Unix sockets with no `set_permissions` (umask-dependent). The Windows named pipes (`runtime-drone/src/ipc.rs:101-110`, `runtime-sandbox/src/ipc.rs:63-72`) rely on `ServerOptions::new()` defaults with an in-code claim ("defaults to 'owner only'", `runtime-drone/src/ipc.rs:23`) that has never been verified or tested — and v0.1 is Windows-first, so the pipe is the production surface.
- **TD-054 — no timeout anywhere on the provider HTTP/SSE path; no retry on 429/500/529.** `crates/runtime-main/src/providers/anthropic.rs:53-58` builds the client with only `.pool_max_idle_per_host(2)`; the run loop awaits `stream.next()` with no timeout (`crates/runtime-main/src/sdk/agent_sdk.rs:561`). A stalled connection parks the session forever — a hang, in a product whose headline is "suspends cleanly." `ProviderError::Timeout(Duration)` **already exists** (`providers/mod.rs`) and nothing produces it; 429 maps to `RateLimit{retry_after}` but is never retried; 500/529 fall into generic `Api{status,body}` with no backoff.
- **TD-055 — seccomp allows `AF_INET`/`AF_INET6` socket creation (no argument filter).** `crates/runtime-sandbox/src/seccomp.rs:103-117` allowlists `socket`, `bind`, `sendto`, … unconditionally via `ScmpFilterContext::add_rule` (`:172`); the comment at `:103` says "Unix domain sockets for IPC" but nothing enforces the family. `connect` is absent (good), but `bind`+`sendto` UDP exfiltration is reachable, and landlock does not cover network. Maintainer decision 2026-06-10: land the filter **now** as cheap defense-in-depth rather than waiting for M12. Note the mechanics: only `socket(2)`'s arg0 (domain) is BPF-filterable — `bind`/`sendto` take an fd plus a pointer BPF cannot dereference — so the correct choke point is a **conditional rule on `socket` (arg0 == AF_UNIX)**, which makes inet fds uncreatable and the rest moot.
- **TD-056 — CLAUDE.md §5 claims a nightly `cargo-mutants` gate that does not exist in CI.** `.github/workflows/` has `ci.yml`, `fuzz-nightly.yml`, `release.yml`; zero `mutants` references repo-wide. Maintainer decision 2026-06-10: **implement the workflow** (make the claim true), landing with Stage A.

**Already closed before this cluster (do not re-do):** TD-057's threat-model statement + the in-code SECURITY note (PR #133); ADR-0032/0033 flips + the spec §0/§0a/§0d reconciliation + the v1.12 adversarial V pass (PR #133); the M09-vs-review TD numbering collision (PR #132 — M09's findings are TD-058…063).

## Stage M09.5.A — Shell perimeter: CSP, store-exposure seam, path confinement (+ the mutants workflow)

### A.1 Problem statement
Three renderer-perimeter gaps and one process-integrity gap, all shell-side. (1) `tauri.conf.json:23` ships `"csp": null` — no Content-Security-Policy at all; Tauri 2 hardening guidance treats a restrictive CSP as the baseline, and M10+ renders progressively more model-generated content. (2) `src/App.tsx` exposes all four Zustand stores on `window` unconditionally — a typed write path into runtime state for any injected script — and **21 e2e files depend on that exposure**, 12 of them driving the *built production binary* via `tauri-driver`, so the gate must be a test-mode seam the production binary honors only when the harness asks. (3) `save_framework`/`load_framework`/`import_artifact(file)` accept raw renderer-supplied paths (`commands.rs:1611/1640/1433-1451`) with no containment — `invoke("save_framework", {dir:"../../x"})` is accepted today. (4) CLAUDE.md §5 claims a nightly cargo-mutants run that does not exist (TD-056); the maintainer chose to make the claim true.

### A.2 Files to change
**`<wire_signature_audit>` — pin before pseudocode:** the `security` object shape in `tauri.conf.json` (Tauri 2.x `app.security.csp` — verify the key path against the installed `tauri.conf.json` schema, not from memory); the `window.__*Store` assignment block + its TS `declare global` block (`App.tsx`); how `wdio.conf.ts` launches the binary (does `tauri:options` carry `env`? if not, the spawned child inherits the runner's env — verify by reading the tauri-driver docs + the conf); `save_framework_with(Path, …)`/`load_framework_with(Path)`/`import_artifact` signatures (`commands.rs:1611/1640/1433`); `tauri_plugin_dialog` Rust-side API (`DialogExt::dialog().file().pick_folder`) — the M08 plugin is already initialized (`src-tauri/src/main.rs`); `fuzz-nightly.yml` trigger shape (the mirror).
- `src-tauri/tauri.conf.json` — the CSP value.
- `src/App.tsx` — the store-exposure gate.
- `src-tauri/src/main.rs` + `src-tauri/src/commands.rs` — the test-mode flag injection; the `pick_framework_dir` command; containment on the three path commands.
- `crates/runtime-main/src/` (new small module, e.g. `path_confine.rs`) — the pure containment helper (canonicalize + allowlist-prefix check), unit-testable with `tempfile`, no Tauri dependency (the path-agnostic persistence archetype, `docs/style.md`).

### A.3 Detailed changes
1. **CSP.** Set `app.security.csp` to a restrictive policy: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src ipc: http://ipc.localhost` — **authored as a starting point, verified against the running app, not trusted from memory** (gotcha #32 class): boot the built app and confirm (a) the bundled IBM Plex `@font-face` loads, (b) React Flow renders (it sets inline style attributes — covered by `style-src 'unsafe-inline'`; if the canvas breaks, check `style-src-attr`), (c) Tauri IPC works on Windows (`http://ipc.localhost`) and Linux (`ipc:`). No `unsafe-inline` for scripts under any outcome. The e2e-tauri suite green on both OSes is the machine check; the CSP-violation assertion is the adversarial check (an injected inline `<script>` does not execute — assert via a DOM-injected script element whose side effect must NOT appear).
2. **Store-exposure seam (shell-resolved test mode).** `src-tauri/src/main.rs` reads `AGENT_RUNTIME_E2E` at startup; when `=1`, the window is built with an initialization script setting `window.__E2E__ = true` (shell-resolves-the-decision archetype — the renderer never reads the env itself). `App.tsx` gates the four assignments on `import.meta.env.DEV || (window as …).__E2E__`. `wdio.conf.ts` exports `AGENT_RUNTIME_E2E=1` into the app's environment (tauri-driver spawns the binary as a child — it inherits the runner env; verify and pin in the wire audit). Playwright runs against the Vite dev server (`import.meta.env.DEV` true) — unaffected. **The adversarial assertion:** launch the built binary WITHOUT the env (one e2e spec spawns it bare or asserts via a dedicated wdio capability run) and assert `window.__graphStore === undefined`; with the env, the existing 12 specs keep passing unchanged. If per-spec env control proves impossible under tauri-driver, the no-env assertion moves to a Rust-side integration test of the initialization-script branch + a documented manual IRL check — state which form landed (rule 11).
3. **Path confinement.** New pure helper in `runtime-main` (e.g. `path_confine::confine(path: &Path, roots: &[PathBuf]) -> Result<PathBuf, ConfineError>`): canonicalize (for save: canonicalize the nearest existing ancestor + re-join the remainder, since the target dir may not exist yet), reject any result not strictly under one of `roots`. Shell side: a new `pick_framework_dir` command opens the **Rust-side** folder dialog (`tauri_plugin_dialog`), registers the picked dir in managed session state (`HashSet<PathBuf>`), and returns it; `save_framework`/`load_framework`/`import_artifact(file)` confine their incoming path against `{registered dirs} ∪ {app_local_data_dir}` before any IO and return a typed `CmdError` (kind `path_not_permitted`) otherwise. The renderer's existing dialog flow migrates to `pick_framework_dir` (the JS-side `dialog:allow-open` usage for THESE flows retires; the capability stays for unrelated pickers if any — check usages). UX is unchanged for the user (same dialog, same save); the only new behavior is that a path the dialog never returned is refused.
4. **`mutants-nightly.yml`** (separate `ci(M09.5.A)` commit): mirror `fuzz-nightly.yml` — `schedule: cron '30 3 * * *'` + `workflow_dispatch`; ubuntu; install `cargo-mutants --locked`; run scoped to the §6 critical modules (`cargo mutants -p runtime-drone -p runtime-sandbox -p runtime-mcp -p runtime-main --timeout-multiplier 3` or the current flag set per `cargo-mutants --help` — verify flags against the installed version, not memory); `continue-on-error: false` on the job but the workflow is nightly-on-main only (advisory posture per §5 — it is not a PR gate); upload `mutants.out` as an artifact on completion. CLAUDE.md §5's claim becomes true; no CLAUDE.md edit needed.

### A.5 construction_reachability_check
`inputs_reachable="true"` — and that is the defect: `save_framework` is reachable with ANY `dir` (`commands.rs:1611`), the stores are reachable on `window` in every build, inline script execution is reachable under `csp: null`. A.3 inverts reachability for the hostile inputs only: traversal → typed refusal; bare-launch stores → absent; inline script → inert. Verify by grep post-impl: `path_not_permitted` arm exists and is produced; `__E2E__` gate present in `App.tsx`; `"csp":` non-null in `tauri.conf.json`; the e2e spec names match A.2.

## Stage M09.5.B — Built-in Read/Write: canonicalize before the check and the IO (TD-052)

### B.1 Problem statement
`execute_builtin` (`crates/runtime-main/src/sdk/builtin_tools.rs:80`) extracts the model-supplied path string and uses it verbatim twice: `file_decl` (`:131`) builds the `CapabilityDeclaration` whose `PathPattern` the enforcer checks, and the IO runs on the same raw string (`fs::read_to_string` `:98`; `fs::write` `:119`). Nothing canonicalizes. Two bypasses of the L1 scope gate follow: a `..` traversal (`workdir/../etc/passwd` matches nothing it should) and the symlink escape (grant `read: workdir/**`; `workdir/link → /etc/passwd`; the *string* matches the glob, the *IO* leaves the scope). This is a real hole in the product's central promise on the two built-ins that already `execute — observed` (E-01/E-02), and it operates entirely inside the ADR-0028 contract (the `file_access` scope IS the boundary — this stage makes the boundary honest; it does not move it).

### B.2 Files to change
**`<wire_signature_audit>` — pin before pseudocode:** `execute_builtin(…)`'s exact signature + the `BuiltinExecError::{Capability, Op}` split (`builtin_tools.rs:56-67`); `file_decl(…) -> CapabilityDeclaration` (`:131`) + `PathPattern::from_str` semantics (separator handling — Windows `\` vs glob `/`; pin how E-02 grants are written today); `enforcer.check(agent_id, &decl)` (`capability/enforcer.rs:198+` — L4-before-L1 order, untouched); the E-02 test fixtures (`capability_live_tool.rs` — `one_agent_fw`, `fwd()` path helper, `WriteToolStub`) so the new adversarial cases reuse them; `std::fs::canonicalize` behavior on Windows (`\\?\` verbatim prefix — the canonicalized GRANT root must compare against the canonicalized TARGET, never a raw glob against a `\\?\` path).
- `crates/runtime-main/src/sdk/builtin_tools.rs` — the canonicalize-then-check-then-IO restructure.
- `crates/runtime-main/tests/capability_live_tool.rs` — the E-02 adversarial extension (the maintainer-directed home for these cases).

### B.3 Detailed changes
1. **Resolve once, use everywhere.** Before the capability check: resolve the model-supplied string to a canonical absolute path. For `Read`: `std::fs::canonicalize(path)` (target must exist — a missing file stays an `Op` error as today). For `Write`: canonicalize the **nearest existing ancestor** and re-join the non-existing remainder (a written file often doesn't exist yet); reject remainders containing `..` after normalization. The canonical path feeds BOTH `file_decl` AND the IO — one resolution, no TOCTOU between check and use at the string level (document the residual OS-level race honestly: we do not hold a handle across check→use in v0.1).
2. **Symlink policy (explicit, documented in-code):** resolve-then-check. A symlink anywhere in the path is followed (that is what canonicalize does) and the **resolved target** must be in scope: a link inside the grant pointing outside it is therefore **denied** (the escape the review proved); a link pointing inside the grant is allowed. The policy lives as a doc comment on the resolver with the TD-052 cite.
3. **Scope-check comparison on canonical forms.** The grant globs (relative or absolute as authored) are anchored the way E-02 writes them today — pin this in the audit and keep the user-visible grant semantics IDENTICAL for in-scope paths: every currently-passing E-02 case must still pass, byte-for-byte the same grants. The change is strictly that escapes now fail. Windows: compare canonicalized-to-canonicalized (both sides through the same normalization) so the `\\?\` prefix and separators never poison the glob match.
4. **Error surface unchanged in shape:** an out-of-scope resolved path produces the same `BuiltinExecError::Capability` → `CapabilityViolation` event + no side effect, exactly as the existing denial arm (E-02 asserts no-file-on-disk; the new cases assert the same).
5. **E-02 adversarial extension (in `capability_live_tool.rs`, per the maintainer's directive):** (a) `..`-traversal Write against a scoped grant → `CapabilityViolation`, **no file anywhere** (assert both the in-scope dir and the escape target absent); (b) symlink-escape Read (`#[cfg(unix)]` — create `granted/link → outside/secret.txt`, grant `granted/**`) → denied, content never returned; (c) symlink-escape Write `#[cfg(unix)]` → denied, target untouched; (d) in-scope relative-with-`..`-that-still-resolves-inside → **allowed** (proves the fix narrows to escapes, not to harmless normalization); (e) the Windows traversal variant (`..\` form, no symlink). Windows junction coverage is OPTIONAL — if junction creation needs elevation on the runner, log the holdout honestly (a 🟡-class note in the retro, not a silent skip).

## Stage M09.5.C — IPC hardening: frame caps, socket modes, the Windows pipe DACL decision (TD-053)

### C.1 Problem statement
All three IPC layers decode with `LinesCodec::new()` — default max length `usize::MAX` — at 13 sites (`runtime-main/src/sandbox_ipc/connection.rs:108,109,203,204`; `runtime-main/src/drone_ipc/connection.rs:102,103,236,237`; `runtime-drone/src/ipc.rs:144,175`; `runtime-sandbox/src/ipc.rs:103,104,439`). The `MaxLineLengthExceeded` error arms already exist in the `From` impls and are unreachable. ### C.3 Detailed changes

### D.1 Problem statement

### D.3 Detailed changes
1. **Client:** `.connect_timeout(Duration::from_secs(10))` on the builder. NO global `.timeout()` — a total-request deadline would kill legitimately long streams; the idle timeout is the right instrument.
2. **Idle timeout at the stream:** wrap the provider's event stream so each `next()` is raced against `IDLE_TIMEOUT` (90s, a named const with the TD-054 cite; generous — model thinking gaps + `ping` events keep healthy streams well under it). On expiry: yield `Err(ProviderError::Timeout(IDLE_TIMEOUT))` and terminate the stream. The adapter lives in `providers/anthropic.rs` (or a small shared `providers/stream_guard.rs` if cleaner) — `agent_sdk.rs:561` is untouched; the loop's existing `Err` handling produces the clean suspend. **Assert the suspend observable** (session ends with the typed error recorded, no further turns, no hang) in an assembled run-loop test with a stalling fake — this is the "suspends cleanly, even against the network" eval.
3. **Typed overload + retry:** map HTTP 529 and `error.type == "overloaded_error"` to `Overloaded`; retry policy at the request boundary: attempt → on 429/500/502/503/529, sleep `min(base * 2^n + jitter, cap)` honoring `retry-after` (max of the two when both present) → re-attempt; 3 attempts total then surface the typed error. 4xx (other than 429) and auth errors never retry. Each retry emits a `tracing::warn!` (observability without telemetry).
4. **No behavior change on success paths:** identical request/stream semantics when the service is healthy — the full existing provider test suite green unmodified is the regression assertion.

The seccomp allowlist (`crates/runtime-sandbox/src/seccomp.rs:103-117`) permits `socket`, `bind`, `sendto`, `recvfrom`, … unconditionally (`add_rule(ScmpAction::Allow, syscall)` `:172`); the comment at `:103` declares "Unix domain sockets for IPC" but nothing enforces the family. `connect` is absent (no client TCP), but `bind` + `sendto` on an `AF_INET SOCK_DGRAM` socket is a reachable UDP exfiltration channel, and landlock does not cover network. Today no untrusted code runs inside the fence (validate-only sandbox), so this is defense-in-depth — but it becomes a hard blocker the moment M12 lands `SandboxRequest::Execute`, and the maintainer chose to land it now (2026-06-10). Mechanics constraint: only `socket(2)`'s arg0 (the domain) is BPF-filterable — `bind`/`sendto` receive an fd plus a pointer BPF cannot dereference — so the correct and sufficient choke point is a **conditional allow on `socket` with `arg0 == AF_UNIX`**, which makes inet fds uncreatable inside the fence.

### E.2 Files to change
**`<wire_signature_audit>` — pin before pseudocode:** the allowlist construction (`seccomp.rs:169-173` — `ScmpSyscall::from_name` + `add_rule`) and the filter's default action (allowlist-with-default-deny? read the `ScmpFilterContext::new_filter` call — the conditional rule's semantics depend on it); the `libseccomp` crate's conditional API on the pinned version (`add_rule_conditional` + `ScmpArgCompare`/`scmp_cmp!` — verify against docs.rs for the lockfile version); `AF_UNIX`'s numeric value source (`libc::AF_UNIX` — is `libc` already a direct dep of runtime-sandbox? pin it); the existing test structure (`:203-282` — `allowlist_excludes_dangerous_syscalls`, the subprocess/OS-signal-class holdout pattern in `docs/coverage-policy.md` for tests that install real fences); where `clone` vs `clone3` stand in the forbidden assertion today (`:225+` — the review notes `clone3` is asserted, `clone` is not).
- `crates/runtime-sandbox/src/seccomp.rs` — `socket` moves from the unconditional allowlist to a conditional rule (`arg0 == AF_UNIX`); a comment explaining the choke-point reasoning (bind/sendto unfilterable by family — the socket-creation gate suffices); `clone` added to the forbidden-syscall test assertion.
- A subprocess behavioral test (new or extending the existing isolation integration tests): inside the installed fence, `socket(AF_INET, SOCK_DGRAM, 0)` fails with `EPERM` (or the filter's kill/errno action — pin which) while `socket(AF_UNIX, SOCK_STREAM, 0)` succeeds. Linux-only (`#[cfg(target_os = "linux")]` — remember gotcha #74: local non-Linux `cargo check` does not compile this branch; CI Linux is the proof).

### E.3 Detailed changes
1. **The conditional rule.** Remove `"socket"` from the unconditional loop; add `ctx.add_rule_conditional(ScmpAction::Allow, socket_syscall, &[ScmpArgCompare::new(0, ScmpCompareOp::Equal, libc::AF_UNIX as u64)])` (exact API per the pinned libseccomp version). With the filter's default-deny posture, a non-AF_UNIX `socket` call then receives the filter's deny action — pin which (`Errno(EPERM)` vs kill) from the existing `new_filter` default and assert that exact observable. `AF_UNSPEC`/`AF_NETLINK` etc. are NOT allowed — if tokio's runtime inside the sandbox needs another family (it should not for UDS-only IPC), the subprocess test will say so loudly; widen ONLY with the family named + justified in the plan (surfaced, Hard Rule 8).
2. **Forbidden-list completeness:** add `clone` next to `clone3` in `allowlist_excludes_dangerous_syscalls` (documentation completeness per the review; `fork`/`vfork` already asserted).
3. **The behavioral subprocess test** (the adversarial acceptance): a helper binary/child process installs the full isolation (the existing install path — landlock-before-seccomp order untouched), then attempts `socket(AF_INET, SOCK_DGRAM, 0)` → asserts the deny observable; attempts `socket(AF_UNIX, SOCK_STREAM, 0)` → succeeds; exits with a code the parent asserts. This is the OS-signal-class holdout pattern (the fence poisons the test runner if installed in-process) — follow the existing subprocess-test convention in the sandbox crate, and check `docs/coverage-policy.md` §A: the existing `seccomp.rs`/`landlock.rs` exclusions stand (the rule-construction units ARE in-process-testable and stay covered; only the install-the-fence body is the holdout). Expected: no exclusion change.
4. **Unsafe discipline:** `libseccomp`'s safe API throughout — expected ZERO new `unsafe`. If any arises, it lands only in `runtime-sandbox` with a `// SAFETY:` naming invariants (Hard Rule 7's one permitted home), and is called out in the surface.
5. **Untouched:** the `!Execute` landlock assertion (`landlock.rs:198`) and the `SandboxRequest` protocol — M12's H-ladder owns flipping those; this stage must not pre-flip anything.
