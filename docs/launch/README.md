# Launch communications

Drafts of the public-facing posts for Stage 1 / Stage 2 / Stage 3 of the launch arc per the playbook in the project history.

**Authenticity guardrails (apply to all posts):**

- No "revolutionary" / "game-changing" / "next-gen." Show, don't tell.
- No emojis (🚀, 🔥, ⚡). The work speaks for itself.
- No "Excited to share..." / "Thrilled to announce..." formulas.
- No tagging large accounts (Anthropic, Karpathy, etc.) for amplification.
- No marketing-tone language. Engineers read posts to find out if it works, not to be greeted.
- AI-assisted development disclosed openly when relevant.
- Prior art credited explicitly.
- Limitations stated in the same breath as features.
- Nothing posted unless there's a concrete artifact (code, demo, repo).
- Single-take video where possible; sped-up footage is a credibility-killer.

**The arc:**

1. **Stage 0 — Silence.** No public posts during early build. Resist the urge to announce.
2. **Stage 1 — First public signal.** When the live-graph + gap-flow demo works end-to-end on the maintainer's machine. Single post + 15–30s screen recording. Establishes existence; doesn't promise dates.
3. **Stage 2 — Build-in-progress.** Every 2–3 weeks at concrete milestones (drone resilient → graph live → workbench loads → first generated skill installed → end-to-end demo). One milestone per post; no "working on X" with no artifact.
4. **Stage 3 — Ship.** v0.1.0 release day. Single post linking the repo + demo video + honest scope.
5. **Stage 4 — Sustained.** Post on milestones, not days. Quality > frequency.

Each draft below is a starting point. Edit before posting; what works in one stage may not fit your voice or the moment.
