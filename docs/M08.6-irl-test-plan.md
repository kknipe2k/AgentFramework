# M08.6 IRL Test Plan — Windows Runbook (cumulative M01–M08.6, archetype-fidelity focus)

Manual verification of the running app as of the M08.6 merge. Default shell is **PowerShell**. Every fenced block is paste-as-is; `[DevTools JS]` = paste into the app's F12 → Console; everything else is PowerShell. No mocks — real app, real backend, real renderer. M08.6's central new focus is **archetype fidelity**: the Builder must load + render + persist the canonical-modular frameworks (`examples/aria/`, `examples/ralph/`), not just the M08-era inline fixtures. Cumulative against M01–M08.6; M08.5 + M08.5.5 IRL carry-forwards are folded into Scenarios A/D/E. Findings feed M09 Stage A.

## 0. Do NOT log these as bugs (scope fence)

Known and carried; seeing them is expected, not a finding:

- **The Tester's VDR panel shows `null`.** `TestOutcome.vdr` is structurally dead (M08.V 🟡 #1); carried to M09.A.
- **You cannot delete a node from the canvas.** `removeNode` is a no-op stub (TD-020).
- **A JSON-tab edit or a Load leaves red badges stale** until you click Validate (TD-019) — continuous validation only re-fires on canvas edits.
- **No Task node in the Palette** → a Hook→Task edge is not canvas-drawable in v0.1.
- **The plan-loop driver never fires on its own** (M08.V 🟡 #2 — `drive_plan` shipped tested but unwired).
- **A saved `framework.json` now has alphabetical-by-key field ordering** (was struct-definition order pre-M08.6). Deliberate M08.6 Stage C change: byte-stable save→load→save (MVP §M8 criterion 8) requires canonicalization through `serde_json::Value` because the typify-generated `Framework` carries `HashMap` fields whose direct serialization is order-randomized.
- **The Inspector capability summary populates on a loaded ARIA, but no e2e-tauri assertion guards it** (TD-029). The structural fix landed at Stage B (the `inline_agents` filter now sees all 8 resolved agents); Scenario F visually verifies it.

**Still actively re-checking — log if observed:**

- **M08.5-IRL 🔴 #4** (tier UI/backend desync — UI shows Novice, backend treats as Promoted; the MCP modal gated on Promoted opens regardless). Routed to M09 but a sanity check in Scenario E catches recurrence.
- **M08.5-IRL 🔴 #5** (Builder Canvas state not persisted across app restart). Stage A's static evidence supports interpretation (a) — user-save-discipline UX, not a save-then-restore bug. Scenario G7's restart walk confirms; if save-then-restore is broken, route the fix to a Stage C-adjacent surface.
- **M06.5-IRL 🔴-1** (MCP-registry mis-resolves to a stray `mcp.sqlite`). M08.5.5.C.fix shipped the cleanup module that renames any pre-existing stray DB to `.stray-mcp.sqlite.bak` at app startup. Scenario E re-confirms: an MCP server Add lands in `session.sqlite::mcp_servers` only, with no new stray file created.

## 1. Where each thing runs

| Context | What |
|---|---|
| **Window A** (PowerShell) | builds + runs the app (`npm run tauri dev`); stays occupied (never returns) |
| **Window B** (PowerShell) | every verify command; `$APPDATA_DIR` / `$REG` / `$SCRATCH` / `$ARIA` / `$RALPH` persist here — keep it open |
| **App** | manual drag / click / type + visual inspection in the running window |
| **App DevTools** (F12 → Console) | a few `[DevTools JS]` checks (Scenario F path-ref resolution verification) |

Dev mode (`npm run tauri dev`) is sufficient for most scenarios. Scenario E (MCP Add on Windows) is the only one whose contract was Windows-production-specific — M08.5.5.B.fix landed the `cmd.exe /C` wrapper that the BatBadBut vuln-bypass + Edge auto-update required. If E behaves oddly, fall back to a release-binary launch: `npx tauri build --no-bundle` + `cargo build --release -p runtime-drone -p runtime-sandbox`, then `.\target\release\agent-runtime.exe`.

**A3. [App]** Close the app fully; in Window A Ctrl+C then `npm run tauri dev`. Expect: **no API-key prompt** — the key persists across restart (M07-IRL #7 close). PASS = no prompt.

**A4. [App]** Click the Import panel (Runtime mode). Look: the panel text is **readable** — no low-contrast / invisible text (M07-IRL #3 close). After a restart the panel still lists previously installed artifacts (M07-IRL #6 close). PASS = readable + survives restart.

Switch back to **Runtime** — the live graph is exactly as you left it. PASS = both views render, switching is lossless.

### Scenario B — Build a framework on the Builder Canvas (M08 baseline + M08.5 🔴-1 drag-enable check)

App in **Builder** mode.

**B1. [App]** Palette → Agents tab. Drag an Agent item onto the empty canvas. Look: an Agent node appears where you dropped it. Drag it to a new spot — it moves and stays. PASS = node instantiates + is draggable. **M08.5-IRL 🔴-1 specific check:** if drag fails to instantiate, the M08.5.B.fix (`tauri.conf.json::dragDropEnabled: false`) regressed — log it.

**B2. [App]** Click the Agent node. Look: an inline config panel — role (text), model (dropdown), allowed_tools / allowed_skills. Set a role; pick a model. Look: below the node, a plain-English capability disclosure. PASS = config edits reflect on the node + the disclosure.

**B3. [App]** Drag a Tool and a Skill from the Palette onto the canvas. Drag an edge from the Agent's handle to the Skill, then Agent → Tool. Look: a wire paints for each. PASS = both edges draw.

**B4. [App]** Drag a wire between two nodes that is **not** a spec edge type (e.g. Tool → Tool). Expect: **no edge appears** — the pair is rejected. PASS = no wire.

**B5. [App]** Drag a second Agent on; draw an Agent → Agent edge. Look: a **narrowing notice** — the child's declared capabilities vs the parent's. Now give the child a capability the parent lacks and re-draw: expect a **rejection** notice + a **red badge** on the child node. PASS = narrowing surfaces; an over-declaring child is flagged red.

**B6. [App]** Mis-configure a node (e.g. clear a required field). Within ~1s: a red badge appears on that node (continuous validation). Fix it: the badge clears. PASS = badges track validity live.
