## Smoke-test env-var fix approach (Option A vs Option B)

Surfaced during the tauri-driver install: smoke test #2 in
`tests/e2e-tauri/smoke.e2e.ts` hardcodes a placeholder API key
(`sk-ant-test-1234567890123456`) and saves it to OS keychain via the
SetupPanel UI. Tests 3-6 then make REAL Anthropic API calls and use the
keychain entry → auth fail with the placeholder.

**Option A — change test code**: smoke test #2 reads
`process.env.ANTHROPIC_API_KEY` when set, falls back to the placeholder
when unset. Tests 3-6 then use the env var via keychain. One-file edit
(`smoke.e2e.ts`), low blast radius. Slight fragility: tests are now
env-dependent.

**Option B — change app code** (recommended): Tauri shell command
that reads the API key checks `ANTHROPIC_API_KEY` env var FIRST, falls
back to keychain when env var unset. Standard pattern (env var as
override). One change in `src-tauri/src/commands.rs` (the key-resolver
command). Tests stay unchanged; both local + CI work cleanly.

## M08.5.5 Stage A.fix — phase-doc errors to log at D.fix close

Three errors in the M08.5.5 phase doc surfaced during Stage A.fix
execution; the build machine corrected them in-stage but they need to
land in the M08.5.5-summary.md "Phase-doc quality notes" section at
D.fix close so the next X.5 phase-doc author (me, M09 cycle onward) is
aware.

1. **§ A.3.1 `onPrepare` recommendation was wrong**. The phase doc said
   `cargo build --release -p agent-runtime -p runtime-drone -p
   runtime-sandbox` for the build hook. But `cargo build -p
   agent-runtime` produces a DEV-mode binary that loads the renderer
   from `tauri.conf.json::devUrl` (unreachable at e2e-run time). The
   correct command is CI's exact form: `npx tauri build --no-bundle`
   for `agent-runtime` (release-build via the Tauri CLI which bundles
   the renderer) + `cargo build --release -p runtime-drone -p
   runtime-sandbox --bins` for the siblings. Build machine adopted
   the CI-mirrored form.

2. **§ A.3.2 verbatim Rust test samples won't compile under workspace
   `forbid(unsafe_code)` + rustc 1.84+**. The phase doc's test samples
   called `std::env::set_var` / `remove_var` directly, which rustc
   1.84+ marks `unsafe` and which the workspace's non-overridable
   `forbid(unsafe_code)` lint blocks even inside test blocks. Build
   machine adopted `temp-env 0.3` (MIT OR Apache-2.0, callback API,
   serializes concurrent calls) as a workspace dev-dependency. Future
   env-var precedence tests should use `temp_env::with_var(…)`
   directly without the unsafe-call wrapping.

3. **§ A.3.6 gotchas.md numbering off by ten**. Phase doc said
   "continue from current highest number" implying the next entries
   are #N+1; the prompt body assumed highest was #72 when actually
   it was #82. The 5 new entries became #83-#87 (not #73-#77). The
   build machine correctly checked the actual highest before
   appending; future phase docs should not pre-state expected
   numbering OR should re-verify at authoring time.

4. **§ B.3.1 unconditional cmd.exe wrap breaks bare-`npx` test**. The
   phase doc prescribed wrap-when-`.cmd`/`.bat`-extension
   unconditional. The existing M06.5 IRL 🟡-2 tests
   (`build_command_resolves_npx_to_npx_cmd_on_windows` etc.) assert
   `get_program() == "npx.cmd"` for bare `npx` (no args). The
   unconditional wrap returns `get_program() == "cmd.exe"` and breaks
   the existing tests. Root cause: the BatBadBut escaping only fires
   when args are present; bare `.cmd` spawn works fine via Windows
   `CreateProcessW`'s transparent cmd.exe handling. Correct shape is
   **conditional on `!self.args.is_empty()`**:
   ```rust
   if (resolved.ends_with(".cmd") || resolved.ends_with(".bat"))
       && !self.args.is_empty()
   {
       // cmd.exe /C wrapper
   } else {
       // direct spawn (preserves M06.5 IRL 🟡-2 contract)
   }
   ```
   The build machine surfaced this at Stage B.fix red-phase as a
   CLAUDE.md §12 ambiguity; orchestration approved the conditional
   form. Future phase docs prescribing platform-specific spawn
   policy MUST enumerate the "bare program, no args" case explicitly.

## M08.5.5 Stage B.fix — toolchain drift discovery

NOT a phase-doc error. Discovery during B.fix red-phase: the integration
test `mcp_add_with_path_args.rs` does NOT deterministically reproduce
the IRL bug on the current toolchain (Windows + Rust 1.95.0 from
2026-04-14). On pre-fix code, the BatBadBut escaping produces a command
line that npx 2026's npm-shipped batch shim parses successfully.

Probable reasons:

1. Rust's BatBadBut escaping has been iteratively refined since 1.77.2;
   current 1.95 handles drive-letter paths more cleanly than the
   1.80-era code that the IRL evidence (2026-05-23) captured.
2. IRL repro was via the release-binary built with a different
   toolchain at app-package time vs the build machine's current Rust
   1.95.
3. IRL arg shape may have included literal shell metacharacters (e.g.,
   the Add modal sent literal `<$SCRATCH>` text rather than the
   PowerShell-expanded path), which would have triggered cmd.exe-level
   parse errors independent of BatBadBut.

**Fix stays in scope and ships**. Two architectural reasons:

- The cmd.exe wrapper is correct regardless. It bypasses BatBadBut
  escaping entirely (we construct the command line ourselves with full
  quoting control). Future-proofs against any Rust escaping regression
  + drives the cross-platform pattern parity (Node.js confirmed).
- Defense-in-depth: the integration test stays as a permanent guard.
  If the bug fires again (toolchain drift, npx version drift, OS
  update), the test catches. It doesn't fail-now on Rust 1.95 but
  pins the contract.

**Empirical RED weight is carried by the two unit tests** that pin the
conditional-wrap contract: (a) bare program with args → cmd.exe wrap;
(b) bare program without args → direct spawn (preserving M06.5 IRL
🟡-2). Both tests fail on pre-fix code (the unconditional original
build_command); both pass on the conditional-wrap impl.

## M08.5.5 Stage C.fix — phase-doc error #7 (SEVERE — would have corrupted user DBs)

Build machine caught this at Stage C.fix pre-implementation grep
verification (2026-05-24). The phase doc's § C.1 Problem Statement
sub-fix 2 inverts the actual migration history. If implemented
literally, migration 004 would have detected session.sqlite's canonical
4-state CHECK + reverted it to the legacy 5-state CHECK with mismapped
status values → data corruption across every user's session.sqlite.

### Grep-verified actual state on main

- `migrations/000_initial.sql` lines 121-122: `mcp_servers.status
  DEFAULT 'configured' CHECK (status IN ('configured', 'connected',
  'errored', 'disabled', 'failed'))` — 5-state, LEGACY pre-spec-§5
  schema (ADR-0006 era).
- `migrations/003_mcp_server_status.sql` (already shipped on main):
  rebuilds to `DEFAULT 'disconnected' CHECK (status IN ('connected',
  'disconnected', 'health_pending', 'error'))` — 4-state, CANONICAL
  spec-§5-aligned schema per CQ-6 alignment to
  `mcp.v1.json::McpServerStatus`.
- Existing test `db.rs::tests::mcp_servers_status_transitions` (line
  474-481): asserts `'errored'` REJECTED post-migration-003.
  `mcp_servers_default_values_applied` (line 522): asserts DEFAULT is
  `'disconnected'`. Both pin 4-state as canonical.

### Phase doc had it inverted

The doc claimed:
- "migration 002 created mcp_servers with the 4-state schema"
- "a later migration (003+) intended to evolve it to the 5-state schema"
- "the IF NOT EXISTS guard suppressed the evolution"

All three premises were wrong. Reality:
- Migration 000 (not 002) created the 5-state (LEGACY).
- Migration 003 evolved to 4-state via `CREATE TABLE _new` +
  data-mapping INSERT + `DROP` + `RENAME` — NOT `IF NOT EXISTS`-gated;
  the evolution ran successfully.
- Direction is 5 → 4. CANONICAL is 4-state.

The fixture SQL in C.3.3 is also internally inconsistent: labels
itself "legacy" but writes the canonical 4-state CHECK with canonical-
value INSERTs.

### Why Sub-fix 2 is dropped entirely

- Sub-fix 1 (stray `mcp.sqlite` cleanup) renames the stray to
  `.stray-mcp.sqlite.bak` — eliminates one side of the "drift" by
  archival.
- session.sqlite already has the canonical schema (migration 003 ran
  on every install before M08.5.5).
- The runtime only opens session.sqlite per ADR-0012 — the stray's
  schema is irrelevant in practice once renamed.
- Migration 004 has no real work to do; migration 003 already did it.
  Any attempt to "reconcile" would either no-op or corrupt.

## M08.5.5 Stage C.fix — phase-doc error #8 (AuditWriter scope creep)

Build machine caught at Stage C.fix pre-implementation (2026-05-24).
Phase doc § C.1 + C.3.1 prescribed `AuditWriter` integration in the
cleanup module signature (`audit: Option<Arc<AuditWriter>>`) and an
`audit_entry(...)` helper. Two problems:

1. `AuditEntryKind` is schema-generated (`runtime_core::generated::
   audit::AuditEntryKind`) — closed enum: `FrameworkLoaded |
   GapDetected | GapResolved | CapabilityGranted | CapabilityDenied |
   McpInstalled | McpUninstalled | McpAuthGranted | McpRequestBlocked
   | TierTransition`. No `StrayDbCleaned` variant. Adding one =
   schema change + new ADR + `cargo xtask regenerate-types` per
   CLAUDE.md §14 — out of stage scope.
2. `AuditWriter::log` is async (`tokio::sync::Mutex`). Tauri's

   `tauri::async_runtime::block_on(...)` — additional indirection +

### Resolution (orchestration approved 2026-05-24)

Drop `AuditWriter` from the cleanup module signature entirely. The
`.stray-mcp.sqlite.bak` file IS the forensic preservation (full
original DB byte-for-byte). User can `sqlite3 .stray-mcp.sqlite.bak`
to inspect or import any row at any time. `tracing::info!` captures
the cleanup event for ops visibility (one INFO line per cleanup with
the original path + rename target + row count read pre-rename).

Corrected signature:

```rust
pub fn cleanup_stray_db(dir: &Path) -> Result<CleanupOutcome, CleanupError> {
    // No AuditWriter — .bak file is the forensic record + tracing INFO
    // captures the cleanup event.
}
```
