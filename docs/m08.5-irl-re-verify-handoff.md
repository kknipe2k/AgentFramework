- **🔴 NEW #4 — Tier state UI/backend desync.** UI shows "Current tier: novice"
  but backend treats user as Promoted (MCP modal opens, which is gated on
  Promoted). Click "Promote to Promoted" → backend logs
  `previous=Promoted target=Promoted; idempotent no-op`. Desync persists
  across `npm run tauri dev` restart.
- **🔴 NEW #5 — Builder Canvas state not persisted across app restart.**
  User's edited framework reverted to an old/different state after Window-A
  restart. No "saved canvas state" load on app start.
- **🔴 NEW #6 — MCP Add/Test buttons enter infinite loop on Windows.**
  Backend gets invoked (button click works — 🔴-3 fix proven) but spawning
  the MCP subprocess fails with Windows OS error: "The filename, directory
  name, or volume label syntax is incorrect." Repeats indefinitely in
  Window A. No UI surface for the failure. No row written to either DB.
  Likely `npx` path resolution or argument quoting in `runtime-mcp` on
  Windows.
- **🔴 NEW #7 — Schema drift between `session.sqlite` and stray `mcp.sqlite`.**
  Both DBs have `mcp_servers` tables with DIFFERENT status CHECK constraints:
  - `session.sqlite`: `status IN ('connected', 'disconnected', 'health_pending', 'error')`
  - stray `mcp.sqlite`: `status IN ('configured', 'connected', 'errored', 'disabled', 'failed')`
  Two SQLite DBs on the same machine with incompatible schemas for the same
  table name — exactly the schema-drift class the architectural audit warned
  about.
- **🔴 STILL OPEN — M06.5 🔴-1 MCP-registry mis-resolve.** The stray
  `mcp.sqlite` file exists at
  `C:\Users\kknip\AppData\Local\dev.aria-runtime.app\mcp.sqlite` (4096 bytes,
  dated 5/17/2026), contains a COMPLETE session-DB schema (sessions,
  signals, snapshots, vdr, plans, tasks, mcp_servers, etc. — not just an
  mcp-only DB), and has an `fs-test` row from 5/17 prior testing.
  `session.sqlite` mcp_servers is empty. The MCP registry is writing to the
  WRONG DB. The E.fix amend planned to mark this RESOLVED — it must stay
