Surfaced during the tauri-driver install: smoke test #2 in
`tests/e2e-tauri/smoke.e2e.ts` hardcodes a placeholder API key
(`sk-ant-test-1234567890123456`) and saves it to OS keychain via the

**Option A тАФ change test code**: smoke test #2 reads
`process.env.ANTHROPIC_API_KEY` when set, falls back to the placeholder

env-dependent.

that reads the API key checks `ANTHROPIC_API_KEY` env var FIRST, falls
back to keychain when env var unset. Standard pattern (env var as
override). One change in `src-tauri/src/commands.rs` (the key-resolver
command). Tests stay unchanged; both local + CI work cleanly.

## M08.5.5 Stage A.fix тАФ phase-doc errors to log at D.fix close

Three errors in the M08.5.5 phase doc surfaced during Stage A.fix
execution; the build machine corrected them in-stage but they need to
land in the M08.5.5-summary.md "Phase-doc quality notes" section at
D.fix close so the next X.5 phase-doc author (me, M09 cycle onward) is
aware.

1. **┬з A.3.1 `onPrepare` recommendation was wrong**. The phase doc said
   `cargo build --release -p agent-runtime -p runtime-drone -p
   runtime-sandbox` for the build hook. But `cargo build -p
   agent-runtime` produces a DEV-mode binary that loads the renderer
   from `tauri.conf.json::devUrl` (unreachable at e2e-run time). The
   correct command is CI's exact form: `npx tauri build --no-bundle`
   for `agent-runtime` (release-build via the Tauri CLI which bundles
   the renderer) + `cargo build --release -p runtime-drone -p
   runtime-sandbox --bins` for the siblings. Build machine adopted
   the CI-mirrored form.

2. **┬з A.3.2 verbatim Rust test samples won't compile under workspace
   `forbid(unsafe_code)` + rustc 1.84+**. The phase doc's test samples
   called `std::env::set_var` / `remove_var` directly, which rustc
   1.84+ marks `unsafe` and which the workspace's non-overridable
   `forbid(unsafe_code)` lint blocks even inside test blocks. Build
   machine adopted `temp-env 0.3` (MIT OR Apache-2.0, callback API,
   serializes concurrent calls) as a workspace dev-dependency. Future
   env-var precedence tests should use `temp_env::with_var(тАж)`
   directly without the unsafe-call wrapping.

3. **┬з A.3.6 gotchas.md numbering off by ten**. Phase doc said
   "continue from current highest number" implying the next entries
   are #N+1; the prompt body assumed highest was #72 when actually
   it was #82. The 5 new entries became #83-#87 (not #73-#77). The
   build machine correctly checked the actual highest before
   appending; future phase docs should not pre-state expected
   numbering OR should re-verify at authoring time.

4. **┬з B.3.1 unconditional cmd.exe wrap breaks bare-`npx` test**. The
   phase doc prescribed wrap-when-`.cmd`/`.bat`-extension
   unconditional. The existing M06.5 IRL ЁЯЯб-2 tests
   (`build_command_resolves_npx_to_npx_cmd_on_windows` etc.) assert
   `get_program() == "npx.cmd"` for bare `npx` (no args). The
   unconditional wrap returns `get_program() == "cmd.exe"` and breaks
   the existing tests. Root cause: the BatBadBut escaping only fires
   when args are present; bare `.cmd` spawn works fine via Windows
   `CreateProcessW`'s transparent cmd.exe handling. Correct shape is
   **conditional on `!self.args.is_empty()`**:
   ```rust
   if (resolved.ends_with(".cmd") || resolved.ends_with(".bat"))
       && !self.args.is_empty()
   {
       // cmd.exe /C wrapper
   } else {
       // direct spawn (preserves M06.5 IRL ЁЯЯб-2 contract)
   }
   ```
   The build machine surfaced this at Stage B.fix red-phase as a
   CLAUDE.md ┬з12 ambiguity; orchestration approved the conditional
   form. Future phase docs prescribing platform-specific spawn
   policy MUST enumerate the "bare program, no args" case explicitly.

## M08.5.5 Stage B.fix тАФ toolchain drift discovery

NOT a phase-doc error. Discovery during B.fix red-phase: the integration
test `mcp_add_with_path_args.rs` does NOT deterministically reproduce
the IRL bug on the current toolchain (Windows + Rust 1.95.0 from
2026-04-14). On pre-fix code, the BatBadBut escaping produces a command
line that npx 2026's npm-shipped batch shim parses successfully.

Probable reasons:

1. Rust's BatBadBut escaping has been iteratively refined since 1.77.2;
   current 1.95 handles drive-letter paths more cleanly than the
   1.80-era code that the IRL evidence (2026-05-23) captured.
2. IRL repro was via the release-binary built with a different
   toolchain at app-package time vs the build machine's current Rust
   1.95.
3. IRL arg shape may have included literal shell metacharacters (e.g.,
   the Add modal sent literal `<$SCRATCH>` text rather than the
   PowerShell-expanded path), which would have triggered cmd.exe-level
   parse errors independent of BatBadBut.

**Fix stays in scope and ships**. Two architectural reasons:

- The cmd.exe wrapper is correct regardless. It bypasses BatBadBut
  escaping entirely (we construct the command line ourselves with full
  quoting control). Future-proofs against any Rust escaping regression
  + drives the cross-platform pattern parity (Node.js confirmed).
- Defense-in-depth: the integration test stays as a permanent guard.
  If the bug fires again (toolchain drift, npx version drift, OS
  update), the test catches. It doesn't fail-now on Rust 1.95 but
  pins the contract.

**Empirical RED weight is carried by the two unit tests** that pin the
conditional-wrap contract: (a) bare program with args тЖТ cmd.exe wrap;
(b) bare program without args тЖТ direct spawn (preserving M06.5 IRL
ЁЯЯб-2). Both tests fail on pre-fix code (the unconditional original
build_command); both pass on the conditional-wrap impl.
