## 🔴 findings (block M9; or M08.6.7 hotfix candidates)

### #12 — No "clear canvas" / "new framework" affordance

**Step:** B-wide. **Maintainer-flagged priority "fix right now."**

Combined with no node/edge delete (#11), broken canvas state requires restarting the app to recover. The Builder is effectively build-up-only with no recovery path in v0.1. Quick fix: one button that resets `builderStore.framework` to `emptyFramework()` + clears `nodePositions` + emits a "Canvas cleared" toast.

**Disposition:** M08.6.7 hotfix candidate.

### #17 — `save_framework` hard-fails for any framework built from-scratch

**Step:** C4. **Compound of #2 + #3** (Palette's blank-Agent template ships schema-invalid: ID `demo-agent@1.0.0` violates regex + missing required `capabilities` block).

The `save_framework` Tauri command rejects at IPC deserialization (typify `FrameworkAgentsItem` untagged-enum mismatch) before reaching the filesystem. Net effect: a user starting from scratch in the Builder cannot save their work. The Canvas mechanics work in isolation; the persistence layer is unreachable.

**End-to-end the M08 headline acceptance ("build a framework in the Builder + save it to disk") is broken for any fresh-start user.**

**Disposition:** M08.6.7 hotfix candidate — fix the Palette's Agent template to ship a valid ID + populate the required `capabilities` block. ALTERNATIVELY: route to M9 (the mentor handles agent authoring, so the Palette can ship a simpler placeholder).

UI displays **Novice**, backend treats user as **Promoted** (proof: MCP Servers Add is available, which is tier-gated on Promoted). Routed to M9 Stage A at M08.6.A intake; still open and re-confirmed during this M08.6 IRL pass.

**Disposition:** M9 Stage A — must close (this is the second IRL re-confirm).

### #22 — Budget cap doesn't persist across app restart (regression of M08.G claimed close)

**Step:** E3.

M06.5-IRL 🟡-4 was claimed RESOLVED at M08.G per the gap-analysis: "globalBudgetCap slot + setGlobalBudgetCap action wire the budget-cap input ... persisted via the existing `set_global_budget` command. FINAL DISPOSITION." **Regression of the claimed close.** Either: (a) the wire to `set_global_budget` is broken, (b) `set_global_budget` writes to a non-persistent location, or (c) the renderer doesn't read the persisted value on app mount.

**Disposition:** M08.6.7 hotfix candidate — budget cap is a safety primitive (spend control).

### #23 — MCP `npx` spawn fails on Windows + Node 24 (new failure class)

**Step:** E4. **NOT a regression of M08.5.5 🔴 #6** — that one was OS error 123 / infinite loop, fixed by the cmd.exe /C wrapper (M08.5.5.B + B2.fix). This is a **second layer** that's missing.

Per Window A log: the spawn invokes npx successfully (the cmd.exe wrapper works), but npx's internal `require()` resolves to `C:\agent-runtime\src-tauri\node_modules\npm\bin\npm-prefix.js` / `npx-cli.js` — paths that don't exist because `src-tauri/node_modules/` doesn't bundle npm. The spawn appears to use `src-tauri/` as CWD or has NODE_PATH pointing there, instead of resolving npx via the system npm install.

User-facing error: "internal: mcp_add_server: MCP transport: MCP connect failed: connection closed: initialize response" (opaque — would be helpful to surface "spawned process failed to initialize, check log").

**Likely root cause classes:**
- Spawn's CWD is set to `src-tauri/` and Node resolves require paths from there
- Or NODE_PATH env is being injected pointing at the local `src-tauri/node_modules`
- Or the cmd.exe /C wrapper isn't preserving the correct PATH/NODE_PATH for system npm

**Disposition:** M08.6.7 hotfix candidate; touches `runtime-mcp` Windows spawn path. Worth checking on Node 22 / older to determine if Node-24-specific.

### #31 — ARIA framework has 6 narrowing failures + 1 unresolved tool reference

**Step:** F3. **The reference framework can't pass its own validator.**

Orchestrator's `allowed_tools` is missing `Write` / `Edit` / `Bash` that its children (planner, implementer, verify-app, simplifier, report-writer) declare. Per L2a narrowing (§8.security spec), children can't hold capabilities the parent doesn't grant. Also: `propose_mode` is referenced by router's `allowed_tools` but not declared in `framework.tools[]` — ARIA authoring oversight.

**This is the validator working correctly + ARIA's framework violating it.**

Two reads:
- (A) ARIA's framework is malformed — orchestrator should declare Write/Edit/Bash so it can transitively narrow to children. Quick fix to `examples/aria/framework.json`.
- (B) The narrowing rule is over-strict for delegation patterns — orchestrator delegates work; it doesn't directly use Bash/Edit/Write. ADR-class decision on delegation-aware narrowing.

M08.6 didn't break this — it exposed it (pre-M08.6, the capability summary was blank on ARIA because the `inline_agents` filter excluded path-ref agents, so narrowing failures were invisible).

**Disposition:** Maintainer pick — quick fix to ARIA file (low-risk; preserves narrowing model) OR ADR-class delegation-narrowing decision (touches §8.security primitive).

### #32 — `save_framework` drops tools/skills companion files (M08.6.C contract regression)

**Step:** G1 + G4 (definitive evidence).

The save wrote `framework.json` + `agents/<8 .md>` correctly. **NO `tools/` or `skills/` subdirectory exists** in the save output, despite `framework.json` declaring 13 tool entries (3 with `path:`) + 5 skill entries (all with `path:`).

Load of the saved framework fails with: `BuilderError::ReferenceResolution: read C:\Users\kknip\AppData\Local\Temp\m086irl-rt-597787158\tools/git_checkpoint.md failed: The system cannot find the path specified. (os error 3)`

**The Companion-write half of the inverse re-split is broken.** Agents work; tools/skills don't.

**MVP §M8 criterion 8 (byte-stable save→load→save round-trip) is functionally violated end-to-end** — the round-trip can never complete because save→load is broken.

**Stage V escape:** M08.6.V returned Sound with 0 🔴, but the `framework_save_resplit.rs` tests assert only on `framework.json` hash, not on companion files being written. Gotcha #66 class.

**Disposition:** M08.6.7 hotfix candidate. Fix is likely small — `save_framework` needs to iterate the `Companions` collection and write each to its `file_name` path. Load side already produces Companions correctly per F1's successful resolve.

## Stage V escape — the M08.6 ship gap

**Finding #32 is a Stage V escape.** M08.6.V (the in-band verifier) returned Sound with 0 🔴, but `framework_save_resplit.rs` tests assert only on `framework.json` hash equality across round-trips. They DON'T verify the companion files (`agents/*.md`, `tools/*.md`, `skills/*.md`) are written. Round-trip "byte-stability" can pass mechanically (the framework.json IS deterministic) while the framework is missing 18 body files.

**This is gotcha #66 class (tests-pass-but-contract-fails) — the same trap M08.6 itself was designed to close for the LOAD side.**

The IRL pass found what Stage V's structured tests didn't. The fix for the test gap: the byte-stability test should verify the FULL DIRECTORY layout (file count + per-file hash), not just framework.json. Worth adding as a Stage V protocol improvement.
